# 프로젝트 헌장 — fastpoten-brand-factory

## 에이전트 자동 호출 규칙 (메인 Claude에게 — 이 섹션은 그대로 둔다)
이 프로젝트에는 전용 서브에이전트들이 설치돼 있다(.claude/agents/). 사용자가 이름을 부르지 않아도 요청 유형에 따라 자동 위임하라:
- "~한 아이디어가 있는데 / 만들면 어떨까" → ideator (선례 질문은 scout)
- 기능 요청·수정 요청 → planner 먼저 (경량이면 바로 implementer)
- "안 된다 / 에러 / 이상하다" → debugger 먼저
- 구현 = implementer, 구현 후 reviewer 검토 기본
- 화면이 새로 생기거나 바뀌는 차수 → 구현 전 designer
- 큰 기능 완료 또는 작업 5~10건 누적 → retro 제안
- 결정 필요 항목은 반드시 사용자에게 멈춰 묻는다. 에이전트끼리 결정하지 않는다.

## 프로젝트 개요
- 한 줄 소개: 부서의 검증된 콘텐츠(카페·블로그 글 1만여 건)를 자산창고로 정리하고, 그 재료로 **자사 브랜드 채널(홈페이지 SEO·워드프레스·인스타)에 콘텐츠를 자동 생산·발행·개선하는 "브랜드 자산 팩토리"**를 만든다.
- 기술 스택: Python 3.11+ / SQLite(자산 전용, CRM과 별개) / 폴더 코퍼스 / 외부 AI API(원고·임베딩) / HTML템플릿→PNG(카드뉴스) / 워드프레스 공식 연동
- 구조: docs/(기획) · src/(코드) · data/(DB, git제외) · corpus/(글·이미지, git제외) · inbox/(수동 투입함) · templates/(카드뉴스 템플릿)
- 5층 구조: [0층 자산창고] → [1층 원고 생성] → [2층 멀티포맷(카드뉴스)] → [3층 발행(워드프레스)] → [4층 성과 수집] → [5층 자기 개선 루프]

## 기준 사용자 ★
- 누구인가: **관리자 1인(비개발자)** — 브랜드 자산을 직접 쌓는 운영자. 직원용 도구가 아니다(직원 확산은 백로그).
- 주 사용 환경: 회사 PC + 집 PC(깃허브로 이어감), 클로드 코드 사용 중.
- 코드 지식 수준: 비개발자. **모든 설명·결정 질문은 코드 용어 없이 화면·업무 흐름 기준 일상 언어로.** 구현 방법은 묻지 않는다.
- 현장 용어: 자산창고(=콘텐츠 DB), 팩토리(=생성~발행 파이프라인), 카드뉴스(=정보성 이미지), 참고 신호(=카페 순위·조회수). 쓰면 안 되는 말: "성과 등급", "P1/P2/P3".

## 진짜 불변 ★
1. **개인정보 마스킹**: 참고용 본문·모든 생성 산출물에서 전화번호·오픈채팅·카페닉네임·직원 실명·OO쌤/멘토/팀장 호칭을 반드시 가린다. 개인정보 든 이미지는 원본 재사용 금지(image_pattern_only).
2. **계정정보 시트 미반입**: 엑셀 `계정 정보` 탭(비밀번호 등)은 어떤 형태로도 DB·코퍼스에 넣지 않는다. account_id 라벨은 '현황' 시트에서만.
   - **2-1. AI 열쇠는 PC 안 파일 한 곳에만(2026-07-21, ※사용자 승인 대기)**: 자리는 `ai_client.KEY_PATH`가 정한다(현재 `%USERPROFILE%\.fastpoten\api_key.txt`). 창고(`data/`)·저장소·구글드라이브 어디에도 두지 않는다(계획 §Q4) — `data/`는 구글드라이브로 옮겨지는 폴더라 열쇠가 클라우드에 실린다. **환경변수 `ANTHROPIC_API_KEY`도 금지** — 거기 두면 클로드 코드 같은 다른 프로그램이 발견해 제 통신 요금으로 태운다(실제로 원고 한 편 뽑기 전에 5만원가량 나갔다). 이 열쇠는 **원고 생성 전용**이다. 읽는 곳은 `ai_client.read_key()` 하나뿐.
     - **감시의 정직한 한계**: `python src/check_leak.py` ④번이 환경변수 잔류를 **화면에 경고할 뿐, 검사를 실패시키지는 않는다**(종료코드 미반영 — 반영 여부는 사용자 판단 대기). 또 ④는 **그 창이 켜질 때의 설정만** 본다. 즉 이 불변은 자동으로 막히는 게 아니라 사람이 보고 지키는 것이다.
3. **성과 데이터 이원화**: **카페 데이터의** 순위·조회수는 참고 신호일 뿐 성과로 단정하지 않는다. **자사 채널(워드프레스·홈페이지·인스타)의 실측 지표(서치콘솔·GA·인사이트)만 성과 데이터로 사용**하며, 자기 개선 루프의 근거는 자사 채널 실측만이다.
4. **본문은 파일에만**: DB에 본문 텍스트 컬럼 금지(경로만). 원문(body_raw)은 절대 수정하지 않는다.
5. **네이버 자동 추출(카페·블로그)은 백로그**: 사용자가 명시 요청하기 전에는 스크래핑 코드를 만들지 않는다.
6. **팩트는 룰북에서만**: 학점·응시자격 등 제도 수치는 룰북 데이터에서만 가져오고, 룰북에 없으면 [검수:] 표시. 생성 원고에 학점은행제 기간·비용을 단정 수치로 쓰지 않는다. **룰북에 있어도 '미확인'이면 [검수:] 표시**(CA-6 계획 §7에서 확장).
   - **6-1. 수치는 공통 팩트에만(2026-07-21)**: 룰북 팩트는 두 종류이고 역할이 다르다 — **공통 팩트 = 제도 사실**(요건·학점·기간·FAQ), **개별 팩트 = 키워드별 원고 지침**(한 줄 요약·강조포인트·우선순위). **수치(시간·학점·과목수·기간·비용)는 공통에만 쓴다.** 개별에 수치가 들어가면 같은 사실이 두 곳에 생겨 조용히 어긋난다 — 실제로 어긋났다: 공통 '사회복지사 2급'은 실습 160시간(구법 120)인데 개별 '사회복지사2급자격증취득방법'은 조건 없이 "실습120시간"(신규 대상자에게 오답), 개별 '고졸 사회복지사2급'에도 "전문학사(80학점)"이 새어 있다. 개별은 수치를 적는 대신 **어느 공통 팩트를 근거로 쓰는지 가리킨다**(잇기 작업은 첫 원고 뒤).
7. **자동 발행은 자사 소유 채널만**(워드프레스·홈페이지). 네이버 채널은 완성 원고 출력까지만(발행은 사람).
8. **발행은 승인 후**: 자동 발행이라도 사람 승인 없이 나가는 글은 없다(승인 큐 필수).
9. DB 마이그레이션은 멱등(ensure 패턴). 컬럼 삭제 금지(숨김 OK). 간결(YAGNI) — 단 입력검증·마스킹·데이터 손실 방지는 간결화 대상 아님.

## 현재 결정 (재검토 가능)
1. 파일럿은 수동 투입(inbox/) 방식 — 자동 추출 전에 정리 품질부터 검증.
2. 검토 화면은 요약(A안), 저장은 전체(B안 — 서비스기획서 v9 부록 A 형식).
3. 임베딩은 외부 AI API + numpy(전용 벡터DB 없음). 구체 API는 CA-5 착수 시 1회 결정.
4. 인스타 발행은 직접 연동보다 기성품(Mirr류) 우선 검토 — 콘텐츠의 뇌만 자체 구축, 손발은 사서 쓴다.
5. 카드뉴스는 "AI는 JSON만, 렌더링은 템플릿" 방식(토큰 절감·글자 깨짐 방지).
6. 직원용 기능(프롬프트 팩·표현 뱅크·검수기 공유)은 백로그 — 기준 사용자가 1인으로 재정의됨.
7. 트랙 B(카드뉴스)는 트랙 A(창고)와 병행 가능 — 룰북만으로 시작.
8. **축소·재초점(2026-07-07)**: 5층 전체 짓기 보류. 자체 구축 정당 = 창고(0층)·개인정보 마스킹·룰북 팩트 강제 3개뿐. 원고 SEO 문장·워드프레스 발행·인스타 예약·성과 수집은 기성품/수동(손발은 사서 쓴다). 3·4·5층(자동발행·성과·개선)은 **워드프레스 트래픽 + 물량(주 4편+) 확인 후**. 채널 현실: 홈페이지 유입 有 / 워드프레스 신규. 근거: docs/plans/20260707-refocus-축소.md (field-critic·scout).
9. **창고는 로컬, 드라이브는 보관소(2026-07-16)**: 창고(data/)를 이 PC 안에 두고 코퍼스(corpus/)만 구글드라이브 연결. 실측 — 같은 질의가 드라이브 25.9초 vs 로컬 0.4초(약 90배)라 뷰어가 사실상 사용 불가였음. SQLite는 파일을 잘게 훑는 접근이라 가상 드라이브에 특히 약하다(**클라우드라서가 아니라 파일 동기화 방식이라서** — 진짜 DB 서비스는 클라우드여도 빠르다). 코퍼스는 통째 읽기라 글당 0.23초로 견딜 만해 그대로 둔다. PC 전환은 `작업시작`·`작업마침` 명령으로. 근거: docs/plans/20260716-창고-로컬화-동기화명령.md
10. **가림 건수는 세서 저장(2026-07-16)**: 목록이 글마다 본문 파일을 다시 열어 개인정보를 다시 세던 방식(3,777건 = 36초)을 폐기. 건수를 창고에 저장하되 **어떤 규칙으로 셌는지 표를 같이 적어**, 그 표가 지금과 다르면 예전 숫자가 조용히 나오지 않고 "다시 세기 필요"로 뜬다(자동 재계산 안 함 — 뷰어는 읽기 전용 유지). **표에 들어가는 것 = ①창고에 든 가림 규칙(개인정보 패턴·직원 이름 목록)은 바뀌는 즉시 자동으로 ②세는 방식 자체를 손볼 때는 사람이 코드 곁의 번호를 올려서.** 이 둘 밖의 변화는 표가 못 잡는다(정직하게 적어둠 — "규칙이 한 글자만 바뀌어도 잡힌다"는 과장이었다). 실측: 36.3초·1,023KB → **0.06초·37KB**. 근거: docs/plans/20260716-글목록-성능-가림건수-저장.md

## 동기화 필수 쌍
- 린터의 개인정보 패턴 ↔ 정제(마스킹)의 개인정보 패턴 (이유: 한쪽만 갱신하면 마스킹 누락 발생 — 패턴 정의는 한 곳에 두고 양쪽이 참조)
- 룰북 금지어 목록 ↔ 생성 프롬프트의 금지어 (이유: 동일)
- AI 열쇠 자리(`ai_client.KEY_PATH`) ↔ 헌장 불변 2-1의 경로 문구 (이유: 하루 만에 두 번 바뀐 값이다 — 한쪽만 고치면 헌장을 따른 사람이 엉뚱한 곳에 열쇠를 둔다. **정본은 코드(`KEY_PATH`)**, 헌장은 현재 값을 괄호로 인용만)

## 검증 방법 ★
- 파이썬 변경 시: `python -m py_compile <파일>` + 해당 모듈 자체 테스트(tests/) 통과
- DB 변경 시: 마이그레이션 후 기존 데이터 건수 불변 확인(감소하면 차단)
- 마스킹 변경 시: 테스트 문장(전화번호·OO쌤 포함)이 반드시 가려지는지 자동 확인
- 발행(3층) 변경 시: 실제 발행 전 드라이런(초안 상태로만 올리기) 결과를 사용자에게 보여주고 승인
- 배포 개념 없음(로컬 독립 실행). 깃허브 push는 사용자 승인 후에만.

## 디자인 규칙
- 카드뉴스 템플릿: templates/에 HTML로. 브랜드 색·글꼴은 templates/brand.css 한 곳에서만 정의(개별 템플릿에 색 하드코딩 금지).
- 웹 뷰어를 만들게 되면 그린필드 모드로 designer가 씨앗 패턴 정의.
- **화면은 만든 뒤 실물을 보고 판정한다(2026-07-21)**: designer는 렌더링된 픽셀을 보지 못한다(스스로 명세에 적어둔 한계). **화면 차수의 완료 조건에 "브라우저로 열어 확인"을 넣는다.** 근거: 실물을 처음 본 날 스크린샷 없이는 못 잡았을 것이 2건 나왔다(고친 칸이 안 접힘 · 접기 삼각형 중복). 테스트가 전부 통과해도 눈으로만 보이는 것이 있다.
- **근거를 감추지 않는다(2026-07-20 승격)**: 자동으로 만들어진 값(추출·정규화·AI 초안·집계)을 보여주는 화면은 **사람이 원본과 대조할 수 있는 길을 같은 화면 안에** 둔다. 값만 보여주고 근거를 감추지 않는다. 같은 요구가 네 번 나와 규칙으로 올림 — ①뷰어 원문 대조(07-07) ②주제 정규화에 원본 키워드 함께(07-14) ③팩트의 엑셀 원본 보기(07-15) ④주제로 거른 글 목록에 묶인 원본 키워드(07-20).
- **표의 지면은 정보량으로 나눈다(2026-07-21)**: 사람이 **읽어서 구별하는 열**(제목·항목명)이 넓다. 값의 종류가 적거나 길이가 고정된 열(숫자·짧은 라벨)은 최소폭. **열 폭을 조정할 때는 그 열에 실제로 든 값의 길이를 재고 나서 정한다** — 실측 예: 글 제목 중간값 30자·95% 39자·최장 55자 / 카페·담당자 최장 5자 / 상태 8자. 근거: 07-20에 "성공(자동추출)이 줄바꿈된다"를 고치며 상태 열을 넓히고 제목 열을 줄인 결과, **3,777건이 전부 같은 값인 열이 제목보다 넓어져 정작 제목이 잘렸다**. 한 불편을 고치다 다른 불편을 만든 것이고, 원인은 폭을 조정하며 각 열의 정보량을 안 본 것이다.
