# 프로젝트 헌장 — fastpoten-brand-factory

## 에이전트 자동 호출 규칙 (메인 Claude에게 — 이 섹션은 그대로 둔다)
이 프로젝트에는 전용 서브에이전트들이 설치돼 있다(.claude/agents/). 사용자가 이름을 부르지 않아도 요청 유형에 따라 자동 위임하라:
- "~한 아이디어가 있는데 / 만들면 어떨까" → ideator (선례 질문은 scout)
- 기능 요청·수정 요청 → planner 먼저 (경량이면 바로 implementer)
- "안 된다 / 에러 / 이상하다" → debugger 먼저
- 구현 = implementer, 구현 후 reviewer 검토 기본
- 화면이 새로 생기거나 바뀌는 차수 → 구현 전 designer
- 큰 기능 완료 또는 작업 5~10건 누적 → retro 제안
- 결정 필요 항목은 반드시 사용자에게 멈춰 묻는다. 에이전트끼리 결정하지 않는다.

## 프로젝트 개요
- 한 줄 소개: 부서의 검증된 콘텐츠(카페·블로그 글 1만여 건)를 자산창고로 정리하고, 그 재료로 **자사 브랜드 채널(홈페이지 SEO·워드프레스·인스타)에 콘텐츠를 자동 생산·발행·개선하는 "브랜드 자산 팩토리"**를 만든다.
- 기술 스택: Python 3.11+ / SQLite(자산 전용, CRM과 별개) / 폴더 코퍼스 / 외부 AI API(원고·임베딩) / HTML템플릿→PNG(카드뉴스) / 워드프레스 공식 연동
- 구조: docs/(기획) · src/(코드) · data/(DB, git제외) · corpus/(글·이미지, git제외) · inbox/(수동 투입함) · templates/(카드뉴스 템플릿)
- 5층 구조: [0층 자산창고] → [1층 원고 생성] → [2층 멀티포맷(카드뉴스)] → [3층 발행(워드프레스)] → [4층 성과 수집] → [5층 자기 개선 루프]

## 기준 사용자 ★
- 누구인가: **관리자 1인(비개발자)** — 브랜드 자산을 직접 쌓는 운영자. 직원용 도구가 아니다(직원 확산은 백로그).
- 주 사용 환경: 회사 PC + 집 PC(깃허브로 이어감), 클로드 코드 사용 중.
- 코드 지식 수준: 비개발자. **모든 설명·결정 질문은 코드 용어 없이 화면·업무 흐름 기준 일상 언어로.** 구현 방법은 묻지 않는다.
- 현장 용어: 자산창고(=콘텐츠 DB), 팩토리(=생성~발행 파이프라인), 카드뉴스(=정보성 이미지), 참고 신호(=카페 순위·조회수). 쓰면 안 되는 말: "성과 등급", "P1/P2/P3".

## 진짜 불변 ★
1. **개인정보 마스킹**: 참고용 본문·모든 생성 산출물에서 전화번호·오픈채팅·카페닉네임·직원 실명·OO쌤/멘토/팀장 호칭을 반드시 가린다. 개인정보 든 이미지는 원본 재사용 금지(image_pattern_only).
2. **계정정보 시트 미반입**: 엑셀 `계정 정보` 탭(비밀번호 등)은 어떤 형태로도 DB·코퍼스에 넣지 않는다. account_id 라벨은 '현황' 시트에서만.
3. **성과 데이터 이원화**: **카페 데이터의** 순위·조회수는 참고 신호일 뿐 성과로 단정하지 않는다. **자사 채널(워드프레스·홈페이지·인스타)의 실측 지표(서치콘솔·GA·인사이트)만 성과 데이터로 사용**하며, 자기 개선 루프의 근거는 자사 채널 실측만이다.
4. **본문은 파일에만**: DB에 본문 텍스트 컬럼 금지(경로만). 원문(body_raw)은 절대 수정하지 않는다.
5. **네이버 자동 추출(카페·블로그)은 백로그**: 사용자가 명시 요청하기 전에는 스크래핑 코드를 만들지 않는다.
6. **팩트는 룰북에서만**: 학점·응시자격 등 제도 수치는 룰북 데이터에서만 가져오고, 룰북에 없으면 [검수:] 표시. 생성 원고에 학점은행제 기간·비용을 단정 수치로 쓰지 않는다.
7. **자동 발행은 자사 소유 채널만**(워드프레스·홈페이지). 네이버 채널은 완성 원고 출력까지만(발행은 사람).
8. **발행은 승인 후**: 자동 발행이라도 사람 승인 없이 나가는 글은 없다(승인 큐 필수).
9. DB 마이그레이션은 멱등(ensure 패턴). 컬럼 삭제 금지(숨김 OK). 간결(YAGNI) — 단 입력검증·마스킹·데이터 손실 방지는 간결화 대상 아님.

## 현재 결정 (재검토 가능)
1. 파일럿은 수동 투입(inbox/) 방식 — 자동 추출 전에 정리 품질부터 검증.
2. 검토 화면은 요약(A안), 저장은 전체(B안 — 서비스기획서 v9 부록 A 형식).
3. 임베딩은 외부 AI API + numpy(전용 벡터DB 없음). 구체 API는 CA-5 착수 시 1회 결정.
4. 인스타 발행은 직접 연동보다 기성품(Mirr류) 우선 검토 — 콘텐츠의 뇌만 자체 구축, 손발은 사서 쓴다.
5. 카드뉴스는 "AI는 JSON만, 렌더링은 템플릿" 방식(토큰 절감·글자 깨짐 방지).
6. 직원용 기능(프롬프트 팩·표현 뱅크·검수기 공유)은 백로그 — 기준 사용자가 1인으로 재정의됨.
7. 트랙 B(카드뉴스)는 트랙 A(창고)와 병행 가능 — 룰북만으로 시작.

## 동기화 필수 쌍
- 린터의 개인정보 패턴 ↔ 정제(마스킹)의 개인정보 패턴 (이유: 한쪽만 갱신하면 마스킹 누락 발생 — 패턴 정의는 한 곳에 두고 양쪽이 참조)
- 룰북 금지어 목록 ↔ 생성 프롬프트의 금지어 (이유: 동일)

## 검증 방법 ★
- 파이썬 변경 시: `python -m py_compile <파일>` + 해당 모듈 자체 테스트(tests/) 통과
- DB 변경 시: 마이그레이션 후 기존 데이터 건수 불변 확인(감소하면 차단)
- 마스킹 변경 시: 테스트 문장(전화번호·OO쌤 포함)이 반드시 가려지는지 자동 확인
- 발행(3층) 변경 시: 실제 발행 전 드라이런(초안 상태로만 올리기) 결과를 사용자에게 보여주고 승인
- 배포 개념 없음(로컬 독립 실행). 깃허브 push는 사용자 승인 후에만.

## 디자인 규칙
- 카드뉴스 템플릿: templates/에 HTML로. 브랜드 색·글꼴은 templates/brand.css 한 곳에서만 정의(개별 템플릿에 색 하드코딩 금지).
- 웹 뷰어를 만들게 되면 그린필드 모드로 designer가 씨앗 패턴 정의.
