# 콘텐츠 자산화 시스템 — Phase 1 개발 기획서 v1

> 서비스 기획서 v9(docs/서비스기획서-v9.md)를 실제로 만들기 위한 설계도.
> **클로드 코드 작업 지시서 겸용** — 각 차수를 그대로 planner→implementer→reviewer 사이클에 넘기면 된다.
> 사용자는 비개발자다. 결정 질문은 화면·사용 흐름 기준 일상 언어로만 한다.

---

## 1. 기술 스택 (확정)

| 항목 | 선택 | 쉬운 설명 |
|---|---|---|
| 언어 | Python 3.11+ | CRM 백엔드와 같은 언어 — 팀에 이미 익숙 |
| DB | SQLite 1개 파일 (`data/content_assets.sqlite3`) | CRM과 **완전 별개 파일**. 목록표(장부) 역할 |
| 본문·이미지 | 폴더 코퍼스 (`corpus/`) | 글 내용과 사진은 파일로, DB엔 위치만 기록 |
| 임베딩 | 외부 AI 임베딩 API + numpy | "비슷한 글 찾기" 지문. 전용 벡터DB 안 씀 |
| 추출(추후) | Playwright — **기존 순위체크 방식 확장** | 백로그. 지금은 안 만듦 |
| 화면(추후) | 간단 웹 뷰어(FastAPI) 또는 CSV 내보내기 | 파일럿 결과를 눈으로 볼 최소 수단 |
| 버전관리 | GitHub (CRM과 별개 저장소) | 회사↔집 이어서 작업 |

원칙: 새 의존성 최소(YAGNI). pandas/openpyxl/numpy/httpx 수준에서 시작.

## 2. 폴더 구조 (확정)

```
fastpoten-content-assets/
├─ CLAUDE.md              ← 클로드 코드가 매 세션 읽는 총괄 문서
├─ README.md              ← 사람용 시작 가이드
├─ BACKLOG.md             ← 미룬 것 목록(네이버 로그인 추출 등)
├─ docs/                  ← 기획서들
├─ src/                   ← 프로그램 코드 (차수 진행하며 생성)
│   ├─ db.py              (DB 연결·스키마)
│   ├─ ingest_excel.py    (1-A: 엑셀 링크·메타 적재)
│   ├─ intake_manual.py   (파일럿: 붙여넣기/파일 투입 → 표준양식 정리)
│   ├─ clean.py           (정제·마스킹)
│   ├─ paragraphs.py      (문단 분리·역할 태깅)
│   ├─ linter.py          (1-D: 자동 검수)
│   └─ embed.py           (임베딩·클러스터링)
├─ data/                  ← content_assets.sqlite3 (git 제외)
├─ corpus/                ← 글·이미지 파일 (git 제외 권장, 용량 큼)
├─ inbox/                 ← 사람이 글을 넣는 투입함(파일럿용)
└─ tests/                 ← 자체 검증
```

## 3. DB 스키마 (v9 데이터모델 그대로 — 요약)

테이블: `posts`(제목분석·경로·태그 포함) / `post_paragraphs` / `post_images`(출처·재사용범위·인물/로고/글자 포함) / `reference_signals` / `keywords`+`post_keywords` / `staff` / `content_clusters` / `post_embeddings`(BLOB) / `extraction_logs` / `review_pairs`(구조만).
상세 필드는 서비스기획서 v9 §3 + 부록 A가 단일 기준. 마이그레이션은 `CREATE TABLE IF NOT EXISTS` + ensure-column(멱등) 패턴(CRM과 동일 방식).

## 4. ★ 파일럿 전략 변경 — "수동 투입"부터 (중요)

네이버 로그인 추출은 백로그다. 그래서 파일럿은 **자동 긁기 없이** 시작한다:
- 담당자가 로그인된 브라우저에서 글을 열어 **본문 텍스트를 `inbox/`에 저장(또는 붙여넣기)** + 캡처/이미지 파일 첨부
- 시스템이 그걸 읽어 **표준 양식(v9 부록 A)대로 정제·문단화·태깅·저장**
- 즉 파일럿의 검증 대상은 "긁기"가 아니라 **"정리 품질"** — 이미 시연으로 형식 합의 완료, 이제 그걸 프로그램으로.
- 자동 추출(Playwright)은 정리 파이프라인이 안정된 뒤 붙인다(백로그 → 요청 시).

## 5. 개발 차수 (클로드 코드 작업 단위)

| 차수 | 내용 | 산출물 | 위험 |
|---|---|---|---|
| **CA-1** | 뼈대: 폴더구조 + db.py(전체 스키마 생성) + 룰북 V5 최소 적재(금지어·개인정보 패턴·카테고리) | DB 파일 생성 확인 | 낮 |
| **CA-2** | 엑셀 적재(1-A): '○○ 현황' 시트만 파싱 → URL 정규화·중복제거 → posts(메타)+reference_signals 적재. 제외 탭(마케팅표·키워드·계정정보·휴일선정표) 스킵 | 적재 건수 리포트 | 낮 |
| **CA-3** | 수동 투입 파이프라인: inbox의 글 1건 → 본문 3버전(raw/clean/pub_ref 마스킹) → 문단 분리·역할 태깅 → 이미지 메타 입력(반자동) → 표준양식 저장 + **사람용 요약(A안) 출력** | 글 5건 처리 샘플 | 중 |
| **CA-4** | 린터(1-D): 금지어·단정수치·개인정보·길이 검사 + 리포트 | 린트 리포트 | 낮 |
| **CA-5** | 임베딩+클러스터링: 외부 API로 지문 생성 → BLOB 저장 → 유사군집·중복의심 태깅 | 클러스터 리포트 | 중 |
| **CA-6** | 분석 산출물: 제목 코퍼스 가이드 초안 / 갭 히트맵 / 룰북 역검증 / 참고용 구조템플릿 | 4종 리포트 | 낮 |
| (백로그) | Playwright 자동 추출(순위체크 확장) / CRM 통합 / 생성 엔진 | — | — |

각 차수 완료 시: 자체 테스트 통과 + 작업노트(MD) 남김 + 사용자 눈검수(요약 화면 확인).

## 6. 사용자(비개발자) 검수 포인트 — 차수별

- CA-2 후: "엑셀의 글 개수와 적재 건수가 비슷한가? 제외 탭이 안 섞였나?"
- CA-3 후: "요약 화면이 시연 때 본 그 형식인가? 전화번호·○○쌤이 가려졌나?"
- CA-4 후: "일부러 금지어 넣은 글이 걸리나?"
- CA-5 후: "비슷한 글끼리 묶인 결과가 감각과 맞나?"
