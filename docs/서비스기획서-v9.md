# 콘텐츠 자산화 시스템 — Phase 1 서비스 기획서 (v9)

> **Phase 1 목표**: AI 원고·이미지 "생성"이 아니라, **기존 자체제작 콘텐츠를 정확히 추출·정제·분류하는 콘텐츠 자산 DB 구축.** 생성 엔진은 별도 Phase(생성 단계)로 분리.
>
> **구축 방식(확정)**: **기존 CRM과 분리해 독립 시스템으로 먼저 제작·운영**하고, 안정화된 뒤 CRM에 통합한다. 통합 시에도 데이터 저장소는 CRM과 **별도 DB**를 유지하고, 화면·로그인만 CRM에 합친다. (기존 마케팅 순위체크와는 역할이 다르며, 해당 기능은 현재 보류)
>
> **v9 확정 3건**: (1) **추출기는 기존 순위체크의 Playwright(가상 브라우저) 방식을 확장**한다 — 순위체크가 "숫자"만 긁는다면, 자산 추출은 "본문 전체 + 이미지"까지 긁는다(동일 도구·동일 접근). (2) 글 1건의 **표준 정리 양식**을 실제 샘플로 확정(부록 A). (3) **검토용 요약(A안)과 실제 저장(B안, 생략 없음)을 분리** — 사람은 요약을 보고, 시스템은 상세를 전부 저장.
>
> **이전 반영(v8)**: 본문은 파일로만 저장(DB는 경로만), 임베딩은 외부 AI API로 생성 후 자산 DB에 저장(전용 벡터 스토어 미도입), 값 채울 경로 없는 필드 제거, 글 종류 자동분류는 기준 확정 후로 보류(길이 분류만 진행).

## 0. 이번 보완의 방향 요약

| 항목 | 기존안 | v9 반영안 |
|---|---|---|
| Phase 1 범위 | 자산화~생성~검수 전체 | **자산화(추출·정제·분류) DB로 축소** |
| 순위·조회수 | "성과 데이터" | **노출·조회 참고 신호** (문의·등록 성과 아님) |
| 글 분류 | 성과 등급 P1/P2/P3 | **다중 활용 목적 태그(usage_tags)** |
| 짧은 글 | 우선순위 낮음 | 길이는 **유형 분류값**일 뿐, 감점 없음 |
| 미노출 글 | 제외 경향 | 문체·비교군·희소키워드 참고자료로 **보존** |
| 작성자/담당직원 | 분리 | **동일인 간주**(staff_name 단일 기준) |
| 카페별 톤/포맷 | 프로파일화 | **제약 두지 않음**(카페명은 메타만) |
| 이미지 | 다운로드·저장 | **위치·역할·권한 분류 + 원본재사용/패턴참고 구분** |
| 개인정보·직원 호칭·연락처 | — | **자동 생성 기본값에서 제외** |
| 생성·발행 | Phase 1 포함 | **생성 단계로 이관** |

전제: 대상은 부서 자체제작 콘텐츠. 외부 이미지·본문은 재발행하지 않고 **참고·패턴 분석 용도로만** 사용하며, 재사용 전 권한 확인(risk_tags)으로 관리. **`계정 정보` 시트는 코퍼스 DB에 반입하지 않는다**(자격증명 성격). 각 글이 어느 계정으로 올라갔는지(`account_id`)는 **현황 시트에서 가져와 업로드 계정 식별용 메타로만** 저장한다.

## 1. Phase 1 세부 범위 재정의

### Phase 1-A. 링크 자산화 및 접근상태 진단
**목적**: 엑셀에 흩어진 카페 링크를 전수 수집하고, 자동 추출 가능 글과 수동 확인 글을 분류.

- **대상 탭 = 각 카페별 '○○ 현황' 시트만**(게시글 링크가 행 단위로 정리된 시트). 아래 탭은 **자산화 대상에서 제외**:
  - `카페 마케팅표`, `휴일키워드선정표` — 운영 계획/배정 시트(게시글 본문 아님)
  - `카페 키워드`, `카페 키워드(대기)` — 키워드 목록(룰북 ⑤시트와 중복, 별도 수집 불필요)
  - `계정 정보` — 코퍼스 미반입(자격증명 성격). 단, 각 글의 업로드 계정(`account_id`)은 **현황 시트에서** 가져와 라벨로만 사용
- 엑셀 대상 시트에서 링크 추출 → **URL 정규화**(카페ID·게시글ID 기준 표준화) → 중복 제거
- 링크별 메타 매핑: 카페명, 키워드(1·2·3차), 작성일, **담당직원(staff_name)**, 게시 계정ID(메타데이터), 원본 시트명, 원본 행번호, 통합탭 변동 순위, 조회수
- **접근상태 진단**(아래 4절 기준)으로 자동 추출 대상 / 수동 처리 대상 분리
- 운영 전제: 전체공개가 기본값 → 전체공개=자동 추출 대상, 멤버공개·권한제한=별도 분류 후 수동

### Phase 1-B. 본문·이미지 추출
**목적**: 접근 가능 글의 본문·이미지를 정확히 추출·저장.

- 추출 항목: 제목, 본문, 작성일, 조회수, 이미지
- **본문 3버전 저장**: `body_raw`(원문) / `body_clean`(정제) / `body_publishable_reference`(참고용 정제본, **개인정보·직원 호칭·연락처 마스킹** — 8절) — 8절
- 이미지 다운로드 + **순서(image_order)** + **본문 내 문단 위치(nearby_paragraph_no)** 매핑
- 실패 글은 사유와 함께 별도 큐로 분류, **추출 로그·재시도 이력 저장**
- **추출 방식 우선순위**:
  1. **모바일 페이지 파싱**(`m.cafe.naver.com`) — PC보다 구조 단순, 우선 시도
  2. **로그인 세션 기반 브라우저 자동화**(Playwright 등) — **기존 순위체크 추출기의 구조를 확장**. 순위체크는 가상 브라우저로 네이버 접속→검색→순위·조회수·댓글 확인까지 하는데, 자산 추출은 같은 방식으로 접속하되 **본문 전체 + 이미지**까지 수집. 본인 계정 세션 유지, 렌더된 본문·이미지 추출, iframe/내부 API 로딩 구조 대응. (서버 쪽 단순 fetch로는 네이버 카페가 차단됨 — 검증 완료)
  3. 실패 글(멤버공개·삭제·권한없음·차단의심·링크오류)은 상태값으로 저장

### Phase 1-C. 콘텐츠 정제·분석
**목적**: 추출본을 AI가 바로 참고 가능한 구조화 데이터로 정제. (분석이지 생성 아님)

- 문단 분리 / 소제목 분리 / 반복 CTA·푸터 탐지
- **문단 역할 태깅**(도입·공감·조건설명·절차안내·비교·사례·주의사항·CTA·마무리 등)
- **이미지 역할 태깅**(패턴 분석·제작 기준 도출 관점)
- **담당직원별 문체 프로파일 초안** 생성(staff_name 단위 집계 — 설계노트 참조)
- 카테고리별 원고 구조 분석 / 참고 신호 연결 / 활용 목적 태그 부여

**분석 산출물 — 자산을 바로 쓰이는 형태로**
- **① 유사도 클러스터링**: 본문 임베딩으로 유사 글 군집화 → 같은 키워드의 변주 패턴 확보, 사실상 복붙 글 `duplicate_suspected` 자동 태깅, 부서 내 암묵 "성공 템플릿" 가시화. (텍스트판 image_hash)
- **② 제목 코퍼스 분석**: 제목 전수를 별도 분석(길이·키워드 위치·질문형/단정형·숫자 포함 × 노출 버킷) → **제목 작성 가이드 초안** 산출. 추출 난이도 낮아 가성비 최상.
- **⑤ 콘텐츠 갭 히트맵**: 룰북 키워드 653개 × 실제 작성 글 매핑 → "검색 많은데 글 없는" 공백 키워드 / 과잉 작성 키워드 도출. **생성 단계 작업 큐 우선순위**로 직결.
- **⑥ 룰북 역검증**: 추출 글의 팩트성 문장(학점 수·조건)을 룰북 V5와 대조 → 현행과 다르면 `outdated_info_risk` 자동 부여(사회복지 신법·유보통합 등 제도 변경 영역), 글엔 반복되나 룰북에 없는 팩트는 **룰북 추가 후보로 역제안**. 코퍼스↔룰북 상호 검증.
- **⑨ 구조 템플릿 초안(채택 — 참고용 한정)**: 문단 role 시퀀스를 카테고리별로 통계 → "도입→조건설명→절차안내→CTA" 식 흐름이 자주 나타나는지 분석해 **실증 기반 "참고용 구조 템플릿 초안"** 산출. **Phase 1에서는 글 작성 제약·고정 양식으로 쓰지 않으며, 직원·AI 생성에 강제하지 않는다.** 생성 단계에서 참고 가능한 추천 흐름으로만 활용.

### Phase 1-D. 자동 검수 린터
**목적**: 생성 단계 전에, 기존 콘텐츠·신규 초안 모두에 재사용 가능한 검수 기반 구축.

- 금지어 / 단정표현 / 수치·학점 표현 / CTA 노출 / 채널별 길이 / 브랜드·전화·URL 노출 검사
- **개인정보·직원 식별정보 탐지**: 전화번호, 오픈채팅 링크, 카페 닉네임/활동명, 담당자·직원 실명, `OO쌤`·`OO멘토`·`OO팀장` 등 내부 호칭 → 검출 시 플래그(정제 단계 마스킹, 생성 단계 자동삽입 차단에 재사용)
- 검수 리포트 생성
- **룰북 V5를 단일 출처로 참조** → 생성 단계 원고에 그대로 재사용

## 2. 단계별 개발 우선순위

```
0. 파일럿(분산)             ← 1차 1곳 30~50건 기본검증 → 2차 여러 카페 총 200건 내외
1. 룰북 V5 최소 재구성          ← 검수·생성의 기준. 먼저 최소 골격 확보
2. 링크 추출/상태 진단 (1-A)   ┐
3. 본문·이미지 수집 저장 (1-B)  ┘ 1-A·1-B는 병렬 진행 가능
4. 본문 정제/문단화/이미지 매핑 (1-C)  ← 클러스터링·제목·갭·룰북역검증 포함
5. 자동 검수 린터 (1-D)         ← 이 단계 전에 룰북 V5 최소구조 필요
────────── Phase 1 종료선 ──────────
8. (Phase 1 완료·안정화 후) CRM 통합 — 화면·로그인만 CRM에 합침, 데이터는 별도 유지
6. 스타일·이미지 패턴 분석 심화  (생성 단계)
7. 생성 엔진                   (생성 단계)
```

> **구축 순서 원칙**: 위 0~5는 **CRM과 완전히 분리된 독립 시스템**으로 제작·운영한다(CRM 코드 무접점). 독립 시스템이 안정화된 뒤에야 8단계(CRM 통합)를 진행하며, 통합은 화면·로그인만 합치고 자산 데이터 저장소는 계속 분리한다.

**⑬ 파일럿 우선 (분산 방식)**: 11,000건 전체 투입 전, **추출 안정성 검증**을 목적으로 2단계 진행. (카페별 톤/포맷 분석 목적 아님)
- **1차**: 카페 1곳 30~50건으로 기본 추출 검증
- **2차**: 주요 카페 여러 곳에서 총 200건 내외로 분산 — **카페별 페이지 구조 차이** 확인

**파일럿 검수 체크리스트 (사람 확인 항목)**
- 본문이 누락 없이 추출됐는가
- 이미지 개수·순서가 맞는가
- 이미지↔문단 위치 매핑이 맞는가
- 조회수·통합탭 변동 순위가 정확히 매핑됐는가
- 접근상태 분류가 맞는가
- usage_tags / risk_tags가 과도하게 붙지 않았는가
- 전화번호·오픈채팅·담당자명·직원 호칭이 생성 참고자료(`body_publishable_reference`)에서 제외(마스킹)됐는가

**⑭ 모델 티어링**: 대량 단순 작업(문단·이미지 태깅, 상태 분류)은 저비용 모델(Haiku급), 판단 작업(문체 프로파일·구조 분석·룰북 역검증)은 상위 모델(Sonnet급)로 분리해 대량 처리 비용 통제.

**병렬/선행 규칙**: 1-A·1-B는 병렬 가능. 단 자동 검수(1-D)와 생성 엔진 진입 전에는 룰북 V5 최소 구조가 선행. 룰북 V5 전체 완성 전에도 수집(1-A·1-B)은 시작 가능.

## 3. 저장 아키텍처 (확정: 폴더 코퍼스 + 관계형 DB + 임베딩, CRM과 분리)

**이 자산 DB는 CRM과 별도로 둔다**(별도 파일). 독립 운영 중에는 물론, 향후 CRM 통합 후에도 저장소는 분리 유지 → 자산화 작업의 오류·부하가 CRM에 번지지 않음. 세 저장소가 역할을 나눠 가진다.

| 저장소 | 담는 것 | 이유 |
|---|---|---|
| **폴더 코퍼스** (파일 시스템) | 원문 JSON(게시글 원본), 정제 텍스트(clean·publishable_reference), 이미지 파일 | 대용량 비정형 데이터. **본문은 파일에만 저장**, DB는 경로만 참조 |
| **관계형 DB** (자산 전용, CRM과 별도) | 게시글 메타데이터, 키워드(다계층), 담당직원, 순위·조회 참고 신호, usage_tags·risk_tags, 문단·이미지 분류, 추출 로그, 클러스터, 임베딩, review_pairs | 검색·필터·조인·집계 |
| **임베딩(벡터)** | 게시글 단위 임베딩 | 유사도 클러스터링 + 향후 RAG 검색 기반 |

- DB의 posts는 폴더 코퍼스의 파일 경로(`body_*_path`, 이미지 `local_path`)를 참조한다. **본문 텍스트를 DB에 중복 저장하지 않는다**(용량 절감·불일치 방지). 본문 검색이 필요하면 DB 내장 전문검색(FTS) 사용 — 새 도구 불필요.
- **임베딩 생성은 외부 AI 임베딩 API**로 하고, 결과 벡터는 **자산 DB 테이블에 저장**(전용 벡터 스토어 미도입). 유사도 계산은 numpy로 처리 — 11,000건 규모에서 충분. (수십만 건으로 커지면 그때 전용 벡터 저장 검토)
- **임베딩은 Phase 1부터 저장**하되, Phase 1 활용 범위는 **유사도 클러스터링까지**. RAG 기반 원고 생성 기능 자체는 생성 단계로 이관.

## 3-0. 데이터 모델 초안

### 3-1. 테이블 관계 개요
```
staff(담당직원=작성자) ──1:N── posts ──1:N── post_paragraphs
                                    ├──1:N── post_images
                                    ├──1:N── reference_signals
                                    ├──1:N── extraction_logs
                                    ├──1:1── post_embeddings
                                    ├──N:1── content_clusters
                                    └──M:N── keywords (via post_keywords)
review_pairs ──N:1── posts (source_post_id, 선택적)
```
**신규**: content_clusters(유사도 군집), post_embeddings(게시글 임베딩). **단순화**: 작성자=담당직원 동일 간주 → `staff` 단일 기준. 게시 계정ID는 posts의 메타 필드로만 저장.

### 3-6. staff / keywords / extraction_logs / review_pairs
- **staff**: staff_id(PK), staff_name(담당직원=작성자), team, active, notes. → 문체 프로파일은 이 테이블 기준으로 집계. (게시 계정ID는 posts.account_id 메타로만)
- **keywords**: keyword_id(PK), keyword, tier(1·2·3차), category. + 연결 테이블 **post_keywords**(post_id, keyword_id, tier)로 다대다.
- **extraction_logs**: log_id(PK), post_id, attempt_no, status, error_detail, method(mobile/browser), attempted_at.
- **review_pairs**: 9절 참조.
- **content_clusters**: cluster_id(PK), category, cluster_type(near_duplicate/variation/template), representative_post_id, member_count, similarity_threshold, created_at.
- **post_embeddings**: post_id(PK/FK), vector(임베딩), model_name(임베딩 모델), dim(차원), source_field(예: body_clean), created_at. → 유사도 클러스터링에 사용, 향후 RAG 검색 기반. (Phase 1 활용은 클러스터링까지)

> **골든샘플(사람이 직접 선별한 대표 우수글) 관리는 Phase 1 필수 범위가 아님.** 추출·정제·태깅 구조가 안정화된 뒤 생성 품질 향상용 후속 작업으로 둔다. 현재는 `fewshot_candidate` 태그만 유지.

### 3-7. 설계 노트
1. **작성자=담당직원 동일 간주**: 게시 계정ID(`account_id`)는 여러 담당직원이 공유·순환하지만, 운영 기준상 각 글의 작성자는 **담당직원(`staff_name`)으로 귀속**한다. 따라서 **문체 프로파일·페르소나는 `staff_name` 기준으로 집계**하고, 계정ID는 업로드 계정 식별용 메타로만 둔다(출처: 현황 시트, `계정 정보` 시트는 미반입).
2. **키워드 다계층**: 시트가 1·2·3차 키워드를 추적하므로 keyword는 다대다(post_keywords)로 설계, tier 보존.
3. **카페별 톤/포맷 제약 없음**: 카페명은 메타로 보관하되 **글 생성 스타일을 제한하는 기준으로 사용하지 않는다.** 카페별 프로파일(`cafe_profiles`)은 Phase 1 범위에서 제외(필요 시 후속 분석 항목으로 추가 가능).
4. **usage_tags/risk_tags 저장**: 초기엔 posts 내 JSON 배열로 두되, 집계·필터 빈도가 높아지면 별도 tag 조인 테이블로 정규화(확장 여지 확보).
5. **임베딩 선저장 + 생성 단계 RAG 대비**: 게시글 단위 임베딩을 **Phase 1부터 저장**(post_embeddings)해 유사도 클러스터링에 사용한다. 생성 엔진은 향후 고정 few-shot 대신 **DB에서 같은 카테고리 + `fewshot_candidate` + 같은 담당직원 문체 글을 임베딩 유사도로 동적 검색**해 주입하는 방식이 유리(DB가 쌓일수록 품질 자동 상승). 이를 위해 Phase 1에서 category·usage_tags·staff_name 인덱스와 임베딩을 미리 갖춘다. **단, RAG 기반 원고 생성 기능 자체는 생성 단계로 이관** — Phase 1은 임베딩 저장 + 클러스터링 활용 기반까지만.

## 5. 노출·조회 참고 데이터 기준

**원칙**: 순위·조회수는 **문의·등록 성과가 아니라 노출·조회 참고 신호**. 순위가 높은 이유(카페 노출력·계정 영향·키워드 경쟁도·구조·제목·이미지·타이밍·로직 변화 등)를 글 단위로 분리 판단할 수 없으므로 성과로 단정하지 않음.

- **사용 데이터**: 통합탭 변동 순위, 조회수, 노출 여부, 작성일, 카페명, 키워드, 담당직원(staff_name), 원본 시트명·행번호
- **제외/후순위**: 최근 순위, 노출 유지일, 키워드 난이도, 발행 후 경과일, 문의 발생·등록 전환·매출 기여도(현재 데이터로 신뢰 확보 불가)
- **버킷화**: rank_bucket = Top3/Top10/Top30/Other/Not Exposed/Unknown. view_bucket = 실제 분포 확인 후 **분위수 방식**(상위10%·상위30%·중간·하위·Unknown).
- 조회수 해석: 노출·클릭유도·관심도 가능성 참고 신호일 뿐, "조회수↑ = 문의↑/등록↑/품질↑"로 해석 금지.

## 6. 활용 목적 태그 기준 (다중 부여)

## 6-2. Phase 1 추가 산출물 — 자산의 즉시 활용물

자산화의 부산물로 다음 산출물을 Phase 1 안에서 함께 생성한다. 모두 기존 코퍼스·태그·신호를 재조합하는 것이라 추가 비용이 낮다.

| 산출물 | 근거 데이터 | 활용 |
|---|---|---|
| **제목 작성 가이드 초안** (②) | posts 제목 분석 필드 × rank/view 버킷 | 클릭 유도 제목 패턴 |
| **콘텐츠 갭 히트맵** (⑤) | keywords(653) × posts × reference_signals | 생성 단계 작업 큐 우선순위 |
| **룰북 추가 후보 리스트** (⑥) | 코퍼스 팩트 문장 × 룰북 V5 대조 | 룰북 V5 보강 |
| **참고용 구조 템플릿 초안** (⑨, 채택) | post_paragraphs role 시퀀스 통계 | 생성 단계 **추천 흐름**(강제 아님) |
| **게시글 임베딩** | body_clean 벡터화 | 유사도 클러스터링 · 향후 RAG 검색 기반 |

> 구조 템플릿은 참고용으로만 산출(작성 강제 아님). 제목 가이드·갭 히트맵·룰북 후보는 가성비가 높아 기본 포함.

## 7. 이미지 권한·역할 분류 기준

핵심 구분: **"원본 파일 재사용"(새 글에 그대로 다시 업로드) ≠ "패턴 참고"(역할·구도·배치·정보 구성 방식만 참고).** 후자는 외부 이미지라도 가능.

- **분류 축**: 출처(image_source_type) + 재사용 범위(reuse_scope) + 역할(image_role) + 유형(image_type)
- **이미지 태그**:
  - `internal_image_asset` — 내부 제작 이미지로 추정
  - `image_reuse_allowed` — 내부 제작물 등 원본 재사용 가능성 있음
  - `image_pattern_only` — 원본 파일은 재사용하지 않고 역할·구도·배치 패턴만 참고
  - `image_rights_review` — 이미지 권한 확인 필요
  - `external_image_risk` — 외부/출처불명 이미지 가능성
- **기본 처리**: 내부제작·AI제작 → `image_reuse_allowed` 또는 패턴 참고 가능. 외부이미지·출처불명·캡처 → `image_pattern_only` + `external_image_risk`(+ 필요 시 `image_rights_review`). 인물 포함 → `contains_person=true` + 초상권 확인 대상.
- `do_not_reuse_image`처럼 의미가 모호한 표현은 사용하지 않고, 위 태그로 대체.

## 8. 본문 저장·길이 기준

- **3버전 저장(모두 폴더 코퍼스의 파일로 저장, DB는 경로만)**:
  - `body_raw` — 원문 그대로
  - `body_clean` — 줄바꿈·반복문구·특수문자·카페 공통문구 정리(분석용)
  - `body_publishable_reference` — 리라이팅·생성 참고용 정제본. **개인정보·직원 식별정보 마스킹**: 전화번호, 오픈채팅 링크, 카페 닉네임/활동명, 담당자·직원 실명, `OO쌤`·`OO멘토`·`OO팀장` 등 내부 호칭을 치환/제거. → 이 버전이 few-shot·리라이팅 참고에 쓰이므로, 생성 엔진이 개인정보를 학습·복제할 여지를 원천 차단.
- **길이는 분류값**: `content_length_type`(short/medium/long)를 Phase 1에서 자동 부여. `length_penalty` 미사용. 짧은 글도 도입부·CTA·자연스러운 말투 참고자료로 보존. **글 종류(`content_use_case`) 자동분류는 분류 기준 확정 후 반자동으로 부여(Phase 1 보류).**

## 8-2. 생성 단계 자동삽입 금지 정보 (Phase 1에서 미리 정의)

생성 엔진은 아래 정보를 **기본값에서 자동 삽입하지 않는다.** 직원이 발행 전 직접 추가하거나, 사용자가 명시적으로 요청한 경우에만 삽입.

| 항목 | 저장 | 생성 기본값 |
|---|---|---|
| 담당직원명 | 내부 메타로 저장 | 제외 |
| 게시 계정ID | 메타로 저장 | 제외 |
| 카페 닉네임/활동명 | 필요 시 저장 | 제외 |
| 전화번호 | 추출돼도 참고본엔 마스킹 | 직원이 직접 추가 |
| 오픈채팅 링크 | 추출돼도 참고본엔 마스킹 | 직원이 직접 추가 |
| 직원 호칭(OO쌤/멘토/팀장) | — | 요청 시에만 |

- **기본 CTA는 일반형으로 생성** (예: "본인 조건에 따라 달라질 수 있으니 상황에 맞춰 확인해보시는 게 좋아요.")
- 자동 생성에서 제외할 표현 예: "○○쌤에게 문의주세요" / "○○멘토가 안내드립니다" / "○○팀장 오픈채팅으로 연락주세요" / "닉네임 ○○에게 쪽지주세요"
- Phase 1-D 린터가 이 항목들을 탐지하도록 구성되어 있어, 정제(마스킹)와 생성 차단에 동일 규칙을 재사용.

## 11. 생성 단계(다음 Phase)로 이관할 항목

> 명칭 정리: 여기서의 "생성 단계"는 **AI 원고·이미지 생성 엔진 개발 단계**로, 콘텐츠 자산 DB(Phase 1)와 별도로 분리한다.

- AI 신규 원고 생성 엔진
- AI 이미지 생성 엔진 / 문단별 이미지 자동제작
- **정보성 이미지 템플릿 3종** (텍스트 많은 정보성 이미지는 AI 생성보다 템플릿 기반이 적합):
  1. **조건 정리 카드** — 응시자격 / 학력별 조건 / 필요학점
  2. **절차 안내 카드** — 수강신청 → 학점인정 → 실습 → 시험 준비 순서
  3. **비교 카드** — 고졸/전문대졸/대졸, 온라인/오프라인, 기간/비용 비교
  → **분위기형 이미지만 AI 생성**으로 처리하고, 정보성 이미지는 템플릿 기반 제작.
- 페르소나 로테이션 생성 (담당직원 문체 기반)
- 채널별 완성 원고 자동 생성 (개인정보·직원 호칭·연락처 자동삽입 제외 — 8-2절)
- 자동 발행
- 블라인드 테스트 기반 무검수 전환
- Before/After 데이터 기반 생성 품질 고도화
- 골든샘플(사람 선별) 관리 — 자산 구조 안정화 후 추가
- *(스키마만 선반영)* RAG 방식 동적 few-shot 검색 / 팩트 출처 인라인 표기 / 검수 UI(=review_pairs 자동 축적) / 문체 재현율 정량 지표 — Phase 1은 인덱스·필드만 준비, 구현은 생성 단계

> 아래는 실제 게시글(임상심리사2급 응시자격 글)을 v9 설계대로 정리한 결과. **사람 검토용 요약(A안)**은 이 중 제목·문단역할·이미지·태그만 추리고, **시스템 저장(B안)**은 아래 전체를 테이블에 저장한다.

**posts** — post_id, URL(원본/정규화), cafe_name(공준모), account_id(평생교육기관67·메타), publish_date(2026-01-06), source_sheet/row, title, 제목분석(길이 긺/질문형/숫자 있음/키워드 앞), body_*_path(3버전 파일 경로), content_length_type(long), content_use_case(보류), cluster_id(후속), extraction_status(성공), usage_tags(style·long_form·image_layout·cta_reference), risk_tags(fact_review_needed·image_rights_review)

**본문 3버전(파일)** — `body_raw`(원문 그대로) / `body_clean`(빈줄·특수문자·중복 정리) / `body_pub_ref`(개인정보 마스킹 — 본 글은 본문에 연락처 없어 clean과 동일, 연락처는 하단 배너 이미지에 존재→이미지에서 처리)
