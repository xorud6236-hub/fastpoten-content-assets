# 개발 기획서 v2 — 브랜드 자산 팩토리 (트랙제)

> v1(직원용 전제, docs/archive-개발기획서-v1.md)을 **브랜드 자산 팩토리** 방향으로 재편.
> 0층(자산창고)의 설계는 서비스기획서 v9 그대로 — 변경 없음. 그 위에 팩토리 층을 트랙으로 얹는다.
> 클로드 코드 작업 지시서 겸용. 사용자는 비개발자 — 결정 질문은 일상 언어로.

## 0. 전체 그림 (5층)

```
[0층] 자산창고        재료: 검증된 글·문체·룰북·키워드653          ← 트랙 A
[1층] 원고 생성       갭히트맵→창고RAG→룰북 팩트→초안→린터→승인    ← 트랙 C
[2층] 멀티포맷        승인 글→JSON→카드뉴스 템플릿(PNG)+캡션       ← 트랙 B
[3층] 발행           워드프레스 자동(승인 후) / 인스타(기성품 우선)  ← 트랙 C
[4층] 성과 수집       서치콘솔·GA·인스타 인사이트 주간 리포트        ← 트랙 D
[5층] 자기 개선       성과·수정기록→다음 생성에 자동 반영            ← 트랙 D
```
설계 근거: Mirr(참고형 — 뇌는 자체, 손발은 기성품) + HOC 사례(JSON+템플릿, 메모리 루프).

## 1. 트랙·차수

### 트랙 A — 자산창고 (서비스기획서 v9 = 단일 기준, 변경 없음)
| 차수 | 내용 | 위험 |
|---|---|---|
| CA-1 | 뼈대: 폴더구조 + DB 전체 스키마 + 룰북 최소 적재(금지어·개인정보 패턴·카테고리) | 낮 |
| CA-2 | 카페 엑셀 적재: '○○ 현황' 시트만 → URL정규화·중복제거 → posts+참고신호 | 낮 |
| CA-2b | **블로그 글 링크 수집(신설)**: 시트가 없으므로 블로그 자체에서 글 목록 확보 — ①공개 글 목록(RSS 등)에서 자동으로 뽑거나 ②사용자가 링크 목록 파일을 inbox에 넣으면 적재. 메타(키워드·담당)는 나중에 보충 가능(비워두고 시작) | 낮 |
| CA-3 | 수동 투입 파이프라인: inbox 글 1건 → 3버전(마스킹)→문단·역할→이미지 메타→표준양식 저장+요약(A안) 출력 | 중 |
| CA-4 | 린터: 금지어·단정수치·개인정보·길이 + 리포트 | 낮 |
| CA-5 | 임베딩+클러스터링(외부 API+numpy) | 중 |
| CA-6 | 분석 산출물: 제목 가이드/갭 히트맵/룰북 역검증/구조템플릿(참고용) | 낮 |

### 트랙 B — 카드뉴스 생성기 (트랙 A와 병행 가능)
| 차수 | 내용 | 위험 |
|---|---|---|
| CB-1 | 조건 카드 1종 시제품: AI는 내용 JSON만 → HTML 템플릿 → PNG 렌더. 룰북 팩트로 샘플 3장 → **사용자 퀄리티 판정** | 중 |
| CB-2 | 3종 완성(조건/절차/비교) + brand.css 브랜드 스타일 통일 | 낮 |

### 트랙 C — 워드프레스 SEO 파이프라인 (CA-5 이후)
| 차수 | 내용 | 위험 |
|---|---|---|
| CC-1 | 원고 생성: 갭히트맵이 주제 결정 → 창고에서 유사 글 검색(RAG) → 룰북 팩트 삽입(출처 주석) → SEO 롱폼 초안 → 린터 → **승인 큐** | 중 |
| CC-2 | 발행: 승인분 워드프레스 자동 업로드(초안→발행) + 카드뉴스 본문 삽입. 첫 실행은 드라이런 | 중 |
| CC-3 | 주 2편 × 4주 파일럿 운영 | 낮 |

### 트랙 D — 성과·자기 개선 루프 (CC 발행 시작 후)
| 차수 | 내용 | 위험 |
|---|---|---|
| CD-1 | 주간 성과 리포트: 서치콘솔·GA·인스타 인사이트 수집 → 글별 실측 성과 기록 | 중 |
| CD-2 | 루프: 성과 발견("질문형 제목 클릭률↑" 등)+사람 수정기록(review_pairs) → 다음 생성 프롬프트에 자동 반영 | 중 |

## 2. 순서 원칙
- 기본 흐름: CA-1 → (CB-1 병행 가능) → CA-2/2b/3 → CA-4/5 → CC-1 → CC-2/3 → CD.
- CB-1은 언제든 먼저 가능 — 눈에 보이는 결과물이 빨리 나와 방향 검증에 좋음.
- 각 차수 = planner(분해)→implementer(구현)→reviewer(검토), 완료 시 작업노트(docs/notes/)+사용자 눈검수.

## 3. 사용자 결정 대기 목록
1. **워드프레스 도메인**: 기존 회사 홈페이지에 얹기 vs 신규. → 착수 전 **서치콘솔 10분 확인**(기존 유입 있으면 얹기가 유리) — CC-1 전까지만 결정.
2. **인스타 발행 방식**: 기성품(Mirr류) vs 직접. → CB 산출물 나온 뒤 결정(기성품 우선 검토).
3. **첫 작업 순서**: CA-1부터 vs CB-1부터(병행도 가능).

## 4. 차수별 사용자 눈검수 포인트 (일상 언어)
- CA-2/2b 후: 엑셀·블로그의 글 개수와 적재 건수가 비슷한가? 계정정보 탭이 안 섞였나?
- CA-3 후: 요약 화면이 합의한 그 형식인가? 전화번호·OO쌤이 가려졌나?
- CB-1 후: **카드 3장의 퀄리티가 "브랜드에 올릴 만한가"?** (이 판정이 트랙 B의 관문)
- CC-1 후: 초안 글에 팩트 출처 주석이 붙어 있고, 승인 전엔 절대 발행 안 되는가?
- CC-2 후: 드라이런 글이 워드프레스 "초안"으로만 올라갔는가?
- CD-1 후: 주간 리포트가 "무엇을 더 쓰고 무엇을 바꿀지"를 말해주는가?
