# CA-6 — 룰북 팩트 적재 + 검수 화면 (계획, 2026-07-15)

목적: 룰북 엑셀 `② 팩트 룰북`(공통 16 + 개별 35 = 51건)을 창고에 넣고, **51건 전부를 '미확인' 상태로** 시작해 화면에서 하나씩 확인·수정한다. 근거: 헌장 불변 6(팩트는 룰북에서만) · 진행지점 "후속: 룰북 팩트 시트 적재(CA-6)".

## 0. 먼저 — 이 작업이 정말 필요한가 (YAGNI)

필요하다. 두 가지가 확인됐다.
1. 팩트 룰북이 없으면 원고 도우미를 만들 수 없다. 창고(0층)는 "어떻게 쓰나"만 알려주고 "무엇이 사실인가"는 룰북만 답한다(불변 6).
2. **이미 틀린 게 나왔다.** 사회복지사2급 실습시간이 룰북 안에서 서로 어긋난다 — 요건란 "160시간 필수" / 주의메모 "2020.1.1 이전은 120시간" / FAQ "120시간 필수" / 개별팩트 29행 "실습120시간". 사용자(도메인 전문가) 확인 결과 **2020.1.1 이전 120 / 이후 160**이 맞다. 즉 FAQ와 29행은 조건을 빼먹고 단정해서 **신규 대상자 대부분에게 오답**이다. 사회복지는 개별 팩트 35건 중 8건으로 최대 주제다.

더 작은 버전은? "엑셀만 고치고 DB는 나중에" → 누가 언제 무엇을 확인했는지가 남지 않고, 엑셀 왕복은 실제로 안 하게 된다(사용자 결정 3). 채택 안 함.

**전제(사용자 증언): "AI가 만들어준 걸 최소한의 검수만 했다. 얼추 맞으면 넘겼다."** → 51건 **전부 미확인이 기본값**. "일부만 의심"이 아니라 "전부 미확인"이 출발점이다.

- 원고 생성기(미검수 → `[검수:]` 표시의 **적용**). 생성기가 아직 없다. 이번은 데이터·화면까지.
- `③ 매칭 규칙`·`⑤ 키워드 목록` 시트(CA-6 범위 밖, 계속 이후 차수).
- 화면 상세 설계(designer 몫). 여기서는 "어떤 화면이 필요한지"까지.
- ~~화면에서 고친 값을 엑셀 파일에 되쓰기(내보내기). 필요해지면 후속.~~ → **2026-07-15 사용자 결정으로 범위 안(6차)**: "화면이 최종본이고 **여기서 엑셀로 출력한 걸 어디서든 자유롭게 사용**하는 쪽이 업무흐름에 맞다". 내보내기가 없으면 D1(화면이 최종본)이 업무에서 반쪽이 된다 — 검수 결과를 창고 밖으로 못 꺼내므로.

- **새 의존성 없음**: openpyxl(적재) · http.server(뷰어) 모두 이미 쓰는 것.

- 새 테이블 2개(팩트 본체 + 수정 이력)는 §4에서 정당화.

## 3. ★ 핵심 난제 — 엑셀이 원천인가, 창고가 원천인가

`load_rulebook.py`는 `DELETE FROM ... WHERE source_version=?` 후 INSERT 하는 구조다(60·86·114행). 팩트를 같은 방식으로 넣으면, **화면에서 검수·수정한 내용이 있는 상태에서 누가 적재를 다시 돌리는 순간 전부 날아간다.** 엑셀은 63행에 "▼ 아래부터 개별 팩트를 추가해주세요 ▼"가 있어 **앞으로도 계속 늘어나는 시트**라 재적재는 반드시 일어난다.

헌장 불변 9: "간결(YAGNI) — 단 입력검증·마스킹·**데이터 손실 방지는 간결화 대상 아님**". 따라서 이 문제는 반드시 푼다.

선택지:

| 안 | 방식 | 장점 | 단점 |
|---|---|---|---|
| **A. 창고가 최종본, 엑셀은 새 항목 입구** (권장) | 재적재는 **새 항목만 추가**. 기존 항목은 절대 덮어쓰지 않음. 엑셀 쪽 내용이 달라진 항목은 "엑셀이 바뀐 것 같은 항목 N건"으로 **알려만 주고** 사람이 화면에서 처리 | 검수 결과 손실 0. 자동 덮어쓰기 0. 엑셀은 계속 새 팩트 넣는 입구로 그대로 씀 | 엑셀에서 고쳐도 화면에 자동 반영 안 됨(알림만) |
| B. 엑셀이 최종본, 검수 상태만 별도 보존 | 엑셀 고치면 바로 반영 | 화면에서 고친 값이 재적재 때 사라짐 → 사용자 결정 3("수정은 화면에서")과 정면 충돌 |
| C. 확인 끝난 항목만 덮어쓰기 제외 | 절충 | 미확인 51건 = 대부분이 여전히 날아감. 지금 상황에선 B와 거의 같음 |

**권장 A.** 이유: 사용자가 "수정은 화면에서"로 정했으므로 화면이 최종본이어야 앞뒤가 맞고, 데이터 손실 위험이 0이다.

항목 식별 방법(A안 구현 시): 엑셀 `No.`는 행이 지워지거나 재번호되면 어긋나므로 보조로만 쓴다. 실제 식별은 **공통=카테고리명 / 개별=상품·키워드명**(사람이 부르는 이름). 여기에 엑셀 원본 내용의 지문(해시)을 함께 저장해 "엑셀이 바뀐 것 같음"을 감지한다.

## 4. 데이터 구조 (멱등 ensure, 컬럼 삭제 없음)

**테이블 1개 vs 2개**: 공통(A)과 개별(B)은 열이 다르지만(A: 요건·학점·기간·단축·FAQ·주의사항 / B: 핵심팩트·학력별경로·강조포인트·우선순위), **한 테이블 `rulebook_facts` + 종류 구분(공통/개별) + 열 합집합**으로 간다. 이유: 검수 상태·화면·나중의 프롬프트 주입이 전부 한 경로로 끝난다. 두 테이블이면 그 세 가지를 두 벌씩 만들게 된다.

- `rulebook_facts` — 종류(공통/개별), 엑셀 No.(참고), 카테고리, 항목명, 각 내용 칸(엑셀 열 그대로 텍스트 보관), **검수 상태 · 확인 시각 · 검수 메모**, 원본 지문, source_version.
  - FAQ TOP3처럼 한 칸에 3개가 든 셀은 **쪼개지 않고 그대로** 보관한다(쪼개는 규칙이 불확실하고, 프롬프트에는 어차피 글로 들어간다).
  - 실제 열 이름 문자열은 구현 착수 시 엑셀 헤더를 다시 읽어 확정한다(이 문서의 열 목록은 과제 브리프 기준).
- `rulebook_fact_edits` — 수정 이력(어느 항목·어느 칸·이전 값·새 값·시각). 새 테이블을 만드는 이유: 이전 값을 본체 행에 넣으면 컬럼이 2배가 되고 두 번째 수정부터 못 버틴다. 되돌리기와 "엑셀 원본이 뭐였지"를 동시에 해결한다.
- `db.py`의 `EXPECTED_TABLES`에 2개 추가(14 → 16).
- 기존 컬럼 삭제·변경 없음. `CREATE TABLE IF NOT EXISTS` + `ensure_column`.

**개인정보(불변 1)**: 룰북 팩트는 원고에 그대로 들어간다(엑셀 2행 안내문: "이 데이터가 AI 프롬프트에 직접 들어갑니다"). 적재·화면 저장 모두 `masking.py`를 통과시켜 **가려진 값으로 저장**한다(예: 담당자 호칭이 섞여 들어오면 가림). 불변 1은 간결화 대상이 아니다.

## 6. ★ 뷰어가 처음으로 쓰기를 한다 — 위험과 대응

지금까지 뷰어는 읽기 전용(`do_GET`만)이라 최악의 경우도 "화면이 안 뜬다"였다. 4차부터는 잘못하면 **데이터가 바뀐다**. 4차에 반드시 포함할 것:

1. **실수 클릭 되돌리기** — 모든 수정을 이력에 남기고 항목마다 '되돌리기'. (지운 값은 사라지지 않는다)
2. **동시 실행** — 추출 배치가 같은 창고 파일에 쓴다. `PRAGMA busy_timeout = 60000` 적용(선례: `src/bulk_extract.py` 71행, 구글드라이브 동기화 잠금 때문). 저장은 짧게 끊어서.
3. **다른 사이트가 몰래 시키는 것 차단** — 브라우저에 열어둔 다른 사이트가 `localhost:8765`로 저장을 밀어 넣을 수 있다. 요청 출처(Origin/Host) 검증 + 127.0.0.1 바인딩 유지.
4. **입력 검증** — 항목 번호는 정수 강제, 고칠 수 있는 칸은 정해진 목록에서만, 값 길이 상한. (기존 화면들이 쿼리값을 안전 리터럴로 좁히는 방식과 같은 결)
5. **저장 후 주소 재이동(303)** — 새로고침으로 같은 저장이 두 번 들어가는 것 방지.
6. **개인정보 가림 통과** — §4 마지막 항.
7. 저장은 **되돌릴 수 있는 것만**. 항목 삭제 기능은 넣지 않는다(숨김만, 불변 9).
8. **가림 규칙이 바뀌면 팩트도 다시 검사**(사용자 결정 2026-07-20, reviewer 발견). 지금 팩트 51건은 **창고에 넣던 순간의 규칙**으로만 가려졌고, 직원 이름 목록이 늘어도 다시 안 걸린다(추가 전용 적재라 기존 항목을 덮지 않으므로). 글 목록은 이미 이 문제를 '가림 규칙 표'로 다룬다(헌장 현재 결정 10) — **팩트에도 같은 장치를 4차에서 붙인다**. 자동 재계산은 하지 않고 "다시 검사 필요"로 뜨게만 한다(글 목록과 동일). 3차 시점 실측 히트 0건이라 지금 당장의 누출은 없다.

## 7. 규칙 확장 명문화 (이번 차수는 문서까지)

- **항목명 바뀜 감지 추가**(reviewer 발견 3 → 사용자 결정 "대비해둔다"). 사용자 원문 이유: *"구조를 내가 설계했던 게 아니라 사용하면서 수정할 수도 있을 것 같아"* — **룰북 항목명 자체가 AI가 지은 것이라 검수하며 바꿀 가능성이 높다**(항목명도 검수 대상). 동작: 내용 지문이 같은데 이름만 다르면 `'옛이름' → '새이름'`으로 보고하고 중복 INSERT 안 함. 자동 rename은 하지 않음(추정이지 확정이 아님).
- **마스킹 오탐 위험 실측 완료**: 실제 직원 이름 82건 × 실제 팩트 시트 450칸 → **0칸 변경**(오탐 없음). 회귀 테스트로 고정. staff에 이름 아닌 값(1글자 조각·점 붙은 이름·업체명 등)이 섞인 건 별건 → BACKLOG 6. **실제 값은 문서에 옮겨 적지 않는다 — 공개 저장소, 불변 1.**

### 4차로 넘긴 미결

- **화면에서 값을 고친 항목은 이름 바뀜 감지가 안 된다** — 검수하며 내용을 수정하면 엑셀과 내용이 달라져 "이름만 바뀐 항목"으로 못 알아보고 새 항목으로 들어간다. 넓게 추정하면 반대로 엉뚱한 항목을 같다고 볼 위험이 있어 지금은 안전한 쪽(감지 안 함)을 택했다. 4차 검수 화면에서 다룰지 판단.
- **빈 섹션 예외가 룰북 적재 전체를 멈춘다**(팩트뿐 아니라 카테고리·금지어까지 같은 실행). 판단: **현행 유지** — 개별 팩트를 전부 비우는 건 실수이고 그때 크게 실패하는 게 맞다. 실제로 그 상황이 오면 그때 분리.
