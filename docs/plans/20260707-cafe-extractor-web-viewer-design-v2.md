# 화면 명세 v2 — 품질 확인 뷰어 상세 화면 "좌우 원문 대조" 개편

작성: 2026-07-07 / 화면설계(designer). **구현 아님, 화면 명세만.**
개정 대상: `docs/plans/20260707-cafe-extractor-web-viewer-design.md`(v1)의 **"화면 A — 글 1건 상세"만** 교체한다.
v1의 씨앗 패턴(§1)·목록 화면(§3)·화면 폭 대응(§4)·접근성(§5)은 **그대로 유효**하다(아래 "바뀌지 않는 것" 참고).

## 왜 바꾸나 (사용자 피드백)
뷰어의 목적은 "원본 게시물과 대조해 정리 품질을 눈으로 보는 것"인데, v1 상세는 **가림 처리된 정리본(문단)만** 떠서 "가공 전과 후"를 나란히 볼 수가 없다. 그래서 상세 화면을 **좌우 2단 대조**로 바꾼다.
- **왼쪽 = 가공 전 원문의 원래 흐름** (문단으로 다시 쪼개지 않고 원본 줄바꿈 유지), **단 개인정보만 가림.**
- **오른쪽 = 우리가 정리한 결과** (문단 분리 + 역할 태그, 지금 v1의 그 화면).
- 한 화면에서 좌우가 대조되면 성공. **마스킹은 양쪽 다 유지**(가림 자리 노랑 하이라이트도 양쪽 유지).

---

## 불변 1(마스킹) — 이 개편이 반드시 지키는 것 ★ (재확인)
왼쪽 패널이 새로 생기면서 원문 유출 위험이 커지므로 **가장 먼저 못 박는다.**

- **왼쪽 패널은 `body_raw`(원문 원본 파일)를 절대 그대로 화면에 내보내지 않는다.** 서버가 원문 파일을 읽어 **`masking.py`(`mask_text`)를 통과시킨 결과 문자열만** 내보낸다. 통과 전 원본 문자열은 어떤 경로로도 브라우저에 나가지 않는다.
- **`body_clean`(마스킹 전 정제본)도 그대로 노출 금지.** 왼쪽에 body_clean 파일 내용을 마스킹 없이 띄우면 안 된다. (왼쪽 소스 확정은 결정 1.)
- 마스킹은 오른쪽이 쓰던 것과 **같은 패턴·같은 직원 이름 목록**을 쓴다(`load_regex_patterns` + `load_staff_names`). 한쪽만 다른 패턴을 쓰면 대조가 어긋난다.
- 왼쪽은 **원본 줄바꿈/흐름을 유지**하되(문단 재분할 금지), 마스킹된 자리는 오른쪽과 똑같이 노랑 하이라이트 + 대괄호 라벨(`[담당자]`·`[가림]`)로만 보인다. 가려지기 전 실제 값(전화번호·이름)은 화면에 다시 띄우지 않는다(v1 결정 3 유지).
- 화면에 HTML로 그릴 때 원문의 특수문자는 반드시 이스케이프(escape) 후 출력한다(원문 속 태그가 화면 구조를 깨거나 스크립트로 실행되지 않게).
- 이미지 규칙은 v1과 동일(안전 분류만 썸네일, 그 외 자리표시).

**구현 메모(설계 근거만, 방식은 implementer):** 지금 `mask_type_counts()`는 body_clean을 읽어 **건수만** 세고 마스킹된 본문 텍스트는 버린다. 왼쪽 패널은 **마스킹된 본문 텍스트 자체**가 필요하므로, 원문 파일을 읽어 `mask_text`로 가린 **텍스트를 반환하는 처리**가 새로 필요하다(원본 문자열은 반환·출력 금지).

## 새 배치 — 상세 화면 (위 → 아래)

1. **상단 고정 띠(스크롤해도 붙어 있음)** — v1과 동일: 왼쪽 `← 목록으로` · 가운데 글 제목(길면 한 줄 말줄임) · 오른쪽 추출 상태 배지.

2. **글 정보 블록(가로 전체)** — v1과 동일: 제목(큰 글자) 아래 한 줄 메타 `카페명 · 게시판 · 담당 · 작성일 · 조회수 (참고 신호) · 길이유형`. 조회수엔 반드시 `(참고 신호)`(불변 3).

3. **대조 안내 한 줄(가로 전체, 정보 블록 바로 아래)**: 두 패널이 무엇인지 짧게 알려 준다.
   문구: `왼쪽은 가공 전 원문 그대로의 흐름(개인정보만 가림), 오른쪽은 우리가 문단·역할로 정리한 결과입니다.`

4. **좌우 2단 대조(데스크탑, 이 화면의 핵심)** — 좌우 폭 **대략 반반(각 ≈50%)**. 왼쪽을 읽으며 오른쪽 정리가 맞는지 눈이 오가는 게 가장 흔한 동작이므로 균등 배치.
   - **왼쪽 패널 — 제목 `가공 전 원문 (개인정보만 가림)`**
     - 제목 옆 **안내 배지**: `개인정보만 가린 원본 흐름` (노랑 note 계열). 사용자가 "원문이 통째로 유출된 것"으로 오해하지 않게, 배지에 커서를 올리면(또는 배지 아래 작은 글씨로) 보조 문구: `원본 파일을 그대로 띄운 게 아니라, 전화번호·이름 등 개인정보만 가린 사본입니다.`
     - 내용: **원본 줄바꿈/흐름 그대로**의 한 덩어리 글(문단 카드로 쪼개지 않음). 가려진 자리는 **노랑 하이라이트 + 대괄호 라벨**.
     - 배경: 한 장의 문서처럼 보이게(테두리 있는 흰 종이 한 장). 오른쪽의 "카드 여러 장"과 시각적으로 대비돼 "이쪽은 안 나눈 원본, 저쪽은 우리가 나눈 것"이 직관적으로 읽히게.
   - **오른쪽 패널 — 제목 `우리가 정리한 결과 (문단·역할)`**
     - 내용: **v1의 문단 스트림 그대로.** 각 문단마다 [역할 태그] + [📌 팩트 / 📣 안내 표시(있을 때만)] + 가림본 문단 글(노랑 하이라이트). `배경설명`/미분류 문단은 회색 태그 + `자동 분류 실패 — 확인 필요`(v1 유지).
   - **정렬·스크롤**: 결정 2. (기본 권장: 두 패널 각자 스크롤 — 왼쪽을 내려도 오른쪽은 제자리.)
   - **문단 경계를 원문에 잇는 색 마커**: 결정 3. (지금 데이터로는 정확히 못 이으므로 이번엔 문단 번호 표시만 권장.)

5. **대조 아래(가로 전체) — 부가 정보 두 블록을 나란히**: v1에서 오른쪽 옆칸(side)에 있던 것을 **대조 2단 아래로 내린다.** 대조가 화면 폭을 반반 쓰므로 옆에 낄 자리가 없다. 탭으로 숨기지 않고(검수 흐름 끊김 방지) 스크롤하면 바로 보이게.
   - **개인정보 가림 결과**: 가려진 항목을 **종류별 건수**로 + 총 건수 배지. 0건이면 안내 문구(아래). (v1과 동일 내용, 위치만 이동.)
   - **이미지 분류 (N개)**: 이미지마다 카드 — 분류 태그 + 재사용 배지(가능/권리확인/원본금지) + 인물 포함 표시. 안전 분류만 썸네일, 그 외 자리표시. (v1과 동일 내용, 위치만 이동.)

## 품질 판단 시각 장치 (개편 반영)
- **좌우 대비 자체가 핵심 장치**: 왼(안 나눈 원본 흐름) ↔ 오(우리가 나눈 문단). "원문의 이 덩어리를 우리가 이렇게 잘랐구나"를 눈으로 대조.
- **가림 누락 감지(양쪽)**: 가려진 자리만 노랑 하이라이트. **하이라이트 없이 맨숫자 전화번호가 왼쪽/오른쪽 어디든 보이면** 마스킹 누락이므로 사용자가 즉시 알아챈다. (왼쪽은 원문 흐름이라 오른쪽 정리 과정에서 놓친 개인정보가 왼쪽엔 남아 보일 수 있음 → 이중 확인 효과.)
- **문단 역할·확인 필요**: 오른쪽 패널에서 v1 그대로.
- **원본 대비 정리 결과의 누락·왜곡**: 왼쪽에 있는 문장이 오른쪽 어느 문단에도 안 들어갔으면 "정리 중 빠졌다"를 눈으로 발견(자동 표시는 이번 범위 아님 — 결정 3 참고).

---

## 상태 4종 (개편 반영)
- **비었을 때(정리된 문단 없음)**: 추출은 됐으나 오른쪽에 문단이 하나도 없는 글 → 오른쪽 패널 자리에 `아직 문단으로 정리되지 않았습니다. (추출은 됐지만 정리 결과가 비어 있어요.)` 왼쪽 원문 흐름은 있으면 그대로 보여 준다.
- **왼쪽 원문이 없을 때(새 상태)**: `body_raw` 파일이 없어 가공 전 흐름을 만들 수 없는 글 → 왼쪽 패널 자리에 `가공 전 원문 파일이 없어 원본 흐름을 보여줄 수 없습니다. 오른쪽 정리 결과만 확인하세요.` 오른쪽은 정상 표시. (양쪽 다 없으면 아래 "비었을 때"+이 문구가 각 패널에.)
- **불러오는 중**: (서버가 바로 그려 대개 안 보임) 잠깐 걸리면 `불러오는 중…` 한 줄. (v1 유지.)
- **실패했을 때** — v1 유지:
  - (a) 추출 실패 글: 대조 2단 대신 상태 배지 + 실패 사유 그대로(삭제된 글/비공개 게시판/로그인 필요/기타) + `이 글은 다시 추출을 돌린 뒤 확인할 수 있어요.`(안내만, 버튼 없음).
  - (b) 없는 주소: `그런 글이 없습니다.` + `← 목록으로`.
- **성공했을 때**: 좌우 대조 + 아래 부가 블록 전체 정상 표시.

## 문구 (실제 문장 — 그대로 사용)
- 대조 안내 줄: `왼쪽은 가공 전 원문 그대로의 흐름(개인정보만 가림), 오른쪽은 우리가 문단·역할로 정리한 결과입니다.`
- 왼쪽 패널 제목: `가공 전 원문 (개인정보만 가림)`
- 왼쪽 안내 배지: `개인정보만 가린 원본 흐름`
- 왼쪽 배지 보조 문구: `원본 파일을 그대로 띄운 게 아니라, 전화번호·이름 등 개인정보만 가린 사본입니다.`
- 오른쪽 패널 제목: `우리가 정리한 결과 (문단·역할)`
- 왼쪽 원문 없음: `가공 전 원문 파일이 없어 원본 흐름을 보여줄 수 없습니다. 오른쪽 정리 결과만 확인하세요.`
- 오른쪽 문단 없음: `아직 문단으로 정리되지 않았습니다. (추출은 됐지만 정리 결과가 비어 있어요.)`
- 부가 블록 제목: `개인정보 가림 결과` · `이미지 분류`
- 가림 0건 / 이미지 0개 문구·조회수 라벨·실패 사유: v1과 동일(변경 없음).
- **금칙 준수**: "성과 등급"·"P1/P2/P3"·"정제/파싱/렌더링" 같은 등급·코드 용어를 화면 어디에도 쓰지 않는다.

### 용어 정리 (word-drift 방지) ★
- v1의 왼쪽 문단 제목 `참고용 본문 (개인정보 가림 완료)` 는 **폐기**하고, 그 내용은 이제 **오른쪽 `우리가 정리한 결과 (문단·역할)`** 로 부른다. (v1 문서의 해당 제목은 이 v2로 대체됨 — 같은 것을 두 이름으로 부르지 않도록 구현 시 옛 제목을 남기지 않는다.)
- 왼쪽은 항상 `가공 전 원문`, 오른쪽은 항상 `정리한 결과`로 부른다(목록·안내·향후 화면 공통).

## 화면 폭 대응 (모바일 체크 대체) — v1 유지 + 개편분
- 데스크탑 우선(로컬 PC 1인 검수 도구). 휴대폰 대응은 이번 범위 아님.
- **좁아지면 좌우 2단은 위아래로 쌓인다(1단)**: 순서는 **위=가공 전 원문, 아래=정리한 결과**. 쌓였을 때는 각 패널 내부 스크롤을 풀고 페이지 흐름으로(좁은 화면에서 이중 스크롤 방지 — 결정 2 참고).
- 긴 원문·긴 제목·긴 태그는 가로로 삐져나가지 않고 줄바꿈/말줄임(원문은 `줄바꿈 유지 + 단어 단위 줄바꿈`).
- 상단 띠는 스크롤해도 붙어 있게(고정 영역).

## 접근성 기본 (단순화 금지) — v1 유지
- 상태·재사용·역할·좌우 구분은 **색만으로 구분하지 않는다** — 항상 글자 제목/라벨 동반(왼=`가공 전 원문` 제목, 오=`정리한 결과` 제목).
- 왼쪽 안내 배지는 색(노랑)만이 아니라 글자(`개인정보만 가린 원본 흐름`)로 뜻을 전한다.
- 클릭 대상(링크)은 넉넉히(작은 글자 단독 링크 금지, 상하 여백 8px 이상).
- 본문/하이라이트 글자 대비 유지. 읽기 전용이라 입력칸 없음.

## 재사용 목록 / 새로 만드는 것 (간결 사다리)
- **재사용(1~2단)**: tokens.css 색 변수, v1 씨앗의 상태 배지·태그 chip·링크·가림 하이라이트(`highlight_masked` — 대괄호 토큰 하이라이트를 왼쪽에도 그대로 적용), 오른쪽 문단 스트림·이미지 카드·가림 건수 목록(위치만 이동).
- **새로(4단, 근거 남김)**: 
  - **좌우 2단 대조 레이아웃**(왼=원본 흐름 한 장 / 오=문단 카드) — 근거: v1의 옆칸(side) 배치로는 "가공 전 원본 흐름"을 나란히 놓을 자리가 없다. 기존 `.cols` 2단 골격은 재사용하되 왼쪽을 "한 장짜리 문서"로 만드는 스타일만 추가.
  - **왼쪽 안내 배지 1종** — 기존 배지/note 색(`--note-bg`/`--note-ink`) 재사용, **새 색 없음**.
  - **패널 내부 스크롤**(결정 2 채택 시) — 새 상호작용이 아니라 CSS 높이·overflow 조정만.
- **새 색 토큰 추가 없음.** 전부 tokens.css 기존 변수 안에서 해결.

# 결정 필요 (designer가 정하지 않음 — 사용자 승인 후 implementer 진행)

### 결정 1 — 왼쪽 "가공 전 원문"의 재료를 무엇으로 할까
- **(권장) 원문 원본(`body_raw`)을 읽어 개인정보만 가려서 표시.** 사용자가 말한 "가공 전 글이 어떻게 생겼나"에 가장 충실 — 진짜 원래 흐름·줄바꿈 그대로. 단점: 원문에 잡음(불필요한 서명·광고 줄 등)이 있으면 그대로 보인다(하지만 그걸 보는 게 대조의 목적이기도 함).
- **(대안) 정제본(`body_clean`)을 읽어 개인정보만 가려서 표시.** 잡음이 이미 정리돼 읽기 편함. 단점: "가공 전"이라기엔 이미 한 번 손댄 상태라, 왼쪽이 "가공 전"이라는 화면 설명과 어긋난다.
- 어느 쪽이든 **반드시 masking.py 통과분만** 화면에 나간다(불변 1). `body_raw`가 없으면 `body_clean`으로 대체하고 그럴 땐 왼쪽에 `(정제본 기준)` 작은 표시를 붙이는 절충도 가능.
- 화면 영향: 왼쪽 패널이 "날것 그대로냐 / 한 번 다듬은 것이냐"가 갈립니다.

### 결정 2 — 좌우 패널 스크롤 방식
- **(권장) 각 패널 독립 스크롤**: 왼쪽/오른쪽이 각자 스크롤바를 가져 한쪽을 내려도 다른 쪽은 제자리. 원문의 특정 대목을 띄워 두고 오른쪽만 훑기 좋음. 단점: 스크롤바가 둘.
- **(대안 A) 페이지 통짜 스크롤**: 두 패널이 함께 아래로. 단순하지만 긴 글에서 좌우 위치가 크게 어긋난다.
- **(대안 B) 동기 스크롤**: 한쪽을 내리면 다른 쪽도 같은 비율로. 대조엔 이상적이나 왼(안 나눔)·오(나눔)의 길이가 달라 "같은 대목"이 정확히 안 맞고, 구현이 무겁다(비권장).
- 화면 영향: 긴 글을 대조할 때의 손맛이 갈립니다. (좁은 화면에서 1단으로 쌓일 땐 어느 안이든 페이지 통짜 스크롤로.)

### 결정 3 — 좌우 문단을 잇는 색 마커를 이번에 넣을까
- **(권장, 이번엔 보류) 잇는 마커 없이 문단 번호만**: 오른쪽 각 문단에 번호(1·2·3…)만. 왼쪽 원본에는 번호를 안 붙임. 이유: 원본 흐름의 "어느 대목"이 오른쪽 몇 번 문단인지 정확히 이으려면 **위치 대응 정보가 지금 저장돼 있지 않다.** 없는 데이터로 색 마커를 그리면 틀린 곳을 이어 오히려 오해를 부른다(간결 사다리: 근거 없는 새 장치 금지).
- **(대안) 위치 대응을 저장한 뒤 같은 색 마커로 잇기**: 왼쪽 원문 각 대목과 오른쪽 문단을 같은 색 점/띠로 연결. 대조가 훨씬 쉬워짐. 단, 정리 단계에서 "이 문단은 원문 몇 글자~몇 글자"를 저장하는 작업이 선행돼야 함(planner/implementer 판단, 추가 저장 필요) → 별도 차수 권장.
- 화면 영향: 이번엔 "눈으로 대충 대조", 나중에 "색으로 딱 연결" 중 선택입니다.

### 결정 4 — 부가 정보(가림 결과·이미지 분류) 배치
- **(권장) 대조 2단 바로 아래에 가로 전체로 두 블록 나란히**(개인정보 가림 결과 | 이미지 분류). 스크롤하면 바로 보임 — 검수 흐름이 안 끊김.
- **(대안) 탭으로 접어 두기**(대조 / 가림결과 / 이미지 탭). 첫 화면이 깔끔하지만 한 번 더 눌러야 보임 + 이 프로젝트에 아직 탭 패턴이 없어 새 상호작용을 하나 늘림(간결 사다리에 걸림).
- 화면 영향: 부가 정보가 "스크롤로 바로" 보이느냐 "눌러야" 보이느냐입니다.

## 반복 수정 감지
이 상세 화면의 **첫 재작업**이다(단일 패널 → 좌우 대조). 같은 종류의 디자인 수정이 반복된 건 아니므로 규칙 승격 제안은 없음. 다만 향후 "왼/오 대조" 배치가 다른 화면(예: 원고 생성 전/후 비교)에도 나오면, 그때 이 좌우 대조 배치를 **공용 패턴으로 승격**할지 검토할 것.
