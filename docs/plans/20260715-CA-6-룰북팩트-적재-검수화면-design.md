# CA-6 팩트 룰북 — 화면 명세 (designer, 2026-07-15)

대상 차수: 3차(보는 화면) · 4차(고치고 도장) · 6차(엑셀 내보내기).
근거 계획: `docs/plans/20260715-CA-6-룰북팩트-적재-검수화면.md` (D1·D2·D3 확정본).
선례 화면: `/topics`(주제 검수) · `/data`(현황 카드) · `/post`(좌우 대조).

## 0. 이 화면의 성패 판정 기준

사용자 증언: **"AI가 만들어준 걸 최소한의 검수만 했다. 얼추 맞으면 넘겼다."**

그래서 이 화면의 성패는 예쁨이 아니라 **51건을 실제로 끝까지 훑게 되는가** 하나다. 명세의 모든 선택은 이 기준으로 판단했다.

- 훑기를 끊는 것은 전부 뺀다 → **확인 대화상자 0개**, 필수 입력 0개, 항목당 클릭 최소.
- 모순은 **한 항목의 칸들을 나란히 놓아야만** 보인다(사회복지사2급 실습시간: 요건 160 / 주의메모 2020.1.1 이전 120 / FAQ 120). 상세 화면은 이 하나를 위해 설계했다.
- 안전은 대화상자가 아니라 **되돌리기 + 삭제 기능 없음**으로 확보한다(계획 §6-1·§6-7).

## 1. 화면 지도

| 주소 | 이름(메뉴 표기) | 성격 |
|---|---|---|
| `/facts` | 팩트 룰북 — 목록 | 51건 훑기 시작점 + 진행 상황 + 엑셀 내보내기 |
| `/fact?id=N` | 항목 상세 | 한 항목의 모든 칸을 한 화면에. 고치기·도장이 여기서 끝남 |
| `/fact?id=N&edit=<칸>` | 상세 + 그 칸만 편집 상태 | **별도 화면 아님** — 나머지 칸이 계속 보여야 모순을 보며 고칠 수 있다 |

메뉴는 6번째로 추가. `nav_menu()` 순서: 글 목록 · 분석 · 주제·시기 트렌드 · 주제 검수 · **팩트 룰북** · 데이터.
위치 근거: '팩트 룰북'은 검수 계열이므로 '주제 검수' 바로 옆. '데이터'는 지금도 맨 끝(현황·열람)이라 그 앞에 넣는다.

## 2. 화면 ① 팩트 목록 `/facts`

### 배치 (위 → 아래)

```
[상단 띠 — 스크롤해도 붙어 있음]
  글 목록 · 분석 · 주제·시기 트렌드 · 주제 검수 · 팩트 룰북(굵게·밑줄) · 데이터
  팩트 룰북                                    [배지: 51건 중 8건 확인함]
──────────────────────────────────────────────
[알림 줄 — 방금 뭔가 했을 때만 보임]
  ✓ 엑셀로 내보냈습니다 — out\룰북팩트_20260715_1430.xlsx

  원고에 들어갈 사실을 사람이 하나씩 확인하는 화면입니다.
  여기서 확인·수정한 내용이 최종본입니다. 엑셀은 새 팩트를 넣는 입구이고,
  결과는 언제든 엑셀로 내보낼 수 있어요.

  [51]      [8]        [2]      [41]              [엑셀로 내보내기]
  전체 항목  확인함     보류     미확인
                                                (버튼은 카드 줄 오른쪽 끝)

  보기: 전체 · 미확인만 · 확인함 · 보류        (현재 보기는 굵게·밑줄)
  종류: 전체 · 공통 · 개별

  ┌ 항목명 ────────── 종류 ─ 카테고리 ── 상태 ── 고친 칸 ─ 확인 날짜 ┐
  │ 사회복지사2급        공통   사회복지    미확인      –        –     │ ← 줄 전체가 링크
  │ 사회복지사2급 최단코스 개별   사회복지    확인함 ✓    2       07-15  │
  └──────────────────────────────────────────────┘

  [정직 박스] 51건 전부 '미확인'에서 시작합니다 — AI가 만든 초안이라 아직 아무도
  끝까지 확인하지 않았다는 뜻입니다. 한 항목을 통째로 읽고 도장을 찍어주세요.
```

### 왜 이 배치인가

- **진행 상황을 상단 띠 배지에 둔다.** 상단 띠는 이미 `position: sticky`(PAGE_CSS 223행)라 스크롤해도 따라온다. 목록을 내려가는 동안 "8/51"이 계속 보이는 게 훑기 동기다. 숫자 카드는 화면 위쪽에 한 번만.
- 숫자 카드 4장은 `/data`의 `.statcards`와 같은 것. **미확인 41이 크게 보이는 게 의도**다(할 일 잔량).
- 필터 2줄은 `/`(글 목록)의 `.filters`(보기/정렬 2줄)와 같은 관례.

### 상태 4종

| 상태 | 화면 |
|---|---|
| **비었을 때**(팩트 0건) | `.state` 박스: "아직 팩트가 창고에 없습니다. 룰북 엑셀의 '② 팩트 룰북' 시트를 창고에 넣는 작업(적재)을 먼저 해야 합니다. 화면에서는 넣을 수 없어요." — 숫자 카드·필터·내보내기 버튼은 숨김(누를 게 없다) |
| **불러오는 중** | 별도 표시 없음. 로컬에서 51건을 읽는 화면이라 기존 5개 화면과 같이 한 번에 그려진다. **로딩 화면을 새로 만들지 않는다**(선례 없음 = 사다리 1단계) |
| **실패했을 때**(창고 파일 없음/깨짐) | `/topics`·`/data`와 같은 문구 형식: "팩트를 불러오지 못했습니다. 자산창고 파일을 먼저 만든 뒤 다시 열어주세요." |
| **성공했을 때** | 위 배치 그대로 |

### 표 열 (6개)

항목명 / 종류 / 카테고리 / 상태 / 고친 칸 / 확인 날짜.
- **항목명**이 링크(진한 파랑·굵게 = `.r-title`). 줄 전체가 클릭 영역(기존 `.listrow` 관례).
- **고친 칸**: 그 항목에서 값이 바뀐 칸 수. 0이면 `–`(회색). 넣는 이유 = "내가 손댄 것"과 "AI 원본 그대로"가 목록에서 구분돼야 두 번째 훑기가 빨라진다.
- **확인 날짜**: 도장 찍은 날. 미확인이면 `–`.
- 좁은 화면에서는 `.tablewrap` 가로 스크롤(기존 `an1`~`an10`과 동일). 열을 숨기지 않는다.

## 3. 화면 ② 항목 상세 `/fact?id=N`

### ★ 핵심 판단 — 좌우 대조는 여기 쓰지 않는다

과제에서 "`/글목록` 상세의 좌우 대조(원문↔정리결과)가 선례인데 팩트에도 '엑셀 원본 ↔ 지금 값' 대조가 맞는지 판단하라"고 했다. **기본 화면에는 쓰지 않는다.** 근거:

1. 좌우 대조는 **두 값이 항상 다르고, 그 차이 자체가 볼거리**일 때 성립한다(원문 흐름 vs 문단 정리). 팩트는 검수 전 **51건 전부 원본 = 지금 값**이다. 좌우로 놓으면 똑같은 글이 두 번 나온다.
2. 더 나쁜 건, 좌우 대조가 **본래 목적을 방해**한다는 점이다. 이 화면이 잡아야 할 모순은 "원본 vs 지금" 사이가 아니라 **"요건란 vs 주의메모 vs FAQ" 칸들 사이**에 있다. 화면 폭을 절반으로 쪼개면 4줄짜리 학력별 경로가 접히고 칸끼리의 세로 비교가 깨진다.
3. 그래서 대조는 **고친 칸에 한해, 그 칸 안에서 접기로** 제공한다(아래 §3-3). 원본에 닿는 길은 유지하되, 안 고친 49건의 화면을 더럽히지 않는다.

### 3-1. 배치 (위 → 아래) — 한 열

```
[상단 띠 — 붙어 있음]
  ← 팩트 목록      사회복지사2급          [배지: 미확인]  [51건 중 8건 확인함]
──────────────────────────────────────────────
[알림 줄 — 방금 저장/도장 했을 때만]
  ✓ '자주 묻는 질문 TOP3'를 고쳤습니다.  [되돌리기]

  사회복지사2급                                        ← 큰 제목
  공통 팩트 · 구분 자격증 · 카테고리 사회복지 · 엑셀 27행 · 마지막 수정 07-15 14:30
  (· 검수 메모 …  ← 메모가 있을 때만. 도장 바는 4차, 메모 표시는 3차부터)

  이 항목의 모든 칸을 아래에 한 번에 폈습니다.
  칸끼리 말이 어긋나는 곳을 찾는 게 이 화면의 목적입니다.
  (예: 요건은 160시간인데 FAQ는 120시간이라고 적힌 경우)

  ┌ 응시/취득 요건 ─────────────────── [고치기] ┐
  │ 고졸: 학점은행제 …                             │  ← 줄바꿈 그대로
  │ 전문대졸: …                                    │
  └────────────────────────────────────┘
  ┌ 주의메모 (시점/예외) ──────────────── [고치기] ┐
  │ 2020.1.1 이전 입학자는 실습 120시간            │
  └────────────────────────────────────┘
  ┌ 자주 묻는 질문 TOP3 ──[고침]───────── [고치기] ┐
  │ Q1 실습은 몇 시간인가요? → 2020.1.1 이전 120…  │
  │ ▸ 엑셀에서 온 원래 값 보기          [되돌리기] │  ← 접혀 있음
  └────────────────────────────────────┘
  … (나머지 칸)

  ▸ 수정 이력 3건 보기                              ← 접혀 있음

[하단 고정 바 — 스크롤해도 화면 아래에 붙어 있음]
  이 항목을 다 읽으셨나요?   [확인함]  [보류]  [미확인으로]
  검수 메모(선택): [__________________________]
```

- **한 열인 이유**: 모순은 세로로 훑을 때 보인다. 요건란에 학력별 4줄이 들어가는 칸이 있어 2단으로 쪼개면 줄바꿈이 깨진다(`.ptext`의 `pre-wrap`을 살려야 엑셀 줄바꿈이 그대로 보인다).
- **칸 카드 = 기존 `.panel` + `h2.sec` + `.ptext`**. 새 클래스를 만들지 않는다(사다리 2단계에서 멈춤).
- **FAQ TOP3는 쪼개지 않는다**(계획 §4). 한 칸 안에 3개가 그대로 보인다.
- **자동 모순 감지는 하지 않는다**(숫자 하이라이트 등). 이번 범위 밖이고, 오탐이 나면 "화면이 잡아준다"는 잘못된 안심을 준다. 모순은 **사람 눈 + 한 화면 배치**로 잡는다. 후속 후보로만 남긴다.

### 3-2. 칸 순서

**공통 팩트**: 응시/취득 요건 → 필요 학점 → 예상 소요 기간 → 기간 단축 방법 → 주의메모(시점/예외) → 자주 묻는 질문 TOP3 → 주의사항/흔한 오해
**개별 팩트**: 핵심 팩트 → **주의메모** → 학력별 경로 요약 → 글 작성 시 강조포인트 → 사용 우선순위 → 비고

※ D-B(주의메모를 요건 바로 아래로)는 **개별에도 같은 원리로 적용**한다 — 공통은 '요건' 아래, 개별은 '핵심 팩트' 아래. 3차 구현 시 확정(2026-07-20). 초안에는 개별만 적용이 빠져 있었다.

※ 엑셀 '구분'(division)은 칸 카드가 아니라 **맨 위 meta 줄**에 넣는다(3차 구현 시 추가) — 값이 짧은 분류표라 카드로 만들면 빈 카드처럼 보인다. 이걸 빼면 화면 문구 "이 항목의 **모든** 칸을 폈습니다"가 거짓이 된다.

### 3-3. 고친 칸의 표시 — 엑셀 원본에 닿는 길

값이 한 번이라도 바뀐 칸에만:
- 칸 제목 옆에 `.chip mark` **[고침]** (노란 칩 — 기존 📌 팩트 칩과 같은 것)
- 카드 안 맨 아래에 브라우저 기본 접기: **▸ 엑셀에서 온 원래 값 보기** → 펼치면 회색 배경에 원본 값 + **[되돌리기]** 버튼
- 접기는 브라우저 기본 동작(`<details>`) = 사다리 3단계. 자바스크립트 없음.

### 3-4. 상태 4종

| 상태 | 화면 |
|---|---|
| **비었을 때** | 항목 상세에 "빈 상태"는 없지만 **값이 빈 칸**은 흔하다(개별 팩트 비고 등). 칸 카드는 **지우지 않고 그대로 두고** 회색 안내: "이 칸은 비어 있습니다." — 지우면 "채워야 할 칸이 있다"는 사실이 안 보인다 |
| **불러오는 중** | 별도 표시 없음(§2와 동일) |
| **실패했을 때** | ① 그런 항목 없음 → `render_not_found` 형식: "그런 팩트 항목이 없습니다. [← 팩트 목록]" ② 저장 실패(§3-6) |
| **성공했을 때** | 위 배치 |

### 3-5. 도장(D2 — 항목 통째)

- 위치: **하단 고정 바**. 칸이 7~8개라 상세가 길다. 다 읽고 도장 찍으러 위로 되돌아가는 왕복이 51번 반복되면 훑기가 끊긴다. 스크롤 어디에 있든 손이 닿는 곳에 둔다.
- 버튼 3개 = 상태 3개(D3). 현재 상태인 버튼은 눌린 모양(굵게·테두리 진하게)으로 표시하고 다시 눌러도 아무 일 없음.
- **확인 대화상자 없음.** 근거: 도장은 되돌릴 수 있고(다시 누르면 됨), 삭제 기능이 없다. 51건 × 대화상자 = 훑기 포기.
- 누른 뒤: 저장 → 같은 주소로 재이동 → 상단 띠 배지가 **미확인 → 확인함**으로 바뀌고 알림 줄 "✓ '확인함'으로 바꿨습니다. [되돌리기]".
- **검수 메모**는 선택. 필수로 하면 도장을 안 찍는다. 안내 문구: "왜 보류인지 적어두면 다음에 볼 때 도움이 됩니다 — 예: 2020년 기준이 맞는지 협회에 확인 필요."

### 3-6. 고치기 상호작용

1. 칸의 **[고치기]** → `/fact?id=N&edit=faq` (같은 상세 화면, **그 칸만** 입력 상자로 바뀜). **나머지 칸은 그대로 보인다** — 요건란을 보면서 FAQ를 고치는 게 이 작업의 본질이라 반드시 지킨다.
2. 입력 상자: 여러 줄 입력(줄바꿈 그대로 저장), 폭 100%, 기본 8줄 높이, 글꼴은 화면 본문과 같게. 위에 이름표: "자주 묻는 질문 TOP3". 아래 도움말: "여기 적은 문장이 그대로 원고에 들어갑니다. 학점·기간 같은 수치는 조건(시점·학력)까지 함께 적어주세요."
3. 버튼: **[저장]** **[취소]**. 저장 → 재이동 → 알림 줄 + [고침] 칩이 붙는다.
4. **동시에 두 칸을 고칠 수 없다**(한 번에 한 칸). 이유: 한 항목 통째 편집 화면은 8개 상자가 한꺼번에 열려 "어디를 고쳤는지" 모르게 되고, 저장 실패 시 손실 범위가 커진다.
5. **저장 실패**(추출 배치가 창고를 쓰는 중 등): 편집 상태를 유지한 채 상자 위에 빨간 줄 —
   "저장하지 못했습니다. 지금 창고에 다른 작업(글 추출)이 쓰고 있는 것 같습니다. 잠시 뒤 [저장]을 다시 눌러주세요. **고치던 내용은 아래에 그대로 있습니다.**"
   → **입력 내용은 절대 날리지 않는다**(데이터 손실 방지 = 간결화 대상 아님).
6. **되돌리기**: 되돌린 것도 이력에 남는다("되돌림"). 되돌리기의 되돌리기가 가능 → 대화상자 불필요.

## 5. 상태 표현 (색만으로 구분 금지)

| 상태 | 글자 | 모양 | 색(tokens.css) |
|---|---|---|---|
| 확인함 | "확인함 ✓" | `.badge ok` | `--ok` / `--ok-bg` |
| 보류 | "보류" | `.badge warn` | `--warn` / `--warn-bg` |
| 미확인 | "미확인" | `.badge dim` (신규) | `--muted` + 회색 배경(`.chip dim`이 이미 쓰는 값) |

- **모든 배지에 글자가 있다.** 흑백 인쇄·색맹에서도 읽힌다.
- **미확인에 `danger`(빨강)를 쓰지 않는다.** 51건 전부 빨강이면 화면이 온통 경고가 되어, 정작 진짜 문제(모순)를 눈에 띄게 할 색이 남지 않는다. 미확인은 잘못이 아니라 "아직 안 봄"이다.

## 6. 엑셀 내보내기 (6차)

- **위치**: 목록 화면 숫자 카드 줄의 **오른쪽 끝**. 상단 띠에는 넣지 않는다(모든 화면에 따라다니면 잘못 누를 자리가 늘어난다). 상세 화면에는 없다(항목 하나만 내보내는 기능 없음).
- **문구**: `[엑셀로 내보내기]` — 버튼 아래 작게: "지금 창고의 팩트 전부를 상태·검수 메모와 함께 새 엑셀 파일로 저장합니다. 원본 룰북 파일은 건드리지 않아요."
- **누른 뒤**: 창고를 읽기만 하므로 대화상자 없음. 저장 → 재이동 → 목록 상단 알림 줄:
  **"✓ 엑셀로 내보냈습니다 — `out\룰북팩트_20260715_1430.xlsx` (51건 · 확인함 8 · 보류 2 · 미확인 41). 원본 룰북 파일은 그대로입니다."**
  → **어디에 저장됐는지 경로를 그대로 보여준다.** 파일을 못 찾는 게 이 기능의 첫 번째 실패 방식이다.
- 브라우저 다운로드가 아니라 `out/` 폴더 저장 = 계획 6차 확정 사항.
- 두 번 눌림: 저장 후 재이동으로 방지. 새로고침해도 다시 만들어지지 않는다.

## 8. 접근성 기본 (절대 단순화 금지)

- 상태·구분은 **색 + 글자** 동시에. 현재 필터·현재 메뉴도 색이 아니라 굵게·밑줄(기존 `.navmenu a.here` 관례).
- 누르는 것은 전부 **최소 44px 높이**. [고치기]는 작은 글자 링크가 아니라 **버튼 모양**.
- 입력 상자에 **이름표**(칸 이름)를 붙인다. placeholder로 대신하지 않는다.
- 상태를 바꾸는 것(도장·저장·되돌리기·내보내기)은 **버튼**, 화면 이동은 **링크**. 섞지 않는다.
- 접기(`<details>`)는 키보드로 열린다(브라우저 기본).

## 10. 위험 — 뷰어 첫 쓰기, 실수 클릭 방지 (화면 차원)

균형: **확인 대화상자 0개.** 51건 훑기를 지키기 위해서다. 대신 4겹:

1. **되돌릴 수 없는 행동이 화면에 없다.** 삭제 버튼 없음(계획 §6-7). 모든 수정은 이력에 남고 [되돌리기]가 있다.
2. **행동한 뒤 반드시 알린다.** 알림 줄이 "무엇을 했는지 + [되돌리기]"를 한 줄로 보여준다. 조용히 바뀌는 것이 실수 클릭보다 위험하다.
3. **버튼 간격.** 도장 3개 사이 최소 12px, 파괴적 버튼이 옆에 없다. [되돌리기]는 접기 안에 넣어 훑는 손가락에 안 닿는다.
4. **엑셀 내보내기는 읽기만** 하는 버튼이라 잘못 눌러도 파일 하나가 더 생길 뿐이다.

※ 출처 검증·303 재이동·입력 상한은 화면 뒤편 안전장치(계획 §6-3·5·4). 화면에는 위 §4의 오류 문구로만 나타난다.

## 12. 규칙 승격 제안 (사용자만 반영)

> - 자동으로 만들어진 값(추출·정규화·AI 초안)을 보여주는 화면은 **사람이 원본과 대조할 수 있는 길**을 같은 화면 안에 둔다. 값만 보여주고 근거를 감추지 않는다.
