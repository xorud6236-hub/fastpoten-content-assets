# 창고는 로컬에서, 드라이브는 보관소로 — 액션플랜 (2026-07-16)

**한 줄**: 창고 파일 하나(60MB)만 이 PC 안으로 들여와 속도를 되찾고, 구글드라이브는 "PC 바꿀 때 거쳐 가는 보관소"로 강등한다. 코퍼스(2.8GB)는 드라이브에 그대로 둔다. 까먹어서 작업이 엇갈리는 사고는 **기억이 아니라 명령이 막는다**.

근거: 같은 질의 드라이브 25.9초 vs 로컬 0.3초(약 90배), 뷰어 3분 미표시 = 사용 불가. 코퍼스는 글당 0.23초로 견딜 만.

## 0. 먼저 — 이 작업이 정말 필요한가(YAGNI)

필요하다. 다만 **필요한 최소 범위는 "창고 파일 1개의 위치"뿐**이다. 더 작은 버전으로 같은 문제가 풀리는지 3개를 검토했고 모두 기각:

| 더 작은 대안 | 기각 이유 |
|---|---|
| 그냥 참고 견디기 | 뷰어가 3분에도 안 뜸 = 도구가 죽음. 견딜 수 없음 |
| 구글드라이브 '오프라인 사용 가능' 설정만 | 이미 그 설정 권고 상태에서 벌어진 일. 가상 드라이브 계층은 남아 있고 60MB 재업로드(CPU 5,300초)도 그대로 |
| 창고를 잘게 쪼개기 / 데이터 줄이기 | 원인이 크기가 아니라 **접근 방식**(잘게 훑는 읽기 × 가상 드라이브). 쪼개도 안 나음. 데이터 손실 위험만 추가 |

**새 의존성 0 · 새 파이썬 코드 0.** 코드는 전부 `data/content_assets.sqlite3`라는 한 경로만 본다(`src/db.py:20` → `viewer.py`·`trends.py`·`bulk_extract.py`가 이걸 그대로 씀). 그래서 **경로는 그대로 두고 `data/`가 가리키는 실체만 바꾼다** — 파이썬은 한 줄도 안 건드린다.

## 1. 사전 판단에 대한 답 — 동의

**"창고만 로컬, 코퍼스는 드라이브 그대로"에 동의한다.** 제시된 세 근거(병목은 창고뿐 / 전환비용 2초 vs 수십 분 / 0.23초는 견딜 만)에 더해, 코드 쪽 근거 2개를 덧붙인다:

1. `src/viewer.py:48-59`의 이미지 경로 안전검사가 `corpus/`의 실제 경로를 풀어서 대조한다. 코퍼스 연결 방식을 지금 건드리면 이 검사가 영향을 받는지 확인해야 하는 일이 새로 생긴다 — **안 건드리는 게 공짜로 안전하다.**
2. 코퍼스는 31,375개 파일이다. 로컬로 옮기면 PC를 바꿀 때마다 구글드라이브가 3만 개 파일의 변경 여부를 대조해야 한다. 60MB 하나 복사(2초)와 비교가 안 된다.

**반대 근거는 못 찾았다.** 검토한 유일한 후보 — "추출이 코퍼스에 이미지를 쓰니 코퍼스도 로컬이어야 빠르다" — 는 기각: 추출 1건은 네이버 페이지 대기(본문 컨테이너 대기 15초 상한, `src/extract_cafe.py` 수리 내역)가 시간을 지배하고, 코퍼스 쓰기는 이미지 몇 개를 통째로 쓰는 방식이라 가상 드라이브가 약한 그 지점이 아니다. **다만 추출 배치를 다시 크게 돌렸는데 체감이 여전히 느리면 그때 재검토** — 지금 예단해서 2.8GB를 옮길 이유는 없다.

## 2. "까먹으면 엇갈린다"를 구조로 막는 법

사고 시나리오는 하나다: **집에서 작업 → 드라이브에 안 올림 → 회사에서 옛날 사본을 내려받아 작업 → 하루치 유실.**

이걸 사람의 기억으로 막지 않는다. **드라이브 보관소에 '창고 이용 카드'(작은 메모 파일)를 두고, 명령이 그 카드를 읽어 스스로 멈춘다.**

### 이용 카드에 적히는 것 (드라이브 보관소 안, 사람이 읽을 수 있는 한 장짜리 메모)
- 지금 **누가 들고 갔는지**: 사용중 / 반납됨
- **어느 PC**가, **언제** 가져갔는지 · 언제 돌려놨는지
- 창고 파일의 **크기와 지문**(내용이 바뀌면 달라지는 짧은 문자열)

### 명령이 스스로 멈추는 4개 지점 (핵심)

| 상황 | 명령의 행동 |
|---|---|
| 집에서 `작업시작` — 카드에 "회사 PC가 아직 사용중" | **멈춘다.** "회사 PC에서 아직 반납 안 됐습니다. 회사 PC를 켜고 `작업마침`을 먼저 하세요." → **하루치 유실 사고가 여기서 원천 차단된다** |
| `작업시작` — 이 PC의 로컬 창고가 드라이브 것보다 새 것 | **멈춘다.** "이 PC에 아직 안 올린 작업이 있습니다." (안 올린 작업을 덮어쓰는 일 자체가 불가능) |
| `작업마침` — 내가 가져간 뒤 드라이브 쪽이 바뀌어 있음 | **멈춘다.** 내 것을 옆에 따로 저장한 뒤 사람에게 알린다(둘 다 살린다) |
| `작업마침` — 뷰어/추출이 아직 창고를 붙잡고 있음 | **멈춘다.** "뷰어/추출을 먼저 끄세요." (반쯤 쓰다 만 창고를 올리는 사고 차단) |

### 되돌릴 수 있게 — 이전본 보관
`작업마침`이 드라이브 사본을 덮어쓰기 **직전에**, 기존 사본을 `창고-이전본/` 으로 옮겨 날짜 이름으로 남긴다(최근 N개 유지, 오래된 건 자동 삭제). 잘못 덮어써도 어제 것으로 돌아갈 수 있다. **불변 9(데이터 손실 방지는 간결화 대상 아님)에 따라 이 부분은 줄이지 않는다.**

### 명령은 2개, 이름은 업무 말로
- `작업시작.ps1` — 드라이브에서 창고를 가져오고 "사용중"으로 표시
- `작업마침.ps1` — 창고를 드라이브에 올리고 "반납됨"으로 표시

> 왜 1개가 아닌가: 시작과 끝은 **하는 일이 반대**(가져오기/올리기)라 한 명령이 알아서 판단하게 하면 방향을 잘못 잡는 순간이 곧 데이터 사고다. 대신 **"마침을 까먹어도 손해가 안 나는" 구조**(위 표 1행)로 해결한다. 명령 1개 안(작업시작이 뷰어까지 켜고 뷰어를 끄면 자동으로 올림)은 아래 결정 D2 참조.

**배포·마이그레이션 차수 없음** — 이 프로젝트는 배포 개념이 없고(헌장 §검증), DB 스키마도 안 건드린다. 1차는 스키마 변경이 아니라 **파일 위치 변경**이지만, 잘못되면 데이터가 죽는 유일한 차수라 단독으로 분리하고 위 완료 조건(건수 불변)을 통과 못 하면 그 자리에서 원복한다.

**차수 사이 주의**: 1차만 하고 2차를 미루면 명령이 아직 없으므로 **그 사이에는 PC를 바꾸지 않는다.** 1·2차는 같은 날 이어서 하는 것을 권한다.

**되돌리는 법(로컬 방식이 싫어졌을 때)**: 로컬 `data/` 폴더를 드라이브 보관소로 되돌려 넣고 `setup_data_link.ps1`(원래 버전)을 한 번 실행하면 예전처럼 연결된다 — 느린 대신 원상복구. 1차에서 드라이브 원본을 지우지 않는 이유가 이것이다.

## 4. 확인한 사실 / 확인 못 한 사실

- 백업본 `C:\Users\xorud\Documents\fastpoten-backup\content_assets_20260716-1226.sqlite3` 존재.
- `setup_data_link.ps1`은 `data`·`corpus` 둘 다 연결을 만든다(44행 `foreach ($name in @("data", "corpus"))`) → 2차에서 반드시 수정.

## 6. 안 하는 것 (범위 밖 — 필요해지면 그때)
