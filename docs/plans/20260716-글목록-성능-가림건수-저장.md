# 글목록 화면 성능 수정 — 가림 건수 저장 + 목록 나누기

관련: `docs/plans/20260716-창고-로컬화-동기화명령.md`(1차 완료) · BACKLOG 6(staff '테스트' 이름 정리)

## 0. 한 줄

글목록 36.3초의 범인은 창고도 쿼리도 아니라 **글 3,777건마다 구글드라이브에서 본문 파일을 다시 열어 개인정보를 다시 세는 것**(`viewer.py:641`). 그 숫자는 매번 달라지지 않으므로 **창고에 저장해 두고, "어떤 규칙으로 센 건지"를 함께 저장**해 규칙이 바뀌면 옛 숫자를 조용히 보여주지 않게 한다. 여기에 **목록 쪽 나누기**를 더한다(사용자 확정: 둘 다).

## 1. YAGNI 점검 — 이 작업이 정말 필요한가

필요하다. 다만 **필요한 것보다 크게 만들지 않는다**. 이 계획이 의도적으로 **하지 않는 것**:

| 안 하는 것 | 이유 |
|---|---|
| 종류별 건수 저장(전화번호 3·이름 2 …) | 종류별을 쓰는 건 **상세 화면 1건뿐**(`viewer.py:495`). 1건은 지금도 빠르다. 목록이 쓰는 건 **합계 하나**(`viewer.py:641`). 합계만 저장한다. |
| 별도 테이블 `post_mask_counts` | 글 1건당 값 1개 = `posts`의 칸으로 충분. 테이블을 새로 만들 이유가 없다(불변 9 간결). |
| `mask_type_counts`의 규칙 재로드 캐시 고치기 | 목록이 이 함수를 안 쓰게 되면 남는 호출은 **상세 1건당 4회** = 밀리초. 고칠 값이 없다. **판단: 안 함.** |
| 정렬·필터를 SQL로 내리기 | 지금 파이썬 정렬(`viewer.py:608`)은 3,777건에 ~10ms, 1만 건에도 수십 ms. SQL로 옮기면 `is_success`(`viewer.py:147`)와 "조회수 없는 글 뒤로" 규칙이 **두 곳으로 갈라진다**(새 동기화 짐). 페이징은 **거르고 정렬한 뒤 잘라내기**로 한다. |
| 새 의존성·새 웹 프레임워크 | 표준 `http.server` 유지(과제 제약). |

**더 작은 버전으로 같은 문제가 풀리는가?** 풀린다 — 위 표가 그 작은 버전이다. 핵심 코드 변경은 사실상 `posts`에 칸 2개 + 채우기 명령 1개 + `render_list` 두 군데다.

## 2. 왜 "저장"이 맞는가, 그리고 그 유일한 약점

사용자 질문("매번 달라지는 게 아닐 텐데 왜 매번 세?")이 정확하다. 같은 글 + 같은 규칙 = 항상 같은 숫자.

**약점은 딱 하나**: 가림 규칙이 바뀌면 저장된 옛 숫자가 남는다. 이건 가상의 걱정이 아니다 — BACKLOG 6에 실제로 있다(직원 이름 목록에 일반 단어 `테스트`가 들어가 글 15건이 훼손됨. 사용자가 나중에 정리 예정). **정리하는 순간 그 15건의 가림 건수는 실제로 달라진다.**

그래서 **숫자만 저장하지 않고 "규칙 지문"을 함께 저장한다**:

- 규칙 지문 = 개인정보 패턴 목록(`rulebook_pii_patterns`) + 직원 이름 목록(`staff`) + 세는 방식 번호(`masking.MASK_ALGO_VERSION`)를 정해진 순서로 이어붙여 만든 짧은 확인값. 앞의 둘(창고에 든 규칙)은 한 글자만 바뀌어도 지문이 자동으로 바뀐다. 세는 방식 자체(코드)는 창고에 없어서 자동으로 못 잡으므로, 그 코드 바로 옆의 번호를 사람이 올리는 방식으로 잇는다.
- 목록을 열 때 지문을 **한 번만** 계산(질의 2번, 밀리초)해서 각 글에 저장된 지문과 비교한다.
- 다르면(또는 아직 안 센 글이면) **숫자 자리에 숫자를 쓰지 않는다.** → 옛 숫자가 조용히 보이는 일이 구조적으로 불가능해진다.

이게 "구조로 막는다"의 실체다. 사람이 기억해서 맞추는 게 아니라, **틀린 숫자는 애초에 화면에 못 나온다.**

> 지문 만드는 자리는 `masking.py`(패턴·이름의 단일 출처)에 둔다 — 다른 파일이 패턴을 또 읽으면 동기화 짝이 하나 더 생긴다.

## 4. 차수

### 1차 — 창고에 자리 만들기 (가림 건수 + 규칙 지문)
- 내용: `posts`에 칸 2개 추가(가림 건수 / 규칙 지문). `masking.py`에 규칙 지문 만드는 자리 신설(패턴+이름 단일 출처 유지).
- 영향 범위: **데이터 구조 변경(가장 신중)** — 단 추가만, 기존 값 손대지 않음
- 위험도: 낮
- 영향 파일: `src/db.py` · `src/masking.py` · `tests/`(지문 테스트 1~2개)
- 완료 조건: `python src/db.py`를 두 번 연달아 돌려도 오류가 없고, 돌린 뒤 **글 10,755건·참고신호 13,890건이 그대로**이며, 같은 규칙에서 지문이 항상 같고 규칙을 한 글자 바꾸면 지문이 달라진다.
- 되돌리는 법: 새 칸을 안 쓰면 그만(값이 비어 있어도 기존 화면 동작 불변). 칸 삭제 금지(불변 9).

### 2차 — 이미 추출된 3,777건 소급해서 채우기 (한 번 도는 명령)
- 내용: 새 명령 `src/count_masks.py`. 본문이 있는 글을 돌며 본문 파일을 열어 가림 건수를 세고 건수+지문을 저장. **이미 현재 지문으로 세어둔 글은 건너뜀**(다시 돌려도 안전·중간에 끊겨도 이어서). `--all`로 전부 강제.
- 새 파일인 이유: `bulk_extract.py`는 "글을 가져오는" 드라이버라 역할이 다르다. 창고 값을 채우는 정비 명령은 별도 1개가 맞다.
- 영향 범위: 되돌리기 쉬움(로컬 파일·로컬 창고만. 배포 없음)
- 위험도: 낮 (읽기는 본문 파일, 쓰기는 숫자 2칸뿐)
- 영향 파일: `src/count_masks.py`(신규) · `tests/`
- 완료 조건: 명령 1회 실행 후 **본문 있는 글 3,777건 전부에 가림 건수가 채워지고**, 아무 글이나 골라 상세 화면의 "개인정보 가림 결과 총 N건"과 저장된 숫자가 **일치**한다. 글·참고신호 건수 불변.
- 되돌리는 법: 채운 값은 언제든 다시 채우면 됨(원본 무손상). 이 명령은 본문 파일을 절대 쓰지 않는다.

### 3차 — 앞으로 추출하는 글은 저장하면서 세기
- 내용: `intake_manual.process_article`은 **이미 가림 결과(`hits`)를 손에 들고 있다**(`intake_manual.py:140`) — 파일을 다시 읽을 필요 없이 그 자리에서 건수+지문을 함께 저장. `extract_cafe.process_one`은 이 함수를 재사용하므로(`extract_cafe.py:500`) 자동으로 따라온다.
- 영향 범위: 되돌리기 쉬움(로컬 파일만)
- 위험도: 낮
- 영향 파일: `src/intake_manual.py` · `tests/`
- 완료 조건: 글을 새로 1건 추출하면 **2차 명령을 돌리지 않아도** 목록에 그 글의 가림 건수가 바로 숫자로 뜬다.
- 되돌리는 법: 되돌려도 2차 명령이 채워주므로 화면은 깨지지 않음.

### 4차 — 글목록 화면 고치기 (저장값 사용 + 쪽 나누기)  ★ 성능이 실제로 좋아지는 차수
- 내용:
  1. 목록이 본문 파일을 읽지 않는다 — `mask_type_counts` 호출 제거(`viewer.py:641`), 저장된 숫자 사용(쿼리 한 번에 함께 가져옴).
  2. 지문이 다르거나 아직 안 센 글은 **숫자 대신 "다시 세기 필요"** 표시 + 화면 위에 안내 한 줄(무슨 명령을 돌리면 되는지). 뷰어는 **읽기 전용 유지** — 화면이 창고에 쓰지 않는다.
  3. 쪽 나누기: 지금처럼 거르고 정렬한 **뒤에** 한 쪽 분량만 잘라 그린다. 필터(전체/성공만/실패만)·정렬(최신순/조회수순) 규칙은 **한 글자도 안 바뀐다**(잘라내기만 추가). 쪽 이동 링크는 현재 필터·정렬을 그대로 달고 다닌다.
  4. 화면에 필요한 것(상세 모양은 designer 몫): 지금 몇 번째 쪽인지 / 전체 몇 건인지 / 앞·뒤 쪽으로 가는 길 / 필터·정렬을 바꿔도 쪽이 꼬이지 않을 것(첫 쪽으로).
- 영향 범위: 되돌리기 쉬움(로컬 파일만)
- 위험도: 중 (사용자가 매일 보는 화면 + 입력값이 링크에 되비침 → 쪽 번호도 기존 `view`/`sort`처럼 **정수만 허용**으로 좁힐 것)
- 영향 파일: `src/viewer.py` · `templates/tokens.css`(필요 시) · `tests/test_viewer.py`
- 완료 조건: **글목록이 1초 안에 뜨고**(다른 화면 수준), HTML이 1,023KB → **100KB 이하**로 줄고, 기존 테스트 33개가 그대로 통과한다.
- 되돌리는 법: 이 파일 변경만 되돌리면 즉시 이전 화면(느리지만 동작). 창고 값은 남아 있어도 무해.

### 5차 — 1만 건에서도 유지되는지 확인 (얇게)
- 내용: 글이 1만 건이 돼도 목록이 1초 안인지 확인. 남는 비용은 (a) 목록 쿼리(현재 실측 0.04초, 1만이면 ~0.15초) (b) 파이썬 거르기·정렬(수십 ms) (c) 한 쪽 그리기(건수 고정). **셋 다 글 수에 완만히 비례** → 유지된다는 게 설계상 결론이고, 이 차수는 그 확인만.
- 영향 범위: 되돌리기 쉬움(측정만, 코드 변경 없을 수 있음)
- 위험도: 낮
- 영향 파일: 없음(필요 시 `src/viewer.py` 쿼리 미세 조정)
- 완료 조건: 현재 3,777건에서 목록 1초 이내가 실측되고, 1만 건 기준 예상치가 1초 이내임이 수치로 남는다.

> **배포·마이그레이션**: 배포 개념 없음(로컬 독립 실행). 창고 구조를 건드리는 건 **1차뿐**이라 단독 차수로 분리했다. 깃허브 push는 사용자 승인 후에만.

## 5. 순서와 안전장치

1차 → 2차 → 3차 → 4차 순. **4차를 먼저 하면 안 된다**(저장값이 비어 있어 온 목록이 "다시 세기 필요"로 뜸). 2차까지 끝내면 3·4차 전에도 창고는 정상이고 화면은 지금 그대로 동작한다 — 즉 **중간에 멈춰도 안전**하다.

작업 중 창고를 만지므로 헌장 데이터 동기화 규칙 준수: **한 번에 한 PC만**(창고는 이제 로컬이지만 코퍼스는 구글드라이브 연결 — 2차 명령이 코퍼스를 대량으로 읽으므로 동기화 완료 후 시작).

## 6. 헌장 §동기화 필수 쌍에 끼는가 — 판단

**아니오.** 그 목록의 뜻은 "같은 정의가 두 곳에 있으니 사람이 함께 고쳐야 한다"이다. 저장된 가림 건수는 **정의가 아니라 계산 결과(사본)**이고, 지문 덕에 **어긋나면 스스로 드러난다**. 여기에 넣으면 오히려 "사람이 기억해서 맞춰야 하는 것"으로 잘못 분류돼, 이 설계가 없애려는 바로 그 짐을 다시 지운다.

대신 **§현재 결정에 한 줄 기록**을 제안한다(반영은 사용자만 — 아래 결정 3번).
