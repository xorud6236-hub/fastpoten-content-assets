> 목적: 5개 층을 관통하는 팩토리 전체가 도는 모습을 먼저 본다.

## 1. 현재 위치 지도 (5층 × 진척)

| 층 | 무엇 | 상태 | 근거 |
|---|---|---|---|
| **0층 자산창고** | 검증된 글·룰북·참고신호 재료 | **거의 완료** | CA-1 DB뼈대 / CA-2 엑셀 10,748건 / CA-3 수동투입 / 카페 본문 자동추출 라이브 / 웹 뷰어. 남은 것: CA-4 린터, CA-5 임베딩(둘 다 미착수) |

| **4층 성과 수집** | 서치콘솔·GA·인스타 실측 | **미착수** | — |
| **5층 자기 개선** | 성과·수정기록 → 다음 생성 반영 | **미착수** | review_pairs 구조만 있음 |

## 2. 두 갈래 경로 비교

### 경로 A — 얇은 엔드-투-엔드 한 줄기 먼저 (thin slice)
창고 글 **1건** → 원고 **1건** 생성 → 린터 검사 → (카드 or 워드프레스 초안) → 사람 확인. 각 층을 **가장 얇게** 한 번씩 관통.

- 장점: 팩토리 전체가 도는 모습을 **가장 빨리** 본다. 층 사이 연결부(원고→카드, 원고→발행)의 진짜 문제를 조기 발견. 비개발자가 "내 팩토리가 돌아간다"를 체감.
- 단점: 각 층이 얕아 처음엔 품질이 거칠다(원고 1건이 완벽하진 않음). 원고 AI(D2)를 지금 정해야 함.
- 리스크: 얇게 뚫는 과정에서 원고 테이블 등 **새 데이터 구조**가 생김(신중 대상). 다만 신설이라 기존 데이터 손실 위험은 없음.
- 비개발자 체감: **"한 번에 끝까지 보인다."** 결과물(카드 1장·초안 1편)이 손에 잡힘.

### 경로 B — 층별 순차 완성
0층을 더 채우고(린터·임베딩·글 10~20건 확대) → 1층 제대로 → 2층 → 3층 순.

- 장점: 각 층이 튼튼. 창고가 깊어져 원고 재료(RAG)가 풍부해진 뒤 생성 시작.
- 단점: 팩토리가 도는 모습을 **한참 뒤에야** 본다. 그동안 1~5층 연결부 문제는 안 보임. "창고만 계속 파는" 느낌 — 사용자가 이미 파킹한 방향.
- 리스크: 층을 완성한 뒤 연결해보니 설계가 안 맞아 되돌리는 큰 재작업 위험.
- 비개발자 체감: **"언제 결과물 나오지?"** 성취가 늦다.

### planner 추천: **경로 A (얇은 한 줄기)**
근거 3줄:
1. 사용자 요청 문구가 정확히 "전체 흐름부터"다.
2. 0층은 이미 한 줄기를 뚫기에 충분한 재료(글·룰북·참고신호)를 갖췄다 — 창고를 더 깊게 팔 실익이 지금은 낮다(YAGNI).
3. 연결부 문제를 일찍 봐야 나중 재작업이 준다.
> 단, 린터(CA-4)는 경로와 무관하게 필요하므로 한 줄기 안에 흡수한다(별도 창고 확장 아님).

## 3. 추천 경로(A)의 다음 차수 — 걸어다니는 뼈대부터

> 1층은 사실상 그린필드(읽을 코드 없음). 원칙대로 **0차 = 가장 얇은 끝-끝 1개**부터. 뼈대가 돌기 전에 살을 붙이지 않는다.

### E2E-0차 — 걸어다니는 뼈대: 창고 1건 → 원고 1건 → 화면 출력
- 목표: 사용자가 **창고 글 1건을 지정**하면, 룰북 팩트 + 그 글의 재료로 **원고 초안 1건**이 만들어져 화면/파일로 나온다. **자동발행 절대 없음.**
- 1층의 가장 작은 시작 형태(구체화):
  - 입력: 사용자가 고른 post 1건(본문 있는 것) + 해당 카테고리 룰북 팩트(rulebook_categories) + 금지어 목록.
  - 처리: 원고 AI(D2)에 "이 재료로 SEO 초안 1편" 요청. **학점·비용 등 제도 수치는 룰북에서만 삽입하고 출처 주석**, 룰북에 없으면 `[검수:]` 표시(불변 6). 참고신호(조회수)는 소재 참고용일 뿐 성과로 문장에 안 씀(불변 3).
  - 출력: 원고 텍스트(파일) + DB에 원고 1건 기록.
- 영향 범위: **데이터 구조 변경**(manuscripts 테이블 신설 — 가장 신중). 단 신설이라 기존 데이터 불변.
- 위험도: 중 (AI API 최초 연동)
- 영향 파일(예상): `src/db.py`(manuscripts 테이블 ensure 추가), 신규 `src/generate_draft.py`, 테스트.
- 완료 조건: **"창고 글 1건을 지정해 실행하면, 룰북 출처 주석이 달린 원고 초안 1건이 화면/파일로 나오고, 룰북에 없는 수치는 [검수:]로 표시된다."**
- 검증: `python -m py_compile` + 생성 원고에 단정 학점/비용 수치가 없고 출처 주석이 있는지 눈검수. DB 건수 불변 확인.
- 불변 관련: **6(팩트는 룰북)** 핵심 관문 · **3(참고신호≠성과)** 문장 표현 주의.
- 의존: D2·D5 결정 선행.

### E2E-1차 — 린터를 원고에 물리기 (CA-4 흡수)
- 목표: 생성된 원고에 금지어·단정수치·개인정보·길이 **검사 리포트**가 자동으로 붙는다.
- 영향 범위: 되돌리기 쉬움(로컬). `masking.py` 재사용 → 동기화 필수 쌍 자동 충족.
- 위험도: 낮
- 영향 파일(예상): 신규 `src/linter.py`, `src/generate_draft.py`(연결), 테스트.
- 완료 조건: **"생성 원고를 검사하면 금지어·단정수치·개인정보·길이 위반이 리포트로 나오고, 마스킹 테스트 문장(전화번호·OO쌤)이 반드시 걸린다."**
- 검증: 마스킹 자동 테스트 통과 + 금지어 포함 샘플이 리포트에 잡히는지.
- 불변 관련: **1(마스킹)·6(단정수치)**.
- 의존: E2E-0차.

### E2E-2차 — 승인 큐 + 첫 내보내기 분기(카드 or 초안)
- 목표: 원고 1건을 **사람이 승인**해야만, 카드뉴스 PNG 또는 워드프레스 **초안(드라이런)**으로 나간다. 승인 전엔 아무것도 안 나감.
- 영향 범위: 데이터 구조 변경(승인 상태 컬럼/큐) + 로컬. 신중.
- 위험도: 중
- 영향 파일(예상): `src/db.py`(원고 승인 상태), 신규 승인 화면/스크립트, `src/cardnews.py` 연결(카드 경로) 또는 워드프레스 드라이런(D3에 따라).
- 완료 조건: **"승인 표시한 원고 1건만 카드 PNG(또는 워드프레스 초안)로 나오고, 미승인 원고는 어떤 경로로도 밖으로 나가지 않는다."**
- 검증: 미승인 상태에서 내보내기 시도 시 차단되는지. 카드/초안 결과물 눈검수.
- 불변 관련: **8(승인 후 발행)·7(자사 채널만)**.
- 의존: E2E-1차, D3.

### E2E-3차 (선택·의존·단독 분리) — 워드프레스 실연동 드라이런
- 목표: 승인 원고를 실제 워드프레스에 **"초안" 상태로만** 업로드(발행 아님).
- 영향 범위: **되돌리기 어려움(외부 연동)** — 그래서 단독 차수로 분리.
- 위험도: 중~높 (외부 API·계정)
- 완료 조건: **"드라이런 실행 시 워드프레스에 글이 '초안'으로만 올라가고, '발행' 상태가 되지 않는다."**
- 검증: 워드프레스 관리자에서 초안 상태 확인 후 사용자 승인(불변 검증 방법: 3층 드라이런 규칙).
- 불변 관련: **7·8**.
- 의존: D3·D4 결정, E2E-2차. **배포성 차수 — 반드시 단독 진행·사전 사용자 승인.**

> 4층(성과)·5층(개선)은 이 한 줄기가 실제로 발행까지 간 **뒤에** 착수(불변 3: 자사 채널 실측이 있어야 성과 루프가 성립). 이번 로드맵의 다음 차수 범위 밖 — 뼈대가 돈 후 별도 계획.

## 5. 한눈 요약
- 재료 창고(0층)는 됐다. 이제 **얇은 한 줄기(경로 A)로 1→2/3층을 한 번 관통**해 팩토리가 도는 모습을 본다.
- 다음 뼈대 차수: **E2E-0(원고 1건 생성) → E2E-1(린터) → E2E-2(승인+내보내기) → (선택)E2E-3(워드프레스 드라이런)**.
- 착수 전 **D1(경로)·D2(원고 AI)·D5(첫 소재)** 답이 필요. D3·D4는 발행 직전.
- 성과·개선(4·5층)은 실제 발행 후 별도 계획.
