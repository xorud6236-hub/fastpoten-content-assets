# 액션플랜 — 카페 본문 자동 추출기(소량→확대) + 로컬 웹 뷰어

작성: 2026-07-07 / 기획자(planner). 구현 아님, 계획만.

## 한 줄 요약
백로그 1(추출기)·5(웹 뷰어)를 앞당김. **글 1건 끝-끝(뼈대) → 소량 10~20건 시범 → 웹으로 눈검수 → 판정 후 전체 확대** 순서. 추출기는 새 스크래퍼를 짜기보다 **기존 intake 파이프라인(마스킹·문단·이미지)에 본문을 넣어주는 앞단**으로만 만든다(불변 1·4 자동 준수).

## 0. 사용자가 먼저 결정할 것 (에이전트가 정하지 않음)

### 결정 1 — 추출 수단 (★ 가장 중요, 착수 전 필수)
카페 글은 대부분 회원 전용이라 로그인 상태가 필요하다. 두 길:

| 방식 | 장점 | 단점 |
|---|---|---|
| **A. 이미 로그인된 내 크롬을 그대로 구동**(claude-in-chrome) | 로그인·비밀번호를 코드가 다룰 일 없음(불변 2 위험 0), 사람이 보는 화면 그대로라 차단위험 낮음, 만들 게 적음 | 속도 느림, 한 번에 대량은 부적합, 세션 화면에 의존 |
| **B. 별도 자동 브라우저(Playwright) 새로 제작** | 대량·무인 실행에 유리, 속도 조절 자유 | 로그인 세션을 코드가 보관·관리해야 함(비밀번호 취급 위험·불변 2 주의), 차단·유지보수 부담, 새로 짤 코드 많음 |

- **권장(planner): 소량 시범(10~20건)은 A**. 이유: "품질을 눈으로 본다"가 목적이라 느려도 됨, 비밀번호를 코드에 안 넣어도 됨, 새로 만들 게 최소.
- **전체 확대(약 1만 건) 단계에서 B를 다시 판단** — 대량 무인 실행은 A로 벅참. 확대 착수 시 1회 재결정(결정 5).
- 최종 선택은 사용자.

### 결정 2 — 접근 실패를 어떻게 남길까 (extraction_status 값)
지금 값: `성공(수동투입)`, `링크오류`(CA-2). 추출기용으로 아래 제안:
- 성공: `성공(자동추출)`
- 실패 분류: `실패-삭제된글` / `실패-비공개게시판` / `실패-로그인필요` / `실패-접근불가(기타)`
- 실패는 예외 없이 `extraction_logs`에도 사유 기록(불변: v9 "실패 사유 100%"). method 컬럼은 `browser`.
- **권장: 위 5개로 시작, 시범 중 새 유형 나오면 추가.** 사용자 확인만.

### 결정 3 — 웹 뷰어 범위
- **권장: 이번엔 "추출 품질 확인용 최소 화면"만.** 목록(추출된 글 N건) + 글 1건 상세(제목 / 문단·역할 태그 / 가려진 참고용 본문 / 분류된 이미지 / 마스킹된 항목 목록)를 한 화면에 나란히. **읽기 전용**(수정·발행 기능 없음).
- 이유: 목적은 눈검수. 편집·발행은 불변 8(승인 큐) 얽혀 지금 범위 아님.
- 새 화면이므로 **2차 착수 전 designer가 씨앗 패턴 정의**(CLAUDE.md 디자인 규칙: "웹 뷰어는 그린필드 모드로 designer가 씨앗 패턴").

### 결정 4 — 추출 시 함께 채울 항목 (제목·조회수·이미지)
CA-2 결과 posts의 **title 비어 있음 / view_count 비어 있음**. 추출하면 같이 확보 가능.
- **권장: 예, 함께 채움.** 제목→`posts.title`, 조회수→`reference_signals.view_count`(불변 3: 카페 조회수는 "참고 신호"일 뿐 성과 아님 — 성과 컬럼에 넣지 않음).
- 이미지: 소량 시범에선 **파일을 corpus/에 내려받아** 눈으로 분류 확인. 단 **재사용 허용값을 자동으로 주지 않음** — 기본 보수값(개인정보 가능성 있으면 `image_pattern_only`, 애매하면 `image_rights_review`). 불변 1: 개인정보 든 이미지 원본 재사용 금지.

### 결정 5 — (확대 단계에서만) 대량 실행 수단·속도
전체 확대 착수 시점에 결정 1을 다시 봄. 지금 정하지 않음.

## 1. 차수 분해

> 원칙: 뼈대(1건 끝-끝)가 돌기 전에 살 붙이지 않음. 웹 뷰어는 별도 트랙(그린필드).

### 1차 — 추출기 뼈대 + 소량 시범 (10~20건)
- **핵심 설계**: 추출기는 **본문을 만들어 기존 파이프라인에 넘기는 앞단**. posts에서 아직 본문 없는 글 N건을 골라 → 선택한 수단으로 제목·본문 텍스트·이미지·조회수 확보 → `inbox/<슬러그>/body.txt`+`meta.json` 형태로 저장 → **`intake_manual.process_article` 재사용**으로 마스킹·문단·이미지·DB 저장. (마스킹/본문 파일화 로직을 새로 짜지 않음 → 불변 1·4 자동 준수, 동기화 필수 쌍 위반 없음.)
- **안전장치**: ① 글 1건마다 커밋(체크포인트) → 중단해도 이어서 재개 ② 글 사이 대기(속도 제한, 소량엔 넉넉히) ③ 접근 실패는 status+로그 기록 후 다음 글로(멈추지 않음) ④ posts는 `normalized_url`로 기존 행 갱신(upsert_post가 이미 지원 — 새 행 안 생김).
- **먼저 1건**으로 끝-끝 확인 후 10~20건으로 늘림(뼈대 우선).
- 영향 범위: **되돌리기 보통**(DB는 재생성 가능하나 추출 데이터는 실측·시간 듦) · 외부 접근 있음
- 위험도: **중** (외부 사이트 접근·차단·약관, 마스킹 정확도)
- 영향 파일: `src/extract_cafe.py`(신규, 앞단만) · `src/intake_manual.py`(함수 재사용, 가급 무수정) · `src/db.py`(컬럼 추가 필요 시에만 — 아래 확인) · `tests/`(추출 파서·상태분류 테스트) · `inbox/`(임시 산출) · `corpus/`(본문 3버전 저장)
  - **DB 컬럼 확인**: posts에 `extraction_status/error`, `title`, `body_*_path` 이미 있음. reference_signals에 `view_count` 있음. extraction_logs에 `method` 있음. **원칙상 새 컬럼 불요**. 필요해지면 멱등 `ensure_column`(불변 9), 삭제 금지.
- 완료 조건: **"지정한 카페 글 10~20건이 자동으로 읽혀 corpus/에 3버전 저장되고, 참고용 본문에서 전화번호·OO쌤이 가려진 채 문단·이미지·조회수와 함께 DB에 저장되며, 접근 실패 글은 사유가 100% 기록된다."**
- 검증(CLAUDE.md ★): `python -m py_compile` + 추출 테스트 통과 / 마스킹 테스트(전화번호·OO쌤 포함 문장이 반드시 가려짐) / DB 기존 posts 10,748건 감소 없음(갱신만) / 실패건 전부 extraction_logs에 사유 존재.

### 2차 — 로컬 웹 뷰어 (품질 확인용 최소, 그린필드)
- **착수 전 designer가 씨앗 패턴 정의**(디자인 규칙). 브랜드 색/글꼴은 templates/brand.css 참조.
- 걸어다니는 뼈대: **글 1건 상세 화면 한 개**(제목·문단·가려진 본문·이미지·마스킹 항목) 먼저 → 그다음 목록.
- 읽기 전용. DB를 열어 보여주기만(수정·발행·추출 트리거 없음).
- 새 의존성 판단: 파이썬 표준 `http.server`+템플릿만으로 최소 뷰어 가능한지 우선 검토(웹 프레임워크 새로 안 들임). 안 되는 이유가 나오면 그때 제안.
- 영향 범위: **되돌리기 쉬움(로컬 파일만)** · 데이터 변경 없음(읽기 전용)
- 위험도: **낮**
- 영향 파일: `src/viewer.py` 또는 `src/webviewer/`(신규) · `templates/viewer/`(신규 HTML) · `templates/brand.css`(참조)
- 완료 조건: **"로컬 웹 주소를 열면 추출된 10~20건 목록이 뜨고, 한 건을 클릭하면 문단 나뉨·가려진 본문·이미지 분류·마스킹된 항목이 한 화면에 나란히 보인다."**
- 검증: 마스킹 안 된 본문(body_raw)이 화면에 노출되지 않는지 확인(참고용 body_pub_ref만 표시 — 불변 1) / py_compile.

### 게이트 — 사용자 눈검수·판정 (코드 아님)
- 사용자가 2차 뷰어로 소량 결과 확인 → **문단 나뉨·마스킹·이미지 분류 품질 OK인가?** 판정.
- 통과해야만 3차로. 미흡하면 1차 규칙 보완 후 재시범(불변 아님, 반복 가능).
- FEEDBACK_LOG.md에 판정 기록.

### 3차 — 전체 확대 (약 1만 건) — **단독 분리, 판정 통과 후에만**
- 결정 5(대량 수단·속도) 먼저. 대량은 수단 A로 벅찰 수 있어 B 재검토 지점.
- 강한 체크포인트·재개, 실패율 모니터, 부분 실행(카페별·구간별), 중단 가능.
- 영향 범위: **되돌리기 어려움(대량·장시간, 재실행 비용 큼)** · 대량 외부 접근
- 위험도: **높** (차단·장시간·약관)
- 영향 파일: 1차 산출물 재사용 + 실행 리포트 스크립트
- 완료 조건: **"전체 대상 글 추출이 끝나고, 성공/실패 건수와 실패 사유가 리포트로 집계된다(실패 100% 사유 기록)."**
- 검증: posts 본문 채움률 리포트 / 실패건 전부 로그 존재 / 기존 건수 감소 없음.

**의존 순서**: 결정 1·2·4 → 1차 → (designer 씨앗 → 2차) → 게이트 → 결정 5 → 3차.
1차와 2차는 병행 가능하나, 2차 완료 없이 게이트 못 넘음.

## 2. 소량 시범을 안전하게 하는 방법 (1차 상세)
- **1건 먼저** 끝-끝 → 눈으로 확인 → 10~20건.
- **속도 제한**: 글 사이 사람 읽는 속도 수준 대기(소량이라 급할 이유 없음). 수단 A면 사실상 브라우저 화면 속도.
- **체크포인트**: 글 1건 성공/실패마다 즉시 DB 커밋 + 진행 위치 기록 → 언제 멈춰도 다음 글부터 재개.
- **중단 가능**: 목표 건수 채우면 자동 종료. 중간 중단해도 이미 저장분은 온전.
- **대상 선정**: 마스킹·문단 다양성 보려 카페·글 길이 섞어 선정(예: 개인정보 있을 법한 상담글 + 정보성 긴 글 혼합) — 사용자와 함께 고름.
- **접근 실패는 건너뜀**: 삭제/비공개/로그인필요면 status·로그 남기고 계속.

## 3. 불변 위반 위험 지점 (구현자·리뷰어 주목)
- **불변 1 (마스킹)**: ① 추출 본문은 반드시 masking.py 통과분(body_pub_ref)만 참고·표시. 뷰어에 body_raw 노출 금지. ② 카페 이미지(채팅캡처·닉네임 노출 가능)는 자동으로 재사용허용값 주지 말 것 — 보수값 기본.
- **불변 2 (계정정보)**: 수단 A로 로그인된 크롬을 쓰더라도 비밀번호·계정정보 시트는 읽지도 저장하지도 않음. 글 본문만.
- **불변 4 (본문은 파일에만)**: 추출 본문을 posts의 텍스트 컬럼에 넣는 지름길 금지. 반드시 corpus/ 파일 + 경로만. (process_article 재사용하면 자동 준수 — 우회 구현 금지.)
- **불변 3 (성과 이원화)**: 카페 조회수는 reference_signals(참고 신호)로만. 성과 데이터로 라벨링·개선근거 사용 금지.
- **불변 5 (자동 추출 게이트)**: 이번 사용자 명시 요청으로 열림 → **DECISION_LOG.md에 "2026-07-07 사용자 명시 요청으로 카페 자동 추출 게이트 개방(소량 시범부터)" 기록**.
- **약관·차단**: 저속·소량·사람 개입 원칙 유지. 대량(3차)은 별도 신중.
- **동기화 필수 쌍**: 추출기는 마스킹 패턴을 새로 정의하지 말고 masking.py만 호출(패턴 단일 출처 유지).
