## 2. YAGNI — 첫 원고 1편을 내기까지 정말 필요한 것

첫 질문: **이 작업이 정말 필요한가.** 필요하다. 단, 필요한 범위가 좁다.

**첫 1편에 필요한 것 (이것만)**
1. 사회복지 팩트 8건이 '확인함'이 되어 있을 것 → **CA-6 4차 화면**(만들고) + 사용자 검수(하고)
2. 화면에서 주제·의도를 넣으면 초안이 나올 것 → **AI 연결 + 원고 화면**
3. 왼쪽 초안을 보며 오른쪽에서 고쳐 완성할 것 → **좌우 화면**
4. 밖으로 나가는 글자에 개인정보가 없을 것 → **마스킹 게이트**(불변 1, 간결화 대상 아님)

**첫 1편에 필요 없는 것 (뒤로)**
- **금지어·개인정보 패턴을 화면에서 고치는 기능**(§4-7). 이유: 금지어는 **고치는 화면 없이도** 창고에 있는 14개를 그대로 프롬프트에 넣으면 되고, 개인정보 패턴 6개는 지금 잘 돌고 있다(팩트 51건 적재 시 오탐 0건 실측). 게다가 패턴을 건드리면 저장된 참고용 본문 3,777건이 그 즉시 낡아 다시 세기가 필요해진다 — **첫 원고를 내는 날 벌일 일이 아니다.**
- 팩트↔주제 잇기(51번 확정) — 첫 편은 **사회복지 팩트 8건을 화면에서 체크로 고르면 끝**이다.
- 글↔룰북 더블체크 화면, 나머지 43건 검수, 엑셀 내보내기, 달력형 계절성, 주제 25개 확대, 의편사 추출.
- 완성본 보관·학습 — **1편을 내 본 다음**. 첫 편은 보관할 완성본이 아직 없다(닭이 먼저다).
- 재생성 이력·A/B·여러 초안 비교·SEO 점수. 전부 첫 편 뒤 판정.
- 자동 발행, 성과 수집(불변 3·7·8 + 현재 결정 8).

**더 작은 버전으로 같은 문제가 풀리는가?** — 어제 계획의 "재료 복사 → 클로드 창"이 그 더 작은 버전이고, 비용 0으로 오늘 당장 된다. 사용자가 그걸 알고도 AI 연결을 택했으므로 큰 버전으로 간다. 다만 **작은 버전을 대비책으로 화면에 남길지는 Q6로 올린다**(키가 없거나 API가 실패했을 때 화면이 죽지 않게 하는 값싼 보험).

## 3. 순서 — CA-6 4차가 왜 원고보다 먼저인가 (문서로 남김)

두 가지 이유가 있고, 둘 다 우회로가 없다.

1. **재료 쪽 이유**: 팩트 51건이 전부 '미확인'이다. CA-6 계획 §7에서 불변 6이 *"룰북에 있어도 미확인이면 `[검수:]`"*로 확장 확정됐으므로, 지금 초안을 뽑으면 **수치가 전부 `[검수:]`**로 나온다. 그건 참고 표시를 보나 마나 하게 만들고, 축소 결정의 kill 기준("팩트로 채워지는 비율이 낮으면 차별점 없음")에 그대로 걸린다. 적재기가 **추가 전용**이라 엑셀에서 고쳐 다시 넣는 우회도 막혀 있다(D1) — **화면이 유일한 수정 경로다.**
2. **안전장치 쪽 이유**: 원고 화면은 뷰어에서 두 번째로 쓰기를 하는 화면이다. CA-6 4차가 만들 여덟 가지(되돌리기 · 창고 잠김 대비 busy_timeout 60초 · 다른 사이트가 몰래 시키는 저장 차단 · 입력 검증 · 저장 후 주소 재이동 · 개인정보 가림 통과 · 삭제 기능 없음 · 가림 규칙 재검사)를 **원고 화면의 오른쪽 편집 칸이 그대로 재사용한다.** 순서를 뒤집으면 같은 여덟 개를 원고 화면에서 처음부터 만들고, 나중에 팩트 화면에서 또 만든다. 한 번 만들어 두 곳이 쓴다.

**중복 계획을 쓰지 않는다.** CA-6 4차의 내용·화면 명세는 이미 있다:
- 계획: `docs/plans/20260715-CA-6-룰북팩트-적재-검수화면.md` §5 4차 · §6 안전장치 8개
- 화면 명세: `docs/plans/20260715-CA-6-룰북팩트-적재-검수화면-design.md` §3-5 도장 · §3-6 고치기 · §10 실수 클릭 방지 (결정 D1~D3·D-A~D-C 확정)
→ 아래 차수 표의 1차는 **그 문서를 그대로 가리킨다.** 여기서 다시 설계하지 않는다.

## 4. ★ 쟁점 판정 (구현 전에 못 박을 것)

### 4-1. 밖으로 나가는 것 — 마스킹 게이트 (불변 1, 간결화 대상 아님)

**무엇이 나가는가**: ①우리 글 본문(예시 재료) ②룰북 팩트 값 ③금지어 목록 ④지킬 규칙 문구 ⑤사용자가 입력한 키워드·의도. 이 중 위험한 건 ①과 ②다.

- ①은 **`body_clean_path`(가려진 본문)만 읽는다.** `body_raw_path`(원문)는 프롬프트 경로에 **아예 등장하지 않는다** — 코드에서 원문 경로를 읽는 함수 자체를 원고 쪽에 두지 않는다.
- ②는 적재 시점에 한 번 가려졌지만, 그 뒤 직원 이름 목록이 늘어도 다시 안 걸린다(추가 전용 적재의 부작용 — CA-6 §6-8이 이미 지적한 구멍).

**따라서 게이트는 "출구 한 곳"에 둔다.** 밖으로 나가는 글자 전체(프롬프트 완성본)를 보내기 직전 `masking.py`의 `mask_text`에 **한 번 더** 통과시킨다. 원고 1편 분량(수만 자)이라 비용은 무시할 수준이다. 규칙은 창고의 개인정보 패턴 + 직원 이름 목록에서 읽는다(**동기화 필수 쌍** — 정의는 한 곳, 양쪽이 참조).

**그리고 게이트는 "통과"가 아니라 "증거"를 남긴다.**
- 보낸 글자 전체를 `logs/ai_calls/`에 파일로 남긴다(가려진 상태 그대로). 나중에 "무엇이 나갔나"를 사람이 눈으로 확인할 수 있어야 한다 — 헌장 디자인 규칙 "근거를 감추지 않는다".
- 가림이 몇 건 일어났는지 화면에 표시한다("보내기 전 3곳을 가렸습니다").
- **회귀 테스트 필수**: 전화번호·OO쌤이 든 문장을 재료에 넣고 프롬프트를 만들면, 만들어진 프롬프트 문자열에 그 원문 조각이 **없어야** 통과(헌장 §검증 방법의 마스킹 항목).

**정직한 한계**: 게이트는 우리가 아는 패턴만 가린다. 새로운 형태의 개인정보(예: 처음 보는 형식의 연락처)는 못 잡는다. "다 가린다"고 쓰지 않는다.

### 4-2. 키를 어디에 두나 (불변 2의 정신 — 비밀은 창고·저장소에 안 넣는다)

- **창고(DB)에 넣지 않는다. 저장소에 넣지 않는다.** 저장소가 공개였던 사고 이력이 있다(메모리 "저장소가 공개였고 실명이 샜음").
- 두는 곳 후보: (가) 내 PC 사용자 폴더의 파일 1개(`%USERPROFILE%\.fastpoten\api_key.txt`) (나) 윈도우 환경변수 (다) 쓸 때마다 화면에 입력. → **Q4로 올린다**(권장 (가): PC를 두 대 쓰므로 각 PC에 한 번만 두면 되고, 저장소·구글드라이브 어디에도 안 실린다).
- **키가 없을 때 화면이 어떻게 동작하나**: 오류로 죽지 않는다. 원고 화면은 열리되 [초안 만들기] 버튼 자리에 안내 — "AI 열쇠가 아직 없어 초안을 만들 수 없습니다. 열쇠 파일을 두는 방법은 …" + (Q6에서 살리기로 하면) [재료 복사하기] 버튼이 대신 보인다.
- 키가 화면·로그·오류 메시지에 **절대 찍히지 않게** 한다(로그 파일에도).

### 4-3. 기다리는 화면 — 실측 확인된 사실 정정

**"뷰어는 한 번에 하나씩 처리한다"는 전제는 사실이 아니다.** `src/viewer.py` 2067행이 `ThreadingHTTPServer`를 쓴다 — 요청마다 별도로 처리하므로 **초안을 만드는 동안 다른 화면은 멈추지 않는다.** 이 부분은 걱정하지 않아도 된다.

남는 진짜 문제는 셋이다.
1. **빈 화면 대기** — 30~60초 동안 브라우저가 흰 화면으로 돌기만 한다. 뭘 하고 있는지 안 보이면 사람은 새로고침을 누른다.
2. **새로고침 = 두 번 결제** — 만드는 중에 새로고침하면 같은 요청이 한 번 더 나가고 돈이 두 번 든다.
3. 브라우저·프록시가 오래 기다리다 끊으면 이미 낸 돈이 허공에 뜬다.

**판정(권장, Q5)**: [초안 만들기]를 누르면 **먼저 '초안 자리'를 창고에 만들고 곧바로 "만드는 중" 화면으로 넘어간다**. 그 화면은 몇 초마다 스스로 다시 열려(`<meta http-equiv="refresh">` — HTML 기본 기능, **자바스크립트 0 원칙 안 깨짐**) 완성되면 결과가 뜬다. 새로고침은 그냥 보기만 하는 동작이라 **눌러도 돈이 더 들지 않는다.** 만드는 일은 뒤에서 따로 돌고, 창고에 쓸 때는 4차의 busy_timeout 60초를 그대로 쓴다.
- 대안(더 작은 버전): 그냥 기다리게 하고 "30~60초 걸립니다, 새로고침 누르지 마세요"라고 적기. 만드는 코드가 단순해지지만 **새로고침 이중 결제를 막을 방법이 없다.** 어차피 초안을 창고에 보관해야 하므로(§5) '초안 자리'는 어느 쪽이든 필요하다 → 권장안의 추가 비용이 작다.

### 4-4. 팩트 대조 — 기계 몫과 사람 몫을 정직하게 가른다 (참고 표시)

**기계가 할 수 있는 것(전부 규칙 기반, AI 판정 없음)**
- 초안에서 **숫자+단위**(시간·학점·개월·년·급·원·%)를 뽑는다.
- 이번 초안에 **재료로 넣은 팩트**들에서 같은 방식으로 숫자+단위를 뽑는다.
- 셋으로 갈라 보여준다: ①**룰북에도 있는 수치** ②**룰북에 없는 수치**(어디서 왔는지 사람이 확인) ③**룰북엔 있는데 초안이 안 쓴 수치**(참고).
- 재료로 넣은 팩트 중 상태가 '미확인'인 것 목록 → 초안에 `[검수:]`가 제대로 붙었는지 대조.
- 각 항목에서 **그 팩트 상세 화면(`/fact?id=N`)으로 가는 길**을 둔다. **고치는 것은 거기서 한다**(헌장 디자인 규칙 "근거를 감추지 않는다" + 한 화면이 두 가지 일을 겸하지 않는다).

**기계가 못 하는 것(사람 몫으로 명시)**
- 어느 쪽이 맞는지 판정. 룰북도 틀릴 수 있다(검수 1호가 그 증거).
- 조건이 붙은 수치의 옳고 그름("2020.1.1 이전 120 / 이후 160"을 문맥에 맞게 썼는지).
- 숫자가 아닌 사실(자격 요건 문장, 기관명, 절차 순서)의 정확성.
- 문장 뜻이 룰북과 어긋나는지(자연어 비교). **이건 AI에게 다시 물어보면 돈과 오판이 늘 뿐이라 하지 않는다.**

**팩트를 어떻게 고르나 — 자동 추론 금지 원칙 유지.** 팩트 항목 이름과 글 주제는 34/17로 어긋나고 그중 최소 7건이 오탐이라는 실측이 있다. 자동으로 이어붙이면 **조용한 오답**이 된다. 첫 편은 **사용자가 화면에서 팩트를 체크로 고른다**(사회복지 8건이 한 화면에 있으므로 충분).

### 4-5. 금지어·개인정보 검수 — 동기화 필수 쌍이 처음으로 실제로 쓰인다

- **금지어**: 창고 `rulebook_banned_words`(14개)를 **한 함수에서 읽어** ①프롬프트의 "쓰지 말 것" 목록과 ②완성본 검사에 **둘 다 쓴다.** 목록을 코드에 베껴 적지 않는다(헌장 동기화 필수 쌍). 검사는 단순 포함 검색 + 룰북의 '대체 표현'을 함께 보여준다. **고치기는 그 단어의 원래 탭에서**(차수 9 전까지는 열람만).
- **개인정보**: 완성된 원고를 `mask_text`에 통과시켜 **몇 곳이 걸리는지** 참고로 보여주고, **보관할 때는 가려진 상태로만 저장한다**(불변 1). AI가 만든 글에 개인정보가 있을 가능성은 낮지만 재료에서 새어 나온 게 있으면 여기서 잡힌다(게이트의 이중 확인).

### 4-6. 발행 범위 — 어디까지가 이 계획인가

**이 계획의 끝은 "화면에서 완성된 원고 글자"까지다.** 홈페이지에 올리는 것은 사람이 손으로 복사해서 한다. 발행 코드·연동은 이 계획에 **한 줄도 없다**(불변 7·8, 현재 결정 8). 따라서 승인 큐도 이번엔 필요 없다 — 사람 손을 거치지 않고 나갈 경로 자체가 없기 때문이다.

### 4-7. ★ 새로 드러난 요구 — "원래 탭에서 고치기"의 실제 상태

사용자는 *"팩트·금지어·개인정보 등은 원래 탭에서 확인하고 수정 가능하게끔 하기로 했었다"*고 기억한다. **실제 상태를 확인한 결과 셋 중 하나도 아직 고칠 수 없다.**

| 대상 | 지금 | 고치는 기능 |
|---|---|---|
| 팩트 51건 | `/facts`·`/fact` 보기만 | **CA-6 4차 — 미착수**(이 계획의 1차) |
| 금지어 14개 | `/data`에서 열람만 | **계획에도 없음** → 아래 |
| 개인정보 패턴 6개 | `/data`에서 열람만 | **계획에도 없음** → 아래 |

**금지어·개인정보 패턴 수정 화면(차수 9)은 만들되, 첫 원고 뒤로 미룬다.** 이유는 §2에 적은 YAGNI + 아래 위험이다.

**정직하게 적는 위험 — 이건 팩트 고치기보다 훨씬 무겁다.**
1. **동기화 필수 쌍**: 린터의 개인정보 패턴 ↔ 정제(마스킹)의 개인정보 패턴. 한쪽만 바뀌면 **마스킹 누락**이 난다. 정의를 한 곳에 두고 양쪽이 참조하는 구조를 깨지 않아야 한다.
2. **고치는 순간 3,777건이 낡는다**: 저장해 둔 가림 건수의 규칙 지문이 달라져 전부 '다시 세기 필요'로 뜬다(헌장 현재 결정 10). 즉 패턴 수정은 **`python src/count_masks.py`를 다시 돌리는 일과 한 몸**이다.
3. **잘못 고치면 실명이 드러나거나(불변 1 위반) 멀쩡한 문장이 훼손된다.** 어제 담당자 정비에서 실측된 사례: 이름 하나를 목록에서 빼려다 **본문에 실제 사람 이름으로 쓰인 3건을 노출할 뻔했고**, 반대로 '테스트'가 이름 목록에 들어 있어 **15건이 이미 훼손돼 있었다.**
4. → **패턴을 고치기 전에 "무엇이 어떻게 달라지는지 먼저 보여주기(드라이런)"가 반드시 붙는다. 간결화 대상이 아니다.** 어제 그 절차가 실제로 사고를 막았고, `src/staff_cleanup.py`가 선례다.
5. 정규식을 화면에서 자유롭게 고치게 하면 비개발자가 한 글자로 전체 마스킹을 무력화할 수 있다. → **자유 편집이 아니라 "이름 목록 추가/빼기"부터**(가장 잦고 가장 안전한 변경), 정규식 자체는 그 뒤에 별도 판단. 이건 차수 9 착수 시 결정 질문으로 올린다.

## 5. 화면 흐름 (designer가 명세를 쓸 수 있을 만큼 — 상세 배치는 designer 몫)

메뉴 7번째로 **'원고 만들기'** 추가(현재 6개: 글 목록 · 분석 · 트렌드 · 주제 검수 · 팩트 룰북 · 데이터).

```
[① 시작]  /write
   무엇에 대해 쓸까요?  ← 주제(자유 입력 + 창고의 주제 목록에서 고르기)
   어떤 의도인가요?     ← 예: 자격 요건을 처음 알아보는 사람에게 설명
   같이 넣을 사실(팩트) ← 체크 목록. 상태 배지(확인함/보류/미확인)가 함께 보임
   참고할 우리 글       ← 그 주제 글 3~5편이 자동으로 뽑혀 있고 체크로 뺄 수 있음
   [초안 만들기]        ← 옆에 "약 30~60초, 이번 초안 예상 비용 약 ○○○원"
                          (키 없으면 버튼 대신 안내 + 대비책 버튼 — Q4·Q6)
        ↓
[② 만드는 중]  /draft?id=N   (몇 초마다 스스로 다시 열림. 새로고침해도 돈 안 듦)
   "초안을 만들고 있습니다… 보내기 전 개인정보 3곳을 가렸습니다."
        ↓
[③ 왼쪽 초안 / 오른쪽 내가 고치기]  /draft?id=N
   ┌──── 왼쪽: AI가 만든 초안 ────┬──── 오른쪽: 내가 고치는 글 ────┐
   │ (원본 그대로. 바뀌지 않음)    │ (편집 가능. 처음엔 왼쪽과 같음)  │
   │ 제목                          │ 제목                             │
   │ 본문                          │ 본문                             │
   └───────────────────────────────┴──────────────────────────────────┘
   [저장]  [다시 만들기]   ← 다시 만들기는 돈이 또 든다는 걸 버튼 옆에 적는다

   ── 참고(주역 아님. 접기 등으로 곁에) ──────────────────────────
   ● 사실 확인   실습 160시간 — 룰북에 있음 / 수강료 30만원 — 룰북에 없음
                 [검수:] 2곳 — 미확인 팩트     → 고치려면 [팩트 룰북 탭으로]
   ● 금지어·개인정보  '보장' 1곳(대체: …) / 개인정보 0곳
                                              → 고치려면 [데이터 탭으로]
   ▸ 이번에 AI에게 보낸 내용 그대로 보기        ← 접기(근거를 감추지 않는다)
```

**왜 왼쪽 원본을 남기나** — 헌장 디자인 규칙: *"자동으로 만들어진 값을 보여주는 화면은 사람이 원본과 대조할 수 있는 길을 같은 화면 안에 둔다."* AI 초안은 자동으로 만들어진 값이다. 내가 고친 글만 남으면 "AI가 원래 뭐라고 했는지"가 사라져, 나중에 프롬프트를 손볼 때 근거가 없다. `/post` 상세(왼: 가공 전 원문 / 오른: 우리가 정리한 결과)와 **같은 씨앗 패턴**이고, **다른 점은 오른쪽이 편집 가능하다는 것 하나**다.

**참고 표시가 주역이 아닌 이유** — 화면 하나는 한 가지 일만 한다. 이 화면의 일은 **글을 완성하는 것**이다. 팩트가 틀렸으면 팩트 룰북 탭에서, 금지어·개인정보 패턴은 데이터 탭에서 고친다(차수 9 전까지는 그 탭에서 확인만). 원고 화면에서 룰북까지 고치게 하면 두 가지 일을 겸하게 되고, 그건 어제 담당자 정비에서 배운 것과 같은 종류의 혼선을 부른다('화면 고르는 칸'과 '가리는 재료'는 다른 것).

designer가 판단할 것으로 넘기는 것: 좌우 폭 비율, 좁은 화면에서 위아래로 접는 방식, 참고 표시를 오른쪽 아래에 둘지 접기로 둘지, 편집 중 왼쪽과의 대조를 돕는 방법(문단 높이 맞춤 등), 저장할 때 참고 표시를 다시 계산하는 동작(권장: 저장 시 자동 재검사).

**재사용**: `page()`·`nav_menu()`·`.panel`·`.ptext`·`.badge`·`.flash`·`.state`·`.fedit`(4차 편집 폼)·`tokens.css`. 새 색 토큰 없음. 좌우 대조는 `/post` 선례를 잇는다.

**금지**: 자바스크립트(기존 원칙), 확인 대화상자 남발, 색만으로 구분, 원고 화면 안에서 룰북 수정.

| **7. 글과 더블체크 화면** | 팩트 상세 아래 "우리 글과 숫자가 다른 곳"(읽기 전용) | 되돌리기 쉬움 | 낮 | `src/viewer.py`, 테스트 | 사회복지사2급 팩트를 열면 룰북의 120/160과 우리 글의 120/160이 건수와 함께 나란히 보이고 원글로 갈 수 있다 |

**권장 순서: 1 → 2 → 3 → 4(여기서 실제 1편 발행) → 5 → 6 → 7 → 8 → 9.**
4차까지가 **가장 얇은 끝-끝**이다. 5차부터는 뼈대가 실제로 돈 뒤에 붙이는 살이다. **9차는 가장 뒤** — 위험이 가장 크고 첫 원고에 필요 없다(§4-7). 다만 실제로 가림이 새는 상황이 발견되면 그때 즉시 앞으로 당긴다(불변 1은 대기하지 않는다).

- **배포·마이그레이션**: 배포 개념 없음(로컬 실행). 창고 구조가 바뀌는 차수는 **4·6·9**이고 각각 단독으로 분리했다. 실행 전 창고 파일 사본 1부를 둔다. 기존 건수(글 10,755 등) 불변 확인은 헌장 §검증 방법대로.
- **3차를 화면과 분리한 이유**: 이번 계획에서 처음 하는 위험한 일(외부 통신·비용·개인정보 유출 가능성)을 화면 작업과 섞지 않는다. 3차가 통과하지 못하면 4차를 시작하지 않는다.
- **9차를 단독으로 둔 이유**: 가림 규칙 변경은 창고 전체의 신뢰도를 흔든다. 다른 일과 같은 차수에 섞으면 문제가 생겼을 때 원인을 못 가른다.

## 7. 비용 — 편당 얼마인가 (근거 포함, 착수 시 재계산)

계산 근거(가정을 그대로 적는다):
- 재료: 팩트 1~3건 약 2,000자 + 우리 글 3편 **전문** 약 9,000자 + 규칙·금지어 약 1,000자 ≈ **12,000자**
- 한국어는 대략 1글자 ≈ 1.5토큰 → 보내는 양 약 **18,000토큰**
- 받는 원고 2,500자 ≈ **4,000토큰**
- 상위급 모델의 흔한 단가대(보내기 100만토큰당 $3 / 받기 $15) 기준 → 0.054 + 0.060 = **약 $0.11 ≈ 160원/편**
- 다시 만들기 2회까지 포함하면 **편당 500원 이내**

**분량이 비용을 좌우한다**: 우리 글을 3편 전문 대신 요약(각 500자)만 넣으면 보내는 양이 1/4로 줄어 **편당 60원 수준**. 반대로 10편 전문을 넣으면 3배가 된다 → **Q3**.
**정직한 한계**: 모델·단가는 시점마다 바뀐다. 위 숫자는 자릿수 감각용이고, 착수 시 실제 모델을 확정해 다시 계산한다. 그리고 **3차 첫 호출 뒤 실제 청구액으로 한 번 검증**한다(추정이 아니라 실측으로 바꾼다).

## 10. 이 계획이 틀릴 수 있는 지점 (미리 적어둠)

- **초안 품질이 기대 이하일 수 있다.** 그때 손댈 곳은 코드가 아니라 **재료와 지시문**이다. 재료를 바꿔가며 두세 번 돌려보고도 안 되면 Q9 기준으로 방식 자체를 재판정한다.
- **팩트 대조가 생각보다 안 잡을 수 있다.** 숫자만 보기 때문이다. 잡히는 게 거의 없으면 참고 표시가 장식이 된다 → 그때 '무엇을 더 볼지'를 실물로 판단한다(지금 추측으로 늘리지 않는다).
- **왼쪽 원본을 아무도 안 볼 수 있다.** 몇 편 써 보고 정말 안 보면 접기로 내리면 된다(지우지는 않는다 — 프롬프트를 손볼 때의 유일한 근거다).
- **비용이 추정보다 클 수 있다.** 3차 후 실제 청구액으로 검증하고, 다르면 이 문서의 §7을 고친다.
