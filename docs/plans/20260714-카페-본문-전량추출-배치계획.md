# 카페 본문·조회수 웹 추출 — 남은 전량 배치 실행계획

작성 2026-07-14 · planner · 구현 없음(계획만)

## 한 줄 요약
남은 미추출 카페 글(약 9,123건)을 **시트 단위 파트**로 쪼개, 기존 `extract_cafe.py`의 `process_one`(1건 멱등·마스킹·실패기록)을 그대로 재사용하는 **얇은 배치 드라이버**로 소형·다양 시트 → 대형 순서로 진행한다. 다만 "정말 전량인가(YAGNI)"와 대형 애매 시트(예전꺼·비현황) 처리는 **사용자 결정 후** 착수한다.

## 먼저 던지는 질문 — 정말 전량인가(YAGNI)
- 이 추출의 실사용 가치는 **조회수(참고신호) 표본**과 **구조DNA 분석 표본**이다. 둘 다 "시트 다양성"이 "총건수"보다 중요하다(공준모 편중은 이미 43.9%로 완화됨).
- 따라서 **전량(9,123건, 수십 시간, corpus 용량 압박)보다 "소형 시트 전량 + 대형 시트 상한 표본"이 같은 목적을 더 싸게 푼다**. 전량은 나중에 특정 분석이 실제로 요구할 때 이어받으면 된다(멱등이라 언제든 재개).
- → 이건 사용자 결정 사항(아래 결정①). 계획은 두 경로(전량/표본) 모두 같은 파트 구조로 진행 가능하게 짰다.

## 새 의존성·새 추상화 점검
- 새 의존성 없음 — Playwright는 이미 `extract_cafe.py`에서 사용 중.
- 새 파일은 `scratchpad/bulk_extract.py`(임시 드라이버) 1개뿐. `process_one`을 재사용하므로 추출 로직 재구현 없음. 시트 필터·지연·집계만 담는 얇은 루프.

## 실행 절차(파트 공통)

1. **1 PC 규칙 확인**(헌장): 다른 PC의 뷰어/추출을 끄고 구글드라이브 동기화 완료를 확인한 뒤 시작(SQLite 동시접근 손상 방지).
2. 드라이버 실행 = 시트 지정 + 세션 상한으로 1세션 돌림.
3. 세션 끝나면 파트요약 확인 → 이어서 다음 세션(멱등이므로 그냥 재실행하면 남은분부터).
4. 시트 미추출분 0 또는 터미널 실패만 남으면 파트 종료 → 사용자 보고.

### 과부화 방지 설계(드라이버가 갖출 것 — 값은 결정④)
- **세션당 상한**: 권장 100~150건(한 세션이 30~40분 내로 끝나 중단·확인이 쉬움).
- **건간 지연**: 권장 3~8초 랜덤(네이버 예의 + 로컬 부담 완화). 현재도 건당 브라우저 1회 기동이라 자연 페이싱은 있으나, 명시 지연 추가.
- **중단·재개**: 멱등(normalized_url 갱신, 새 행 금지) → 드라이버 재실행이 곧 이어받기. Ctrl+C로 언제든 중단, 커밋은 건마다(체크포인트).
- **실패 무한루프 차단(중요·구현 주의)**: `select_target`은 `body_raw_path IS NULL`만 보므로 **실패 글도 매번 다시 선택된다**. 대량 루프에서 같은 실패 글을 무한 재시도하지 않도록, 드라이버는 **터미널 실패(삭제된글·비공개게시판)는 건너뛰고, 접근불가(기타)는 1~2회까지만 재시도** 후 통과시켜야 함(이번 파트 처리집합에서 제외). 이 로직 없이는 P1 이상에서 진행이 막힘.
- **시트 선택**: `process_one`은 그대로 두고, 드라이버가 `cafe_name`(또는 현황 시트 라벨) 필터로 대상을 골라 넘김. 새 선택 함수는 드라이버 안에 최소로.

## corpus 용량(무료 15GB) 관리
- 현재 1,567건 추출됨. 남은 최대 9천여 건 × 글당 이미지 다수 → 이미지까지 받으면 용량 급증 위험.
- **레버(결정④)**: 이번 대량분은 **텍스트+조회수만 받고 이미지 다운로드 생략** 권장. 분석 목적(조회수 표본·구조DNA)에 이미지는 불필요하고, 필요한 글만 나중에 개별 이미지 보강 가능. `process_one`에 `download` 스위치가 이미 있어 재구현 불필요.
- **용량 가드**: 파트 시작 전 corpus 폴더 현재 용량 기록, **12GB 도달 시 드라이버 자동 정지 + 경고**(구글드라이브 15GB 여유 확보). 임계값은 결정④.
- 파트요약에 "이번 파트 corpus 증가량 / 누적 용량"을 포함.

## 헌장 검증방법 준수(자동 유지)
- **마스킹 100%**: `process_one`이 `intake_manual → masking.py`를 통과시키므로 패턴 재정의 없이 불변1 자동 준수. 파트요약에 "마스킹 히트 합계"를 표기(0건 시트는 눈검수 표본 확인 권장).
- **데이터손실 가드**: `build_meta`가 기존 CA-2 값(날짜·라벨·cafe_name)을 보존하며 갱신 → 불변9. **DB posts 건수는 10,755 불변**(멱등, 새 행 없음)임을 파트마다 확인·보고.
- **실패 사유 100%**: `_fail`이 posts 상태 + extraction_logs에 사유 기록. 파트요약에 실패유형별 건수 표.
- **조회수는 참고신호만**: `reference_signals`에 `자동추출:조회수` 라벨로만 저장(불변3).
