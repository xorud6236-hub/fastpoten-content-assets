근거: 사용자가 실제 브라우저(1288px)로 열어 눈으로 확인한 관찰 + FEEDBACK_LOG 2026-07-20 + 기존 명세 두 건

**이번 명세는 새 화면·새 기능을 하나도 만들지 않는다.** 있는 화면의 지면 배분·밀도·위계만 고친다.

## 0. 성패 판정 기준 (하나)

**"51건을 덜 지치게 끝까지 훑게 되는가."**

지금 화면은 작동한다(사회복지 9건 25분, 6건 수정). 그러므로 이번 작업의 목표는 "되게"가 아니라 **"손과 눈이 덜 움직이게"** 다.
모든 선택은 아래 두 가지로만 판단했다.

1. **세로로 덜 흐르는가**(스크롤 횟수) — 51번 반복되는 동작이라 한 번의 스크롤이 51번이 된다.
2. **눈이 먼저 갈 곳이 정해져 있는가**(위계) — 지금은 모든 줄·모든 칸이 같은 무게다.

예쁨은 판정 대상이 아니다. 화려한 것을 늘리지 않는다.

## 1. 문제 우선순위 — 훑기를 방해하는 정도

| 순위 | 문제 | 왜 이 순위인가 | 차수 |
|---|---|---|---|
| **1** | 팩트 상세의 **긴 칸이 화면 하나를 넘긴다**(필요 학점 = 과목 30줄) | 가장 오래·가장 자주 쓰는 화면에서, 한 항목당 스크롤을 5~6회 더 만든다 × 51건 | 1차 |
| **2** | **[고치기]가 칸 아래**에 있어 긴 칸에서는 버튼까지 스크롤해야 한다 | 명세 §3-1은 원래 "칸 제목 옆"이었다 — **구현이 명세에서 벗어난 것**(drift). 고치는 동작이 매번 왕복을 만든다 | 1차 |
| **3** | 팩트 상세 **상단 띠 항목명 + 본문 큰 제목이 같은 글자로 두 번**, 도장 바와 검수 메모가 겹쳐 보임 | 첫 화면의 세로를 아무 정보 없이 두 줄 먹는다. 겹침은 "눌러도 되나" 망설임을 만든다 | 1차 |
| **4** | **글 목록의 지면 배분이 거꾸로** — 제목이 잘리고, 값이 한 종류뿐인 상태 열이 넓다 | 목록의 목적은 "제목을 읽고 고르기"인데 그 정보가 가장 먼저 잘린다. 다만 팩트 검수만큼 자주 쓰지는 않는다 | 2차 |
| **5** | **화면 폭을 못 쓴다**(1288px 창에 콘텐츠 1100px) | 폭이 넓어지면 같은 글이 더 적은 줄을 차지한다 = 1번 문제도 함께 줄어든다. 한 줄 고치기라 효과 대비 비용이 가장 낮다 | 1차(폭만 먼저) |
| **6** | **팩트 목록 밀도**(51건 중 16건만 보임) + 개별 팩트가 한참 아래 | 훑는 화면인데 세 번 스크롤해야 전체가 보인다 | 2차 |
| **7** | 팩트 목록 **카테고리 열이 공통 16건에서 항목명과 완전 중복** | 자리는 먹지만 눈을 방해하진 않는다(같은 값이라 읽지 않고 지나감) | 2차 |
| **8** | 가림 `0건`과 `8건`이 같은 무게 · 확인함 행이 배지 말고는 티 안 남 | 위계 문제. 고치면 좋지만 스크롤을 줄이진 않는다 | 2차 |
| **9** | 글자 크기 13px/12px 난립 · 회색 하드코딩 4곳 · 메뉴 현재 위치 약함 | 지금 당장 아프지 않다. 다만 화면이 늘수록 되돌리기 비싸진다 | 3차 |

**고치지 않는 것**: 정보 밀도를 위해 여백을 0으로 몰지 않는다. 아래 §5 "지킬 것" 참조.

## 2. 화면별 개선안

### 2-1. 공통 — 화면 폭과 지면 원칙

#### (1) 폭: `max-width: 1100px` → **1320px** (모든 화면 통일)

- 근거: 사용자 창 1288px. 좌우 패딩 24px씩 빼면 실사용 1240px가 나오고, 그보다 넓은 창에서도 1320px에서 멈춘다.
- **화면마다 다른 폭을 만들지 않는다.** 표 화면만 넓히면 메뉴를 오갈 때 본문 왼쪽 끝이 좌우로 흔들려 "다른 사이트에 온 느낌"이 된다. 폭은 씨앗 패턴이므로 값 하나로 유지한다.
- 팩트 상세도 함께 넓힌다. **한 열 유지**(기존 명세 §3-1 판단 그대로 — 2단으로 쪼개면 엑셀 줄바꿈이 깨진다). 넓어진 폭은 2단이 아니라 **긴 칸의 줄 수를 줄이는 데** 쓴다.
- 글 상세(좌우 대조)는 각 열이 약 620px가 되어 읽기 폭으로 적당하다. 구조는 손대지 않는다.
- 읽기 폭 제한(한 줄 최대 글자 수)은 **이번에 넣지 않는다** — 팩트 값은 엑셀 줄바꿈이 그대로 살아 있어(`pre-wrap`) 아주 긴 한 줄이 드물다. 실물에서 "한 줄이 너무 길다"가 나오면 그때 판단한다.

#### (2) 표의 지면 배분 원칙 (앞으로 모든 표에 적용)

> **사람이 읽어서 하나씩 구별하는 열이 넓다. 값이 짧거나 거의 같은 열은 좁다. 값이 한 종류뿐인 열은 아예 없앤다.**

- 넓어야 하는 것: 제목 · 항목명 · 주제 이름 — **무엇을 고를지 정하는 정보**.
- 좁아도 되는 것: 숫자 · 날짜 · 상태 배지 — 모양이 정해져 있어 흘깃 보면 읽힌다.
- 없애도 되는 것: 지금 목록의 모든 줄이 같은 값인 열(§2-2 상태 열).
- 넘침 처리는 기존 그대로 — 칸은 접히지 않고 `…`로 줄이며, 표 전체는 `.tablewrap` 가로 스크롤(최소 폭 유지).

#### (3) 밀도 — 간격 사다리를 못 박는다

지금 여백 값이 4·6·8·12·14·16·24·32로 흩어져 있다. **앞으로 쓰는 값은 4 · 8 · 12 · 16 · 24 다섯 개뿐**(32는 접기 섹션 위 간격에만 남긴다).

| 자리 | 지금 | 바꿈 | 효과 |
|---|---|---|---|
| 표 한 줄 안쪽 여백 | 12px 16px | **8px 14px** | 한 화면 18줄 → 약 24줄 |
| 카드(`.panel`) 안쪽 | 16px | 14px | — |
| 카드 사이 | 24px | **16px** | 팩트 상세 칸 7~8개에서 세로 60px 절약 |
| 큰 제목 아래 | 24px | 16px | — |

### 2-2. 글 목록 `/`

#### 배치 (위 → 아래) — 순서는 그대로, 지면만 바뀐다

```
[상단 띠]  글 목록(굵게·밑줄·현재 위치 표시 강화) · 분석 · … 	총 10,755건 · 본문 확보 3,777건
──────────────────────────────────────────────────────────
  보기: 전체 · 성공만 · 실패만   |  정렬: 최신순 · 조회수 높은 순
  걸러 보기: [카페 ▾] [담당자 ▾] [적용]

  (걸러 보는 중 조건 줄 — 조건이 있을 때만, 기존 그대로)
  이 100건은 모두 본문을 가져온 글입니다.        ← 상태 열을 숨겼을 때만 뜨는 한 줄
  3,777건 중 1~100번째 보는 중

  ┌ 제목 ─────────────────────────── 카페 ─ 담당자 ─ 가림 ─ 문단 ─ 이미지 ─ 조회수 ─ 작성일 ┐
  │ 안전관리 자격증 종류 알아보기 — 산업안전기사부터…   공준모   OOO   8건   7   6   456   07-02 │
  └──────────────────────────────────────────────────────────┘
```

#### 열 비율 (지금 → 바꿈)

| 열 | 지금 | 바꿈 | 근거 |
|---|---|---|---|
| 제목 | 2fr | **3.6fr** | 목록의 목적. 1320px 폭 + 상태 열 제거로 제목에 약 2배를 준다 |
| 카페 | 1fr | 1fr | 그대로 |
| 담당자 | 0.8fr | 0.8fr | 그대로 |
| **상태** | 1.7fr | **조건부 삭제**(§아래) | 값이 한 종류면 정보가 아니다 |
| 가림 | 1.2fr | 0.9fr | '다시 세기 필요'가 들어갈 최소만 |
| 문단·이미지 | 0.5·0.55fr | 그대로 | |
| 조회수 | 0.8fr | 0.8fr | |
| 작성일 | 1fr | 0.9fr | `07-02` 형식이면 충분 |

#### 상태 열 — 조건부로 숨긴다 (**결정 D2**)

- **지금 보이는 목록의 글이 전부 같은 상태면 열을 그리지 않는다.** 대신 목록 위 한 줄(회색, 14px):
  > **"이 100건은 모두 본문을 가져온 글입니다."**
- 실패한 글이 섞이면 열이 자동으로 다시 생긴다. `보기: 실패만`을 고르면 당연히 다시 보인다.
- 근거: 3,777건 전부 `성공(자동추출)`이고 초록 배지가 제목보다 눈에 띈다 = **정보량 0인 열이 시선을 먼저 가져간다.** 다만 "왜 상태가 안 보이지"라는 의문이 생기지 않도록 **글자 한 줄로 반드시 알린다**(숨기는 것과 감추는 것은 다르다).

#### 위계 — 제목만 무겁게

- 제목: 굵게 + `--brand`(지금 `.r-title` 그대로 유지).
- 나머지 열(카페·담당자·날짜): **`--muted` 회색**으로 한 단계 낮춘다. 지금은 전부 `--ink`라 같은 무게다.
- 숫자 열(문단·이미지·조회수): `--ink` 유지(비교하며 읽는 값이라 흐리면 안 된다).
- 가림 건수: **0건은 회색(`.num-dim` — 이미 있는 클래스)**, 1건 이상은 진한 글자. 색만이 아니라 **0은 `0건`, 1 이상은 `8건`으로 글자도 다르므로** 색맹 조건에서도 구분된다.

### 2-3. 팩트 목록 `/facts`

#### 배치 (위 → 아래)

```
[상단 띠]  … 팩트 룰북(현재) …                     [배지: 51건 중 8건 확인함]
──────────────────────────────────────────────────────────
  (기존 머리말 2줄 · 숫자 카드 4장 · [엑셀로 내보내기]  ← 전부 그대로)

  보기: 전체(51) · 미확인만(42) · 확인함(8) · 보류(1)
  종류: 전체(51) · 공통(16) · 개별(35)              ← 건수를 붙인다

  ▣ 공통 팩트 16건                                  ← 묶음 머리(기존 h2.sec)
  ┌ 항목명 ──────────────── 카테고리 ─ 상태 ─ 고친 칸 ─ 확인 날짜 ┐
  │ 국가기술자격 — 기사              –        미확인    –       –   │
  └──────────────────────────────────────────────┘
  공통 팩트는 항목명이 곧 카테고리라 '–'로 둡니다.

  ▣ 개별 팩트 35건
  ┌ 항목명 ──────────────── 카테고리 ─ 상태 ─ 고친 칸 ─ 확인 날짜 ┐
  │ 사회복지사2급 최단코스           사회복지  확인함 ✓   2    07-15 │
  └──────────────────────────────────────────────┘

  [정직 박스 — 기존 그대로, 맨 아래]
```

| 항목 | 확정 |
|---|---|
| **종류 열 삭제** | 묶음 머리(`공통 팩트 16건` / `개별 팩트 35건`)가 그 정보를 이미 말한다. 51줄에 같은 글자를 반복해 적지 않는다. 정렬이 D-A(공통 먼저 → 개별)로 확정돼 있어 묶음이 반드시 연속으로 나온다 |
| **카테고리 열은 남긴다** | 개별 35건에서는 다른 값이다. 다만 **항목명과 값이 같으면 `–`(회색)** 로 두고 표 밑에 한 줄로 이유를 적는다. 열을 없애면 개별 팩트에서 "이게 무슨 분야였지"를 상세로 들어가야만 알 수 있다 |
| **개별 팩트로 바로 가기** | 묶음 머리 자체가 눈에 띄는 표지가 된다 + 종류 필터에 건수를 붙여 "개별(35)"가 누를 수 있는 것임을 분명히 한다. **새 '바로가기' 장치를 만들지 않는다**(사다리 1단계 — 이미 있는 필터로 된다) |
| **확인함 행 표시** | 배지에 더해 **항목명 앞에 `✓`** 를 붙인다(글자로 구분 — 색 의존 금지). 행 배경색은 바꾸지 않는다(줄무늬가 생기면 51줄이 어지럽다) |
| **밀도** | §2-1(3) 공통 적용. 한 화면 16건 → **24건 이상** |
| 열 비율 | 항목명 3.2fr · 카테고리 1fr · 상태 0.9fr · 고친 칸 0.7fr · 확인 날짜 0.9fr |

### 2-4. 팩트 상세 `/fact?id=N` — **이번 작업의 중심**

#### (1) 긴 칸을 어떻게 다루는가 — **앞부분은 항상 보이고, 뒷부분만 접는다**

**확정: 길이가 기준을 넘는 칸만, 앞 8줄을 그대로 두고 나머지를 브라우저 기본 접기로 넣는다.**

- 기준: 칸 값이 **12줄을 넘을 때만** 접는다. 12줄 이하는 지금처럼 전부 편다(대부분의 칸이 여기 해당 — 화면이 달라 보이지 않는다).
- 접기 제목 문구: **"▸ 나머지 22줄 더 보기 (이 칸은 모두 30줄입니다)"** — 몇 줄이 숨었는지 숫자로 말한다(헌장 §디자인 규칙 "화면의 숫자는 그게 무슨 숫자인지 화면이 말한다").
- **편집 중인 칸은 절대 접지 않는다.** 고칠 때는 전체가 보여야 한다.
- **[고침] 칩이 붙은 칸도 접지 않는다.** 내가 손댄 칸은 다시 볼 이유가 있다.

왜 이 방식인가 (세 후보 비교):

| 후보 | 판정 | 근거 |
|---|---|---|
| 높이 제한 + 칸 안쪽 스크롤 | **쓰지 않음** | ① 페이지 스크롤과 칸 스크롤이 겹쳐 휠이 어디서 멈출지 예측이 안 된다 — 51번 반복하면 확실히 지친다 ② "더 있다"는 사실이 **스크롤바 모양으로만** 전달된다(색·모양 의존 = 접근성 기본 위반) ③ 뷰어에 이미 있는 내부 스크롤 두 곳(`.rawdoc`·`.rightstream`)은 **좌우 두 열의 높이를 맞춰야 하는** 사정이 있었다. 팩트 상세는 한 열이라 그 사정이 없다 |
| 통째로 접고 요약만 보이기 | **쓰지 않음** | 이 화면의 목적은 **칸끼리 어긋난 곳 찾기**다. 통째로 접히면 그 칸은 아무도 안 읽고 도장이 찍힌다 — 성패 기준을 정면으로 깬다 |
| **앞 8줄 + 나머지 접기** | **채택** | 앞 8줄에 그 칸이 무슨 말을 하는지가 대개 들어 있다. 모순이 나는 칸(요건·주의메모·FAQ)은 원래 짧아 **접히지 않는다.** 실제로 폭발하는 칸은 '필요 학점'의 과목 **나열**이라 모순 검수 대상이 아니다. 접기는 브라우저 기본(`<details>`) = 사다리 3단계, 자바스크립트 0 |

※ **8줄·12줄은 실물에서 조정할 값이다.** 나는 렌더링을 못 본다 — 첫 구현 후 스크린샷으로 "짧은 칸까지 접히지 않는지"만 확인하면 된다.

#### (2) [고치기] 위치 — 칸 제목 줄 오른쪽 끝

- 원래 명세 §3-1이 그랬는데 구현이 칸 아래로 내려갔다. **명세로 되돌린다.**
- 칸 제목 줄을 좌우 배치로: 왼쪽 `칸 이름 + [고침] 칩`, 오른쪽 `[고치기]`.
- 버튼 크기는 44px 그대로. 좁은 화면에서는 제목 아래로 접혀 내려간다.

#### (3) 제목 중복 제거

- **본문 큰 제목(h1)을 지운다.** 상단 띠가 `position: sticky`라 항목명이 스크롤 내내 보인다 — 같은 글자를 두 번 쓸 이유가 없고, 지운 자리만큼 첫 화면에 칸이 하나 더 들어온다.
- meta 줄(공통/개별 · 구분 · 카테고리 · 엑셀 행 · 마지막 수정)은 **그대로 남긴다** — 여기에만 있는 정보다.
- 상단 띠 항목명은 지금보다 한 단계 크게(20px 유지, 굵게).

#### (4) 도장 바와 검수 메모 겹침

- 검수 메모를 **접기 안으로** 넣고 도장 버튼 줄과 **같은 줄**에 둔다: `[확인함] [보류] [미확인으로]   ▸ 검수 메모 적기`.
- 메모를 펼치면 도장 바가 위로 자란다(본문을 덮지 않는다).
- 본문 맨 아래에 **도장 바 높이만큼 빈 자리**를 둔다 — 마지막 칸이 도장 바에 가려 안 읽히는 일이 없게.
- 근거: 모바일 명세 §9에 이미 "좁은 화면에서는 검수 메모를 접는다"가 있었다. **넓은 화면에도 같은 규칙을 적용**한다(메모는 51건 중 몇 건만 쓴다).

### 2-5. 나머지 화면 (분석·트렌드·주제 검수·데이터·글 상세)

**배치는 손대지 않는다.** 공통 변경(폭 1320 · 간격 사다리 · 글자 크기 4단계 · 표 위계)만 자동으로 적용된다.
표 최소 폭(`an1`~`an13`)은 그대로 둔다 — 좁은 화면에서 짜부러지지 않게 하는 장치이지 넓은 화면의 폭이 아니다.

**상단 띠 메뉴 현재 위치 강화**(전 화면 공통): 지금은 굵게+밑줄뿐이라 흰 글자 6개 속에서 약하다.
→ 현재 항목에 **밝은 밑줄 띠(3px, 지금 쓰는 흰색)** 를 붙이고 나머지 항목은 살짝 흐린 흰색(투명도)으로 낮춘다. **새 색을 만들지 않는다.**

## 3. 색·타이포 — 지금 있는 것의 쓰임 정리 (tokens.css 한 곳)

### 3-1. 새 색을 만들지 않는다. 이미 쓰고 있는 회색에 이름을 붙인다 (**결정 D1**)

뷰어에 색 리터럴이 4곳 있다: `#eef1f5`(`.chip.dim`·`.placeholder`·`.badge.dim`) · `#eef4fb`(`.listrow:hover`) · `#f4f6fa`(body 배경) · `#fff`(상단 띠 글자·버튼 글자).

제안: **값은 하나도 바꾸지 않고** tokens.css에 이름만 준다.

| 새 이름 | 값(지금 그대로) | 쓰는 곳 |
|---|---|---|
| `--fill` | `#eef1f5` | 흐린 칩·자리표시·미확인 배지 바탕 |
| `--hover` | `#eef4fb` | 표 한 줄 위에 마우스 올렸을 때 |
| `--bg` | `#f4f6fa` | 화면 바탕 |
| `--on-brand` | `#ffffff` | 진한 파랑 위의 글자 |

- **색 가짓수는 늘지 않는다**(4개 값 → 4개 이름). 헌장 "색은 tokens.css 한 곳" 문언을 실제로 지키게 되는 것뿐이다.
- 완료 조건: `viewer.py`에서 `#`으로 시작하는 색 리터럴 **0개**.
- 이 항목은 헌장 규칙 문언 해석이 갈렸던 지점(팩트 명세 §7 정정)이라 **사용자 결정으로 올린다.**

### 3-2. 글자 크기 — 4단계로 정리

지금 16 / 15 / 14 / 13 / 12 / 11px이 섞여 있고, 특히 13px이 8곳·12px이 4곳이다.

| 단계 | 값 | 쓰는 곳 |
|---|---|---|
| 큰 제목 | 24px | 화면 제목(h1) |
| 섹션 제목 | 20px / 카드 안 16px | `h2.sec` |
| 본문 | 16px | 읽는 글·팩트 칸 값 |
| 보조 | **14px** | 표 내용·표 머리·배지·칩·안내 문구 (지금 13·15px이 섞인 자리) |
| 각주 | 12px | 정말 작아도 되는 것(히트맵 칸 숫자·범례)만 |

- **13px을 없앤다.** 비개발자 1인이 오래 읽는 화면에서 13px은 작다. 11px도 12px로 올린다.
- 줄 간격은 본문 1.6(지금 값) 유지, 표 한 줄만 1.4로 낮춰 밀도를 얻는다.

### 3-3. 위계 — 색으로 세 단계만

| 단계 | 색 | 쓰는 곳 |
|---|---|---|
| 눈이 먼저 갈 곳 | `--brand` + 굵게 | 목록의 제목·항목명, 섹션 제목, 누를 수 있는 것 |
| 읽는 값 | `--ink` | 본문, 비교하는 숫자 |
| 배경 정보 | `--muted` | 카페·담당자·날짜·설명·단위 |

의미색(`--ok`/`--warn`/`--danger`)은 **상태 배지에만** 쓴다. 표의 일반 값에 의미색을 칠하지 않는다.

## 4. 상태 4종 (바뀌는 화면만)

| 화면 | 비었을 때 | 불러오는 중 | 실패했을 때 | 성공했을 때 |
|---|---|---|---|---|
| 글 목록 | 기존 문구 그대로. **상태 열을 숨긴 상태에서 0건이면** 안내 줄도 함께 사라진다(셀 것이 없다) | 별도 표시 없음(기존 관례 — 로컬에서 한 번에 그려진다) | 기존 문구 그대로 | §2-2 배치 |
| 팩트 목록 | 기존 문구 그대로. **묶음 머리는 그 묶음이 0건이면 그리지 않는다**(빈 '개별 팩트 0건' 머리를 남기지 않는다) | 없음 | 기존 문구 | §2-3 배치 |

**이번 명세는 문구를 새로 만들지 않는다.** 아래 두 줄만 추가된다.

| 자리 | 문장 (그대로) |
|---|---|
| 글 목록 — 상태 열을 숨겼을 때 | **"이 100건은 모두 본문을 가져온 글입니다."** (걸러 본 상태면 "이 37건은 모두 …") |
| 팩트 목록 — 카테고리 `–` 설명 | **"공통 팩트는 항목명이 곧 카테고리라 '–'로 둡니다."** |
| 팩트 상세 — 긴 칸 접기 | **"▸ 나머지 22줄 더 보기 (이 칸은 모두 30줄입니다)"** |
| 팩트 목록 — 묶음 머리 | **"공통 팩트 16건" / "개별 팩트 35건"** |

쓰지 않는 말(그대로 유지): 성과 등급 · P1/P2/P3 · 레코드/필드/컬럼/행 · 필터(→ 걸러 보기) · 정규화(→ 묶기) · 반응형/그리드/토큰(화면 문구에).

## 5. 하지 말 것 — 지금 잘 작동하는 것 (건드리면 되돌린다)

1. **숫자 카드 4장**(`/facts`·`/data`) — 사용자가 "잘 작동한다"고 확인. 크기·순서·문구 그대로.
2. **글 상세 좌우 대조** — 왼쪽 원문 흐름 / 오른쪽 정리 결과. 구조·내부 스크롤·70vh 전부 그대로.
3. **정직 박스**(노란 `.honest`) — 한 화면에 하나 원칙 유지. 색·굵기 낮추지 않는다.
4. **하단 고정 도장 바** — "실제로 잘 고정돼 있다". 위치·버튼 3개·확인 대화상자 0개 그대로.
5. **`<details>` 접기** — 자바스크립트 없이 도는 유일한 접기 수단. 새 접기 장치를 만들지 않는다.
6. **배지에 글자 병기**(확인함 ✓ / 보류 / 미확인) · 색만으로 구분 금지.
7. **누르는 것 최소 44px** — 밀도를 얻겠다고 버튼·쪽 이동·✕를 줄이지 않는다. 밀도는 **표 한 줄**에서만 얻는다.
8. **자바스크립트 0 · 새 의존성 0 · 웹폰트 0** — 글꼴은 `--font`(윈도우 기본) 그대로.
9. **새 색·그림자·둥근 모서리 값·애니메이션·아이콘 세트를 늘리지 않는다.** 모서리는 지금 쓰는 6px/8px/999px 세 가지만.
10. **글 목록·분석·트렌드·주제 검수·데이터는 읽기 전용** — 이번 작업으로 어떤 화면에도 쓰기 동작이 생기지 않는다.

## 6. 차수 분해 (효과 큰 것부터 · 각 차수 단독으로 끝난다)

### 1차 — 팩트 상세를 덜 지치게 (가장 자주·오래 쓰는 화면)

바꾸는 것: 화면 폭 1320px(전 화면 공통) · 긴 칸 접기(12줄 초과 시 앞 8줄 + 나머지) · [고치기]를 칸 제목 줄로 · 본문 큰 제목 삭제 · 검수 메모를 도장 바 접기로 + 본문 아래 빈 자리 · 카드 사이 간격 24→16.

**완료 조건**
- 1288px 창에서 **칸이 가장 많은 항목**(사회복지사2급 등)의 첫 화면에 칸 카드가 **3개 이상** 보인다(지금 1~2개).
- 모든 칸의 [고치기]가 **칸 제목과 같은 줄**에 있다.
- '필요 학점'처럼 30줄짜리 칸이 있는 항목도 **스크롤 3회 이내**로 도장 바까지 닿는다.
- 12줄 이하 칸은 **하나도 접히지 않는다**(접기가 짧은 칸까지 먹으면 실패).
- 편집 중인 칸·[고침] 칸은 접히지 않는다.
- 기존 뷰어 테스트 전부 통과, 창고 건수 불변.

### 2차 — 표의 지면 배분과 밀도 (글 목록 · 팩트 목록)

바꾸는 것: 글 목록 열 비율(제목 3.6fr) · 상태 열 조건부 숨김 + 안내 한 줄 · 보조 열 회색 낮추기 · 가림 0건 흐리게 · 팩트 목록 종류 열 삭제 + 묶음 머리 · 카테고리 `–` · 필터에 건수 · 확인함 앞 ✓ · 표 한 줄 간격 8/14 + 줄 간격 1.4.

**완료 조건**
- 1288px에서 글 제목이 **30자 이상** 잘리지 않고 보인다(지금 약 15자에서 잘림).
- 팩트 목록이 한 화면에 **24건 이상** 보인다(지금 16건).
- 상태 열: 성공만 있는 목록에서 열이 사라지고 안내 줄이 뜬다 / `보기: 실패만`에서는 열이 다시 보인다.
- 표 안 어떤 칸도 두 줄로 접히지 않는다(기존 규칙 유지).

### 3차 — 색·글자 정리 (되돌리기 비싼 것을 마지막에, 결정 D1 이후)

바꾸는 것: 회색 하드코딩 4곳을 tokens.css 이름으로 · 글자 크기 4단계(13px 제거) · 상단 띠 현재 위치 표시 강화.

**완료 조건**
- `viewer.py`에 색 리터럴 **0개**(tokens.css 변수만 참조).
- tokens.css의 **색 가짓수가 늘지 않았다**(이름만 4개 추가, 값은 기존 그대로).
- 화면에 13px·11px 글자가 없다.
- 7개 화면 전부에서 현재 메뉴 항목이 한눈에 구분된다.

| 칸 제목 줄의 좌우 배치(`h2.sec`에 변형 하나) | 지금 `h2.sec`는 제목만 담는다. [고치기]를 같은 줄 오른쪽 끝에 놓으려면 좌우 배치가 필요하다. `.cmp-col > h2.sec`가 이미 같은 방식(제목 + 배지 한 줄)을 쓰고 있어 **그 관례를 따르는 변형**이지 새 종류가 아니다 | 기존 토큰만 |
| 긴 칸 접기(`.ptext` 뒤에 붙는 `<details>`) | 브라우저 기본 접기 + 기존 `.foldsec` 스타일 재사용. **새 클래스를 만들지 않는다** | 색 없음 |
| 팩트 목록 묶음 머리 | 기존 `h2.sec` 그대로 사용 | 기존 토큰만 |
| 상단 띠 현재 위치 밑줄 띠 | 지금 `text-decoration: underline`은 링크 밑줄과 같아 약하다. 굵은 띠 3px로 바꾼다 — **모양만 바뀌고 색은 기존 흰색** | 기존 값 |

**tokens.css에는 색을 추가하지 않는다.** 3차의 이름 4개는 **이미 코드에 있는 값을 옮겨 담는 것**이며 가짓수를 늘리지 않는다(결정 D1).

## 8. 접근성 기본 (절대 단순화 금지)

- 누르는 것은 전부 **최소 44px** — 밀도를 얻으려고 버튼·쪽 이동·✕·접기 손잡이를 줄이지 않는다. 줄이는 것은 **표 한 줄의 위아래 여백뿐**이고, 표 한 줄은 줄 전체가 클릭 영역이라 8px+글자 높이로도 44px에 가깝다(구현에서 최소 높이 40px 이상 확인).
- **색만으로 구분하지 않는다** — 상태 배지는 글자 병기 유지, 확인함은 `✓` 글자, 가림 0건은 색과 함께 `0건`이라는 글자, 현재 메뉴는 굵게 + 띠.
- **숨기는 것과 감추는 것을 구분한다** — 상태 열을 숨길 때, 접힌 줄이 있을 때는 **반드시 글자로 알린다**(§4 문구).
- 입력에는 이름표(팩트 편집 — 기존 그대로).
- 접기는 키보드로 열린다(브라우저 기본).
- 글자 최소 12px, 본문 16px 유지.

## 11. 규칙 승격 제안 — **이 계열 두 번째입니다** (사용자만 반영)

같은 종류의 수정이 두 번째이고, 두 번째가 첫 번째의 부작용입니다. 헌장 **§디자인 규칙**에 아래 한 줄을 넣으면 앞으로 모든 표에 자동 적용됩니다 — 넣을까요?

> - **표의 지면은 사람이 읽어서 구별하는 열에 준다**: 제목·이름처럼 하나씩 읽어 고르는 열이 가장 넓고, 숫자·날짜·상태처럼 모양이 정해진 열은 좁힌다. **지금 보이는 목록에서 값이 한 종류뿐인 열은 숨기고, 숨겼다는 사실을 글자 한 줄로 알린다.**

세 번째가 나오기 전에 올립니다. 반영 여부는 사용자만 정합니다.
