# 뷰어 피드백 — 화면 명세 (designer, 2026-07-20)

대상 차수: **2차(글 목록 걸러 보기 + 눌러서 이동)** · **5차(1~12월 달력형 계절성)** · **6차(분석 탭 역할 전환)**.
근거 계획: `docs/plans/20260720-뷰어-사용피드백-7건.md` (§4 차수표 · §6 진단 · §8 Q1~Q7).
선례 화면: `/`(글 목록) · `/analysis` · `/trends` · `/facts`(가장 최근 패턴) · `templates/tokens.css`.
형식 기준: `docs/plans/20260715-CA-6-룰북팩트-적재-검수화면-design.md`.

## 0. 이 명세의 성패 판정 기준

사용자 피드백의 뿌리는 하나다: **"보이는 숫자를 가지고 다음에 뭘 하면 되는지 모르겠다."**

- 히트맵(3번) = 숫자가 뭔지 모름 → 판단으로 못 이어짐
- 분석 탭(6번) = 숫자는 아는데 그 다음 동작이 없음
- 클릭 이동(7번) = 눈에 띄는 주제를 봤는데 **거기서 갈 곳이 없음**

그래서 이 세 차수의 성패는 예쁨이 아니라 **"화면에서 본 주제 이름 하나를 눌러 그 주제 글 목록에 도착하기까지 몇 번 막히는가"** 하나다. 명세의 모든 선택은 이 기준으로 판단했다.

- 새 지표·새 점수·새 등급을 만들지 않는다(불변 3). **늘리는 건 길(이동)이지 숫자가 아니다.**
- 새 화면 주소를 만들지 않는다. 세 차수 전부 기존 3개 화면(`/` · `/analysis` · `/trends`) 안에서 끝난다.

## 1. 화면 지도 (새 주소 0개)

| 주소 | 이름(메뉴 표기) | 이번에 바뀌는 것 |
|---|---|---|

# 화면 ① — 1~12월 달력형 계절성 (`/trends`, 5차)

## 2-1. 두 표가 공존한다 — 헷갈리지 않게 하는 장치

사용자 결정으로 기존 '쏠림 순위표'를 **없애지 않는다.** 축이 반대인 두 표가 한 화면에 있으므로, **각 표가 어떤 질문에 답하는지를 제목 바로 밑에 한 줄로 못 박는다.** 이것이 이 섹션 설계의 전부다.

| 표 | 답하는 질문 (화면에 그대로 적을 문장) | 보는 방향 |
|---|---|---|
| **달력형(새것)** | "**3월에 우리는 주로 무슨 주제를 썼나?**" | 달 → 주제 |
| **쏠림 순위표(기존)** | "**이 주제는 주로 몇 월에 쓰나?**" | 주제 → 달 |

- 두 표는 **같은 큰 제목("월별 계절성") 아래 위아래로** 둔다. 나란히(좌우) 두지 않는다 — 좌우로 두면 "같은 것의 두 버전"으로 읽혀 오히려 헷갈리고, 좁은 화면에서 어차피 위아래로 접힌다.
- 순서는 **달력형이 위**. 사용자가 요청한 각도이고, "지금이 7월이니 8월 준비"라는 실제 동작에 바로 닿는다.

## 2-2. 배치 (위 → 아래) — `/trends` 화면 전체 순서

```
[상단 띠 — 붙어 있음]  글 목록 · 분석 · 주제·시기 트렌드(굵게·밑줄) · 주제 검수 · 팩트 룰북 · 데이터
                        주제·시기 트렌드            [배지: 대상 10,429건]
──────────────────────────────────────────────
  (기존 머리말 2줄 — 그대로)

  ▣ 월별 비중 히트맵            (기존, 3차에서 숫자 설명이 붙음)

  ▣ 월별 계절성                                        ← 큰 제목 하나
  ┌ 1월부터 12월까지 — 그 달에 많이 쓴 주제 ────────────────┐
  │ 답하는 질문: "3월에 우리는 주로 무슨 주제를 썼나?"        │
  │ 그 달에 **우리가 많이 쓴** 주제입니다 — 사람들이 그 달에   │
  │ 많이 찾는 주제가 아닙니다.                              │
  │ 2023~2026년의 같은 달을 모두 합쳤습니다.                 │
  │                                                        │
  │ ┌ 1월 ─ 총 812건 ┐ ┌ 2월 ─ 총 640건 ┐ ┌ 3월 …          │
  │ │ 사회복지사2급 41 │ │ 보육교사   38 │ │                │
  │ │ 평생교육사   33 │ │ 학점은행제 27 │ │                │
  │ │ 한국어교원   28 │ │ …            │ │                │
  │ │ 심리상담사   21 │ │              │ │                │
  │ │ 아동학대예방 17 │ │              │ │                │
  │ └────────────┘ └────────────┘ └────         │
  │ (넓은 화면 4장씩 3줄 · 좁으면 자동으로 접혀 내려감)        │
  └──────────────────────────────────────────┘

  ┌ 특정 달에 쏠린 주제 ──────────────────────────────┐
  │ 답하는 질문: "이 주제는 주로 몇 월에 쓰나?"              │
  │ (기존 표 그대로: 주제 · 총 글수 · 가장 많이 쓴 달 · 그 달 비중) │
  └──────────────────────────────────────────┘

  ▣ 월초·중순·말 분포            (기존)

  [정직 박스 — 기존 것 하나만, 화면 맨 아래]
```

- **주제 이름은 전부 링크**(진한 파랑·굵게 = `.r-title` 관례). 누르면 그 주제로 걸러진 글 목록으로 간다(→ §4).
- **가장 흔한 동작**(= 주제 눌러 글 보기)이 카드 안 각 줄 전체라, 손이 닿는 면적이 넓다.

## 2-3. 달 카드 한 장의 내용 (확정)

| 자리 | 내용 |
|---|---|
| 카드 머리 | `1월` (큰 글자) + `총 812건` (작은 회색) + 이번 달이면 `[이번 달]` 칩 |
| 본문 | 그 달 글이 많은 주제 **5개**, `주제이름 ····· 41건` |
| 동률 정리 | 건수가 같으면 **주제 이름 가나다순**. (정하지 않으면 화면을 열 때마다 순서가 흔들려 보인다) |
| 글 적은 달 | 그 달 총 글수가 **30건 미만**이면 카드 아래 회색 한 줄: "글이 적은 달이라 순서가 쉽게 바뀝니다." |
| 그 달 글 0건 | 카드를 **지우지 않고** 남기고 회색 한 줄: "이 달에 쓴 글이 없습니다." — 지우면 12칸이 어그러지고 "안 쓴 달"이라는 사실이 안 보인다 |

## 2-4. 정직성 — 노란 박스를 하나 더 만들지 않는다

판단: **달력형에는 노란 정직 박스를 새로 두지 않는다.** 대신 섹션 부제에 검은 글씨 한 줄로 넣고, 화면 맨 아래 기존 정직 박스는 그대로 둔다.

근거: 한 화면에 노란 박스가 2개가 되면 둘 다 배경처럼 읽혀, 정작 아래 박스(발행 습관 ≠ 검색 수요)의 무게가 깎인다. 정직 문구는 **읽히는 자리에 한 번**이 원칙이다.

넣을 문장(그대로):
> 그 달에 **우리가 많이 쓴** 주제입니다 — 사람들이 그 달에 많이 찾는 주제가 아닙니다.

## 2-5. 상태 4종

| 상태 | 화면 |
|---|---|
| **비었을 때** | 트렌드 화면 전체가 비는 경우는 기존 처리 그대로(창고 비었음 / 날짜·키워드 형태 문제). 달력형만 비는 경우는 없다 — 트렌드가 그려지면 달은 반드시 12칸이 나온다. 개별 달이 0건인 경우는 §2-3 |
| **불러오는 중** | 별도 표시 없음. 기존 6개 화면과 같이 한 번에 그려진다(로딩 화면 선례 없음 = 사다리 1단계) |
| **실패했을 때** | 기존 문구 그대로: "트렌드를 불러오지 못했습니다. 자산창고 파일을 먼저 만든 뒤 다시 열어주세요." |
| **성공했을 때** | 위 배치 |

# 화면 ② — 분석 탭 역할 전환 (`/analysis`, 6차)

## 3-1. 무엇이 바뀌는가 한 줄

**"무엇이 잘 됐나(순위표)" → "이 주제로 우리가 쓴 게 뭐가 있나(재료 찾기)".** 지표를 하나도 새로 만들지 않고, **있던 숫자의 배치 순서와 이동 경로만** 바꾼다.

## 3-2. 배치 (위 → 아래)

```
[상단 띠]  … 분석(굵게·밑줄) …      주제로 우리 글 찾기      [배지: 창고 글 10,755건]
──────────────────────────────────────────────
  이 화면은 창고에서 **원고 재료를 찾는 입구**입니다.
  주제·담당자를 누르면 그 조건의 글 목록이 열립니다.

  [정직 박스 — 화면 맨 위]
  여기 숫자로 **알 수 있는 것**: 어떤 주제로 몇 편을 썼는지, 그 주제 글이 보통
  몇 문단·몇 장짜리인지, 아직 거의 안 쓴 주제가 무엇인지.
  여기 숫자로 **알 수 없는 것**: 어떤 글이 잘 됐는지. 카페 조회수는 참고 신호일
  뿐이고, 조회수가 있는 글도 전체의 15%뿐입니다. 잘 됐는지는 우리 홈페이지·
  워드프레스 실측이 쌓여야 답할 수 있습니다.

  ▣ 주제별 우리 글                                   ← 섹션 A
  ┌ 주제 ── 쓴 글 ── 조회수 있는 글 ── 평균 문단 ── 평균 이미지 ┐
  │ 사회복지사2급   412건    38건       7.2      5.1        │ ← 주제 이름이 링크
  └───────────────────────────────────────┘
  정렬: 많이 쓴 순 · 적게 쓴 순 · 가나다순
  ‘평균 문단·이미지’는 그 주제 글이 보통 어떻게 생겼는지입니다.
  잘 된 글의 기준이 아니라 원고 틀을 잡을 때 쓰는 숫자예요.

  ▣ 아직 거의 안 쓴 주제                              ← 섹션 B
  ┌ 팩트 항목 ── 카테고리 ── 그 주제로 쓴 글 ── (팩트 보기) ┐
  │ 요양보호사      복지         3건          팩트 보기 →  │
  └───────────────────────────────────────┘
  룰북 팩트에 정리해 둔 51개 항목 중, 그 이름으로 쓴 글이 적은 것부터입니다.
  팩트 항목 이름과 글 키워드가 서로 다른 말이면 실제보다 적게 잡힐 수 있어요 —
  0건으로 보이면 글 목록에서 직접 찾아보고 판단해 주세요.

  ▣ 담당자별 우리 글                                  ← 섹션 C
  ┌ 담당자 ── 쓴 글 ── 조회수 있는 글 ── 평균 조회수 ┐  ← 담당자 이름이 링크
  └────────────────────────────────┘
  ※ 구현 실물(2026-07-21): E3는 designer 권장(가)이 아니라 **사용자 결정 (나) 펼쳐 두기**로
    확정됐다. 그래서 담당자 표를 둘로 나누지 않고 **하나로 합쳤다** — 접기 안의 '담당자별 평균
    조회수' 표를 없애고 그 숫자를 이 표의 열로 옮겼다(같은 표를 두 번 그리지 않으려고).
    줄 순서는 **쓴 글 많은 순**(평균 조회수 순이 아니다 — 사람 평가표로 읽히지 않게).
    표 밑 한 줄: "‘평균 조회수’는 조회수를 확보한 글만으로 낸 참고 신호예요 — 사람을 견주는
    숫자가 아닙니다."

  ▸ 조회수로 본 참고 신호 — 결론: 글의 형식으로는 갈리지 않았습니다  ← 접혀 있음
    (펼치면 기존 표가 그대로: 조회수 높은 글 · 주제별 조회수 · 원본 키워드별 ·
     형식과 조회수의 관계 + 기존 정직 박스·표본 편중 경고·조절 바.
     담당자별 평균 조회수만 섹션 C로 올라갔다 — 위 ※ 참조)
    ※ 접기 안의 조절 바(정렬·기간)를 누르면 화면이 다시 열리므로, 기본값이 아닐 때는
      접기가 **펼쳐진 채로** 나온다(누른 결과가 접힌 채 사라지지 않게).
```

- **정직 박스를 화면 맨 위로 올린다.** 지금은 아래쪽에 있어 "참고할 게 없다"는 느낌만 남고 이유가 안 읽혔다. 이 화면은 **할 수 있는 판단과 못 하는 판단을 먼저 말하고 시작**한다.
- **섹션 A가 첫 표**인 것이 이번 전환의 핵심이다. 지금 첫 표는 '조회수 높은 글'이고, 그게 이 화면을 순위표로 읽히게 만든 주범이다.
- **지운 것 0개.** 결론 난 것은 전부 접기 안으로 들어간다(브라우저 기본 `<details>`).

## 3-3. 접힌 섹션의 한 줄 요약 (그대로 쓸 문장)

> ▸ **조회수로 본 참고 신호** — 결론: 문단 수·이미지 수·글자 수로는 많이 본 글과 아닌 글이 갈리지 않았습니다. 조회수 표는 여기 접어 두었어요.

## 3-4. 새로 계산하지 않는 것 (명시)

- 추천 점수 · 우선순위 등급 · "이 주제를 쓰세요" 같은 판정 **만들지 않는다**(근거 없는 숫자 = 불변 3 위반).
- 섹션 B의 순서는 **글 수 적은 순**이라는 사실만으로 정한다. "추천도"가 아니다. 제목도 '추천 주제'가 아니라 **'아직 거의 안 쓴 주제'**(사실 서술).
- 섹션 A의 평균 문단·이미지는 **성과 주장이 아니라 형태 설명**이라고 표 밑에 적는다(위 문구).

## 3-5. 상태 4종

| 상태 | 화면 |
|---|---|
| **비었을 때** | ① 창고 글 0건 → 기존 문구 유지 ② 섹션 B에서 팩트가 0건 → "룰북 팩트가 아직 창고에 없습니다. 팩트를 넣으면 여기서 '아직 안 쓴 주제'를 볼 수 있어요. [팩트 룰북 →]" ③ 접힌 섹션 안이 빌 때(조회수 있는 글 0건) → 기존 "아직 분석할 글이 없습니다" 문구를 **접기 안에서** 보여준다 — 이 경우에도 섹션 A·B·C는 정상으로 보인다(조회수와 무관하게 만들어지므로) |
| **불러오는 중** | 별도 표시 없음 |
| **실패했을 때** | 기존 문구: "분석을 불러오지 못했습니다. 자산창고 파일을 찾을 수 없어요. 자산창고를 먼저 만든 뒤 다시 열어주세요." |
| **성공했을 때** | 위 배치 |

★ 중요: **조회수가 하나도 없어도 이 화면은 쓸모가 있어야 한다.** 섹션 A·B·C는 조회수를 쓰지 않는다. 이것이 역할 전환의 실질이다.

# 화면 ③(공통) — 눌러서 걸러진 글 목록 (`/`, 2차)

## 4-1. 걸러 보기 줄 (목록 맨 위)

기존 '보기/정렬' 두 줄 **아래에 셋째 줄**로 붙인다. 순서 = 자주 쓰는 것이 위.

```
보기: 전체 · 성공만 · 실패만    |  정렬: 최신순 · 조회수 높은 순
걸러 보기:  [카페 ▾] [담당자 ▾]  [적용]
```

- **카페·담당자 모두 고르는 칸(`<select>`)** + **[적용] 버튼** 하나.
  ※ 구현 실물(2026-07-21): 고르는 칸의 값은 **목록에 실제로 나오는 글(본문을 가져온 글)에서만** 뽑는다 — 창고 전체 명단이 아니다. 그래서 담당자 칸은 담당자 표(78명)보다 적은 61개다.
  근거: 자바스크립트가 없어 고르는 즉시 이동시킬 수 없다 → 버튼이 필요하다. 카페 18개를 링크로 늘어놓으면 좁은 화면에서 세 줄로 삐져나가고, 두 조건의 조작 방법이 서로 달라지면 배우는 게 두 배가 된다.
- **게시판은 넣지 않는다**(10,755건 중 8,955건이 비어 있어 대부분 글이 사라져 보인다).
- 주제는 **고르는 칸에 넣지 않는다** — 1,309종이라 고를 수 없다. 주제는 **트렌드·분석에서 눌러 들어오는 길로만** 걸린다.

## 4-2. 걸러진 상태를 어떻게 알리는가 (확정)

목록 위 '범위 줄'(`N건 중 1~100번째 보는 중`) **바로 위**에 조건 줄을 둔다. 조건이 하나도 없으면 이 줄 자체가 없다.

```
걸러 보는 중:  [주제 ‘플래너’ ✕]  [담당자 ‘○○○’ ✕]      모두 지우기
주제 ‘플래너’에는 원본 키워드 3종이 묶여 있습니다: 플래너, 플래너자격증, 플래너 과정
37건 중 1~37번째 보는 중
```

| 요소 | 확정 내용 |
|---|---|
| 조건 하나의 모양 | `[주제 ‘플래너’ ✕]` — **무슨 종류의 조건인지(주제/담당자/카페)를 값 앞에 붙인다.** 값만 있으면 "○○○"이 담당자인지 카페인지 모른다 |
| 조건 지우기 | 각 조건의 **✕가 링크**(그 조건만 빠진 목록으로 이동). ✕는 글자 아이콘이 아니라 **누를 수 있는 크기(최소 44×44)** |
| 전부 지우기 | 줄 오른쪽 끝 **`모두 지우기`** — 글자 링크. 조건이 2개 이상일 때만 보인다 |
| 조건 2개 이상 | **모두 만족하는 글만**(그리고). 화면에도 그렇게 읽히도록 조건 사이를 띄어 나열하고, 건수는 걸러진 뒤 숫자로 말한다 |
| 주제 조건일 때 한 줄 더 | **그 주제에 묶인 원본 키워드를 나열한다**(위 예시 둘째 줄). 왜 건수가 예상과 다른지를 화면이 스스로 설명한다 |
| 주제 조건일 때 두 숫자 (사용자 결정 2026-07-21) | 트렌드·분석의 주제 옆 숫자(창고 글 수)와 목록 건수(본문을 가져온 글)는 **다르다. 맞추지 않고 둘 다 보여주며 뜻을 적는다** — "42건 썼는데 본문은 12건뿐"이라는 사실 자체가 못 가져온 글의 크기를 알려준다. 문구: "창고에 주제 ‘플래너’로 쓴 글은 **42건**, 그중 **본문을 가져온 글은 12건**입니다. 아래 목록에는 본문을 가져온 글만 나옵니다. 트렌드·분석 화면에서 주제 이름 옆에 보이는 숫자는 창고 글 수 쪽이에요." |
| 상단 띠 배지 | 창고 전체 기준 숫자 그대로 유지(`총 10,755건 · 성공 …`). 걸러진 숫자는 범위 줄이 말한다 — 두 숫자가 섞이면 "글이 사라졌다"고 읽힌다 |

## 4-3. 걸러 봤는데 0건일 때

```
[상태 박스]
고른 조건에 맞는 글이 없습니다.
조건: 주제 ‘플래너’ · 담당자 ‘○○○’
조건을 하나씩 지워보세요.  [담당자 조건 지우기]  [모두 지우기]
```

조건 줄은 **지우지 않고 그대로 남긴다** — 무엇 때문에 비었는지가 안 보이면 되돌릴 방법이 없다.

## 4-4. 원래 화면으로 돌아가는 흐름

- **브라우저 뒤로 가기**를 쓴다. 별도 '← 트렌드로' 링크를 만들지 않는다.
  근거(사다리 3단계): 되돌아갈 곳을 화면이 알려면 어디서 왔는지를 주소에 달고 다녀야 하는데, 그러면 주소가 길어지고 즐겨찾기·새로고침 때 엉뚱한 링크가 남는다. 상단 띠 메뉴가 모든 화면에 항상 있으므로 길을 잃지 않는다.
- **보기·정렬·쪽 이동을 눌러도 걸러 놓은 조건은 유지된다.** 단 **조건을 바꾸면 언제나 1쪽부터**(기존 관례와 동일 — 다른 목록이 됐는데 27쪽에 서 있으면 텅 빈 화면이 뜬다).
- 주소에 조건이 그대로 남으므로 **즐겨찾기·새로고침해도 같은 화면**이 나온다.

| 화면 | 누를 수 있게 되는 것 | 도착 |

| `.fchip`(걸러진 조건 칩 + 지우기) | 기존 `.chip`은 누를 수 없는 표시용이라 지우기 손잡이가 없고 높이가 44px에 못 미친다. 조건 지우기는 이 차수의 주 동작이라 손가락 크기가 필수 | **기존 토큰만**(`--line` 테두리 · `--muted` 글자) |
| `.an12` / `.an13`(분석 섹션 A·B 표 비율) | 기존 관례대로 그리드 비율 + 최소 폭만 추가 | 색 없음 |

**tokens.css에는 아무것도 추가하지 않는다. 새 회색 하드코딩도 늘리지 않는다**(이미 3곳 있고 그 이상 늘리지 않기로 한 상태 — `.chip.dim` · `.placeholder` · `.listrow:hover`).

## 7. 접근성 기본 (절대 단순화 금지)

- **누를 수 있는 것은 전부 링크 모양(진한 파랑·굵게) + 최소 44px.** 조건 지우기 ✕, 달 카드 안 주제 줄, 고르는 칸과 [적용] 버튼 전부 포함. **작은 글자 링크로 만들지 않는다.**
- **색만으로 구분하지 않는다.** 걸러진 상태는 색이 아니라 **글자("걸러 보는 중")와 조건 이름**으로 알린다. 히트맵 색 농도는 3차에서 붙는 숫자·범례와 함께여야 뜻이 전달된다.
- **고르는 칸에는 이름표를 붙인다** — `카페` `담당자`라는 글자를 칸 앞에 두고, 칸 안 기본값(`카페 전체`)으로 대신하지 않는다.
- **화면 이동은 링크, 조건 적용은 버튼.** 섞지 않는다.
- 접기(`<details>`)는 키보드로 열린다(브라우저 기본).
- 달 카드는 **읽는 순서가 곧 1월→12월**이어야 한다(화면 낭독기도 같은 순서로 읽는다).

## 10. 규칙 승격 제안 (사용자만 반영) — **세 번째입니다**

어제 명세 §12에서 같은 계열 요구가 **두 번** 나왔다고 적고 보류해 두었습니다. **이번이 세 번째입니다.**

- 2026-07-07: "뷰어가 원문 대조 불가(가려진 본문만 표시)" → 좌우 대조로 개편
- 2026-07-14: "정규화 D2 눈검수 미완 — 숫자를 신뢰시키지 말 것" → 원본 키워드를 함께 보여주는 화면으로 해결
- 2026-07-15: 팩트 화면 — 고친 칸에서 '엑셀 원본 보기'
- **2026-07-20(이번)**: 주제로 걸렀을 때 "이 주제에 묶인 원본 키워드"를 함께 보여주기(§4-2) — 같은 요구가 또 나왔습니다

헌장 **§디자인 규칙**에 아래 한 줄을 넣으면 앞으로 모든 화면에 자동으로 적용됩니다 — 넣을까요?

> - 자동으로 만들어진 값(추출·묶기·AI 초안)을 보여주는 화면은 **사람이 원본과 대조할 수 있는 길**을 같은 화면 안에 둔다. 값만 보여주고 근거를 감추지 않는다.

**그리고 이번 피드백이 새로 드러낸 별개 계열**(FEEDBACK_LOG 2026-07-20 ★★ — "화면이 무슨 숫자인지 설명하는 데 실패했다")도 같이 올립니다. 아직 **한 번**이라 승격이 이르지만, 두 번째가 나오면 바로 올리겠습니다. 예비 문안:

> - 화면에 숫자를 놓을 때는 **그 숫자가 무엇을 센 것인지**를 같은 화면에 적는다(단위·기준·범례). 결정 기록에 적어 둔 것은 사용자가 아는 것이 아니다.

구현 중 사용자 요청으로 화면이 이 명세와 달라지면 **이 파일을 함께 고친다.** 명세와 실물이 어긋난 문서는 다음 화면을 오염시킨다.
